# ТЗ: Система управления номенклатурой
## Сфера применения: вентиляция и холодильное оборудование

---

## 1. Общее описание

Система позволяет гибко создавать и управлять номенклатурой технических элементов. Вместо фиксированного списка позиций — настраиваемый механизм: сначала создаются справочники, затем шаблоны типов элементов, затем конкретные позиции.

### Принцип работы (сверху вниз)

```
Справочники (единицы, свойства, категории)
        ↓
Шаблоны типов элементов (Воздуховод, Винт, Вентилятор...)
        ↓
Шаблон названия (из каких свойств собирается имя позиции)
        ↓
Конкретные позиции номенклатуры (Воздуховод 500х300х1000 оц.сталь)
```

### 2.1 `categories` — Категории элементов

Дерево категорий через `parent_id`. **Принцип: плоское функциональное дерево + фасетная навигация.** Категория группирует элементы по назначению (одна ось — функция). Дальнейшая навигация вглубь — через динамические фасеты, не через подкатегории.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| name | varchar | Название категории |
| parent_id | int FK→categories | Родительская категория (null = корневая) |
| sort_order | int | Порядок отображения |

**Стартовое дерево (8 функциональных категорий, 1 уровень):**
```
id | name                          | parent_id
1  | Вентиляция и фасонные части   | null
2  | Вентиляционное оборудование   | null
3  | Холодоснабжение и трубы       | null
4  | Крепёж и метизы               | null
5  | Монтажные системы             | null
6  | Электрика и автоматика        | null
7  | Изоляция и расходники         | null
8  | Прочее                        | null
```

**Фасетная навигация (не хранится в схеме — строится запросом):**
```
Категория (функция)         → Крепёж              ← плоское дерево
  └─ Тип изделия (фасет №1)  → Болт, Гайка, Шайба  ← reference 57
       └─ Фикс. свойства     → Тип резьбы, Покрытие, Класс  ← element_properties (fixed)
            └─ Свойства позиц.→ Диаметр, Длина      ← element_position_properties
```

**Почему плоское дерево, а не подкатегории:**
```
✓ Категория = одна ось (функция/назначение)
✓ Форма (круглый/прямоугольный), материал, размер — это СВОЙСТВА, не ветки
✓ "Все отводы" = категория + фасет по Тип сечения (не две ветки дерева)
✓ Тип изделия — первый фасет, заменяет ручные подкатегории
✓ Ноль поддержки — новое значение свойства = новый фасет автоматически
✓ Категория на элементе (не на типе) — один тип (Труба) попадает в разные
  категории через разные элементы (медная→Холод, ПВХ→Электрика)
```

`parent_id` оставлен в схеме на случай если для отдельной категории всё же понадобится подуровень — но по умолчанию дерево плоское.

### 2.3 `references` — Значения измерений

Центральная справочная таблица. Хранит все возможные значения — и физические единицы (мм, кг, Па) и значения перечислений (оц.сталь, НО, RAL9010). Поле `type` определяет поведение в UI.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| reference_type_id | int FK→reference_types | Тип измерения |
| name | varchar | Полное название |
| symbol | varchar | Символ или краткое обозначение (для имени: "оц.сталь", "прямоуг.") |
| article_code | varchar | Короткий код для артикула (для enum: "ОЦ", "П", "ВЗ") |
| type | enum | `unit` = физическая единица, `enum` = значение перечисления |
| sort_order | int | Порядок в выпадающем списке (DN15 перед DN20, RAL9010 перед RAL9016) |
| is_active | bool | Активно (false = скрыто из новых выборов, но старые позиции сохраняют ссылку) |

**Поведение `type` в UI:**
```
unit → числовое поле ввода + это значение как единица  [ 500 ] [ мм ▼ ]
enum → только выбор из списка                          [ оц.сталь ▼ ]
```

### 2.7 `elements` — Элементы (настраиваемые шаблоны)

Определяет что за элемент и к какой категории относится.

> **⚠️ ОТКРЫТЫЙ ВОПРОС (решить позже):** роль поля `name`. Варианты: (1) убрать совсем — имя всегда из `template_groups`; (2) кэш базового типа изделия ("Воздуховод") для быстрых списков; (3) хранить сам паттерн. Полное имя позиции в любом случае генерируется на лету из групп шаблона.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| name | varchar | См. открытый вопрос выше |
| category_id | int FK→categories | Категория |
| status | enum | `active` (можно создавать позиции), `blocked` (нельзя новые, старые работают), `archived` (в архиве, скрыт) |
| replaced_by_id | int FK→elements | Ссылка на новый элемент-замену (null если не заменён) |
| created_at | timestamp | Дата создания |
| updated_at | timestamp | Дата изменения |

**Жизненный цикл элемента (immutable-подход):**
```
Нужно изменить элемент (добавить/убрать свойство)?
  → НЕ меняем существующий
  → Создаём новый элемент
  → Старому ставим status='blocked', replaced_by_id=новый
  → Старые позиции продолжают работать (имена, артикулы стабильны)
  → Когда все проекты завершены → status='archived'

status='active'   → доступен для создания новых позиций
status='blocked'  → нельзя новые позиции, старые работают, виден с пометкой "см. новый"
status='archived' → скрыт из основного интерфейса, только в истории
```

**Пример данных:**
```
id | name        | category_id        | status   | replaced_by_id
1  | Воздуховод  | 3 (Прямоугольные)  | blocked  | 15           ← устарел, заменён
2  | Воздуховод  | 4 (Круглые)        | active   | null
15 | Воздуховод  | 3 (Прямоугольные)  | active   | null         ← новая версия с доп. свойством
```

### 2.7.1 Шаблоны названия и артикула — `template_groups` + `group_tokens`

Шаблон собирается из **групп токенов**. Группа — это блок (например «Ширина х Высота х Длина» или «Ду · DN»). Между группами автоматически ставится разделитель по назначению: пробел для имени, дефис для артикула. Пустая группа (все её свойства не заполнены) исчезает целиком вместе со своим разделителем — висящих разделителей не возникает.

**Две таблицы:**

`template_groups` — группы шаблона
| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| element_id | int FK→elements | Элемент |
| purpose | enum | `name` = читаемое имя, `article` = артикул |
| order | int | Порядок группы в шаблоне |

`group_tokens` — содержимое группы (свойства + внутренние литералы)
| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| group_id | int FK→template_groups | Группа |
| order | int | Порядок токена внутри группы |
| type | enum | `element_name`, `property`, `literal`, `text` |
| value | varchar | Зависит от type (см. ниже) |

**Значение `group_tokens.value` по типу:**
```
type = 'element_name' → value = null           (берётся elements.name автоматически)
type = 'property'     → value = ep_id          (FK→element_properties)
type = 'literal'      → value = name_literal_id (FK→name_literals — внутренний разделитель: х, Ф, Ду, М)
type = 'text'         → value = произвольная строка (только если нет подходящего литерала)
```

**Правила сборки:**
```
1. Разделитель МЕЖДУ группами — по purpose:
     purpose = 'name'    → пробел " "
     purpose = 'article' → дефис "-"
   (не хранится у группы, определяется правилом сборки)

2. Внутри группы — токены склеиваются как есть (с внутренними литералами х, Ду...)

3. Пустая группа (все property-токены без значения) → группа НЕ выводится,
   её межгрупповой разделитель тоже не выводится → нет висящих разделителей

4. Внутренние литералы (х, Ду, М) живут в группе → исчезают вместе с ней
```

**Что подставляется в `property`-токен (зависит от purpose и типа значения):**
```
purpose = 'name':
  reference.type = unit → введённое число + symbol единицы ("500", "оц.сталь")
  reference.type = enum → reference.symbol ("прямоуг.", "оц.сталь")

purpose = 'article':
  reference.type = unit → введённое число ("500")
  reference.type = enum → reference.article_code ("П", "ОЦ", "ВЗ")
```

**Автопредзаполнение:** при создании элемента шаблон `article` генерируется автоматически — каждое свойство (тип изделия + fixed + обязательные) становится отдельной группой, между ними дефис. Администратор группирует/правит по необходимости.

**UI конструктора (drag-and-drop):**
```
Группы — это видимые блоки. Внутри блока — свойства и внутренние литералы.

  ┌──────┐ ┌──────────┐ ┌─────────────────────────┐ ┌────────┐ ┌───────┐
  │ Болт │ │ М·Резьба │ │ Ширина·х·Высота·х·Длина  │ │ Металл │ │ Класс │
  └──────┘ └──────────┘ └─────────────────────────┘ └────────┘ └───────┘
       └─ пробел ─┘  └─ пробел ─┘            └─ пробел ─┘   └─ пробел ─┘
            (между группами — авто, по purpose)

ПАЛИТРА:
  Свойства  → одноразовые (одно свойство в шаблоне один раз)
  Литералы  → многоразовые (х, Ф, Ду, М — кладутся ВНУТРЬ группы)
  Текст     → многоразовый

Действия:
  - перетащить свойство → создать новую группу или вложить в существующую
  - литерал/текст → только внутрь группы (как внутренний разделитель)
  - группы перетаскиваемы (смена порядка)
  - токены внутри группы перетаскиваемы
  - разделитель между группами не ставится вручную — он авто по purpose
```

**Пример — Воздуховод прямоугольный (purpose='name'):**
```
Группы (между ними пробел):
  group 1: [element_name]              → "Воздуховод"
  group 2: [property ТипСечения]       → "прямоугольный"
  group 3: [Ширина][лит х][Высота][лит х][Длина]  → "500х300х1250"
  group 4: [property Металл]           → "оц.сталь"

Сборка через пробел → "Воздуховод прямоугольный 500х300х1250 оц.сталь"
```

**Пример — Болт метрический (purpose='name'):**
```
  group 1: [element_name]              → "Болт"
  group 2: [property ТипРезьбы]        → "метрический"
  group 3: [лит М][Резьба][лит х][Длина] → "М10х50"
  group 4: [property Стандарт]         → "DIN933"
  group 5: [property Класс]            → "8.8"

Сборка через пробел → "Болт метрический М10х50 DIN933 8.8"

Если Класс не заполнен → group 5 пустая → исчезает с разделителем:
  → "Болт метрический М10х50 DIN933"   (без висящего пробела)
```

**Пример — Кран шаровой (обозначение Ду внутри группы):**
```
  group 1: [element_name]              → "Кран"
  group 2: [property ТипКлапана]       → "шаровой"
  group 3: [property ТипСоединения]    → "ф/ф"
  group 4: [лит "Ду "][DN]             → "Ду 125"   (Ду внутри группы)
  group 5: [лит "Ру "][PN]             → "Ру 16"

Сборка через пробел → "Кран шаровой ф/ф Ду 125 Ру 16"
```

**Пример — Шумоглушитель трубчатый (qualifier внутри группы):**
```
  group 1: [element_name]              → "Шумоглушитель"
  group 2: [property ТипШумогл.]       → "трубчатый"
  group 3: [лит Ф][Диам.внутр.][лит /][лит Ф][Диам.наружн.][лит -][Длина]
                                       → "Ф250/Ф350-900"

Сборка через пробел → "Шумоглушитель трубчатый Ф250/Ф350-900"
```
Qualifier (Внутренний/Наружный) зашит в ep_id — токен ссылается на конкретное свойство.

**Пример — артикул того же Воздуховода (purpose='article', разделитель дефис):**
```
  group 1: [property ТипИзделия]  → "ВЗ" (article_code)
  group 2: [property ТипСечения]  → "П"
  group 3: [Ширина]               → "500"
  group 4: [Высота]               → "300"
  group 5: [Длина]                → "1250"
  group 6: [property Металл]      → "ОЦ"

Сборка через дефис → "ВЗ-П-500-300-1250-ОЦ"
```

### 2.8 `element_properties` — Свойства элемента

Шаблон: какие свойства входят в элемент, из какого типа измерений выбирать значение, и какое значение зафиксировано заранее.

Два ключевых поля:
- `reference_type_id` — определяет **откуда брать список** при вводе (тип Длина → показываем мм/м/см, тип Тип сечения → показываем Круглое/Прямоугольное/Овальное)
- `fixed_reference_id` — если заполнен, значение **заблокировано** и пользователь не может его изменить. Если null — пользователь выбирает сам

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| element_id | int FK→elements | Элемент |
| property_id | int FK→properties | Свойство |
| reference_type_id | int FK→reference_types | Тип измерения — определяет список значений в UI |
| fixed_reference_id | int FK→references | Фиксированное значение (null = пользователь выбирает) |
| qualifier_id | int FK→qualifiers | Уточнение (null если не нужно) |
| is_required | bool | Обязательное поле |
| min_value | decimal | Минимум для числовых (unit), null = без ограничения |
| max_value | decimal | Максимум для числовых (unit), null = без ограничения |
| step | decimal | Кратность ввода (50 = только 100, 150, 200...), null = любое |
| sort_order | int | Порядок в форме |

**Валидация (только для unit-свойств):**
```
min_value=100, max_value=2000, step=50  (Ширина воздуховода)
  500  → ✓
  5000 → ✗ "Максимум 2000 мм"
  50   → ✗ "Минимум 100 мм"
  523  → ✗ "Шаг 50 мм: допустимо 500 или 550"

min_value=null, max_value=null  → без ограничений
min_value=0, max_value=null     → только положительное
```

## 3. Роли и процесс работы

### Администратор / технолог — Настройка
1. Создаёт `reference_types` — простой справочник названий типов (Длина, Масса, Материал...)
2. Создаёт `references` — значения для каждого типа с указанием `type` (unit/enum)
3. Создаёт `qualifiers` — глобальный справочник уточнений
4. Создаёт `properties` — каждое свойство привязано к `reference_type_id`
5. Связывает через `property_qualifiers` — какие уточнения допустимы для каждого свойства
6. Создаёт категории (`categories`)
7. Создаёт элементы (`elements`) — базовое имя типа "Воздуховод", "Болт", "Клапан"
8. Настраивает свойства элемента (`element_properties`) — свойства + reference_type_id + fixed_reference_id + qualifier
9. Настраивает шаблоны (`template_groups` + `group_tokens`) — `name` (читаемое имя) и `article` (артикул). Артикул предзаполняется автоматически

### Пользователь — Номенклатура
1. Выбирает элемент из списка
2. Заполняет форму — поля генерируются из `element_properties`
3. Для `unit` — вводит число, единица уже задана в шаблоне
4. Для `enum` — выбирает из дропдауна
5. Видит название в реальном времени
6. Сохраняет позицию — `element_positions` + `element_position_properties`

### Пользователь — Номенклатура
1. Выбирает элемент из списка
2. Заполняет форму — поля генерируются автоматически из `element_properties`
3. Видит название в реальном времени (вычисляется по группам шаблона)
4. Сохраняет позицию — создаётся `element_positions` + `element_position_properties`
5. Использует позиции в проектах с фильтрацией по любому свойству

## 4. Ключевые принципы схемы

- **`reference_types`** — справочник типов, группирует все значения по смыслу (Длина, Металл, Тип сечения, Тип изделия...)
- **`references`** — центральная справочная таблица всех значений системы (мм, кг, оц.сталь, НО, Воздуховод...). Поле `type` определяет поведение в UI: `unit` = число + выбор единицы, `enum` = только выбор из списка
- **`elements.name` автовычисляемое** — заполняется автоматически из фиксированного свойства `Тип изделия` при сохранении элемента. Не вводится вручную. Нужно для быстрого отображения в списках без JOIN
- **`element_properties` имеет два ключевых поля** — `reference_type_id` определяет из какого типа показывать список в UI, `fixed_reference_id` фиксирует значение заранее. Если заполнен — поле заблокировано 🔒 и значение копируется автоматически при создании позиции
- **`template_groups` + `group_tokens` — шаблоны из групп**, два шаблона через `purpose`: `name` (читаемое имя) и `article` (артикул). Группа = блок свойств с внутренними литералами (х, Ду, М). Между группами авторазделитель по purpose (пробел для имени, дефис для артикула). Пустая группа исчезает целиком со своим разделителем — нет висящих разделителей. Токены: `element_name`, `property` (по ep_id), `literal` (из name_literals, внутри группы), `text`. Для артикула enum берёт `references.article_code` ("ОЦ", "П"), unit — введённое число
- **`name_literals`** — единый справочник разделителей и стандартных обозначений: " ", "х", "Ф", "Ду ", "Ру ", "М", "R=", "EI", "IP"... Покрывает 99% случаев
- **`properties.reference_type_id` всегда заполнен** — нет null, каждое свойство имеет тип
- **`property_qualifiers` защищает от ошибок** — уточняет именно свойство, а не тип. "Наружный" применим к Диаметру но не к Ширине
- **`qualifier_id` опциональный** — используется когда одно свойство нужно в нескольких вариантах (Диаметр наружный / внутренний, Давление рабочее / максимальное)
- **`property` в шаблоне ссылается на `element_property_id`** — точная ссылка на конкретную строку `element_properties`. Устраняет неоднозначность когда одно свойство встречается несколько раз с разными qualifiers
- **Категория привязана к элементу** (`elements.category_id`), не к каждой позиции
- **Плоское дерево категорий + фасетная навигация** — категория группирует по функции (8 категорий, 1 уровень). Глубже навигация идёт через динамические фасеты: Тип изделия (фасет №1) → фиксированные свойства (Тип резьбы, Покрытие) → свойства позиций (Диаметр, Длина). Фасеты строятся запросом из `element_properties` и `element_position_properties`, не хранятся в дереве. Это убирает ручную поддержку подкатегорий и комбинаторный взрыв веток
- **Поиск по свойствам** работает через `element_position_properties` — каждое значение отдельная строка, поддерживает точный поиск, диапазоны и комбинированные фильтры
- **Все значения через ключи** — без JSON, каждое значение отдельная строка
- **Элементы immutable** — существующий элемент не меняют. Нужно изменить (добавить/убрать свойство) → создаётся новый элемент, старый получает `status='blocked'` и `replaced_by_id`. Старые позиции работают без изменений — имена и артикулы стабильны. Это убирает необходимость в версионировании позиций и сложной миграции

## 5. Навигация и поиск (фасетная модель)

Система поддерживает фильтрацию в **двух направлениях** — от категории вниз и от свойства вверх. Обе работают на существующих таблицах без изменений схемы.

### 5.1 Прямая навигация (сверху вниз)

```
Категория (функция)          → Крепёж              ← плоское дерево
  └─ Тип изделия (фасет №1)   → Болт, Гайка, Шайба  ← reference 57
       └─ Фикс. свойства      → Тип резьбы, Покрытие, Класс  ← element_properties (fixed)
            └─ Свойства позиц. → Диаметр, Длина      ← element_position_properties
```

Пользователь знает что ищет (болт) → идёт деревом → сужает фасетами.

### 5.2 Обратная навигация (от свойства)

```
Свойство                     → "Расход воздуха"
  └─ все типы где оно есть    → Вентилятор, Решётка, Диффузор, Фильтр
```

Пользователь знает характеристику но не тип → выбирает свойство → видит что подходит из всего каталога.

### 5.3 Режимы фильтрации

```
1. По наличию свойства:
   "что вообще имеет Ширину" → Воздуховод, Отвод, Решётка, Фильтр...

2. По значению свойства:
   "Ширина = 500" → конкретные позиции всех типов

3. По диапазону:
   "Ширина 300-600" → позиции в диапазоне

4. Комбинированный (пересечение нескольких условий):
   "Ширина 500 И Металл оц.сталь И Класс герметичности B"
   → AND по всем условиям
```

### 5.4 Как это работает на схеме

```sql
-- Какие типы изделий имеют свойство "Ширина" (обратный фасет по наличию)
SELECT DISTINCT r.name AS тип_изделия
FROM element_properties ep
JOIN elements e ON ep.element_id = e.id
JOIN element_properties ep_type ON ep_type.element_id = e.id
JOIN references r ON ep_type.fixed_reference_id = r.id
WHERE ep.property_id = (SELECT id FROM properties WHERE name='Ширина')
  AND ep_type.reference_type_id = 57  -- Тип изделия

-- Позиции где Ширина=500 И Металл=оц.сталь (комбинированный по значению)
SELECT p.* FROM element_positions p
WHERE EXISTS (SELECT 1 FROM element_position_properties v
              WHERE v.position_id=p.id AND v.property=Ширина AND v.value='500')
  AND EXISTS (SELECT 1 FROM element_position_properties v
              WHERE v.position_id=p.id AND v.property=Металл AND v.reference_id=оц.сталь)
```

Поскольку каждое значение — отдельная строка с FK (не JSON), такие запросы выполняются напрямую через индексы. Фасеты (списки доступных значений + счётчики) строятся агрегацией по `element_properties` (фиксированные) и `element_position_properties` (значения позиций).

## 6. Модуль мэппинга номенклатуры поставщиков

Механизм сопоставления строк из прайс-листов поставщиков с позициями в нашей системе. Каждый поставщик пишет названия по-своему — система учится распознавать их и накапливает соответствия.

### Принцип работы (три уровня поиска)

```
Строка из прайса поставщика
          ↓
Уровень 1: supplier_mappings — точное совпадение строки?
  ДА → 100% уверенность → принимаем автоматически
  НЕТ ↓
Уровень 2: supplier_patterns — подходит активный паттерн?
  ДА → 85-95% → предлагаем пользователю
  НЕТ ↓
Уровень 3: нечёткий поиск по токенам (числа + синонимы)
  НАЙДЕНО → 50-80% → предлагаем несколько вариантов
  НЕ НАЙДЕНО → предлагаем создать новую позицию
```

### Самообучение

```
Пользователь подтверждает соответствие
          ↓
1. Запись в supplier_mappings (100% на будущее)
2. Проверка — есть ли новый синоним enum-значения?
   "оц/ст" ещё не в supplier_synonyms → предлагаем сохранить
3. Накопилось 3+ подтверждений с похожей структурой?
   → Автогенерация паттерна → статус "ожидает подтверждения"
   → Администратор проверяет и активирует
```

---

### 5.1 `suppliers` — Справочник поставщиков

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| name | varchar | Название поставщика |
| code | varchar | Краткий код |
| created_at | timestamp | Дата добавления |

**Пример данных:**
```
id | name                    | code
1  | ООО Альфа Вентиляция   | ALPHA
2  | ИП Петров В.С.          | PETROV
3  | ТД Климат Плюс          | KLIMAT
```

---

### 5.2 `supplier_mappings` — Подтверждённые соответствия

Главная таблица обучения. Каждая строка = одно подтверждённое пользователем соответствие. При следующем импорте та же строка → мгновенное 100% совпадение.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| supplier_id | int FK→suppliers | Поставщик |
| raw_text | varchar | Строка из прайса (точная копия) |

| created_at | timestamp | Дата подтверждения |

### 5.3 `supplier_synonyms` — Синонимы enum-значений

Словарь как разные поставщики пишут одно и то же enum-значение. `supplier_id = null` — глобальный синоним для всех поставщиков. Поле `hits` — сколько раз использовался при мэппинге.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| supplier_id | int FK→suppliers | Поставщик (null = глобальный) |
| raw_text | varchar | Как пишет поставщик |
| reference_id | int FK→references | Соответствующее значение в нашей системе |
| hits | int | Счётчик использований |

**Пример данных:**
```
id | supplier_id | raw_text    | reference_id      | hits
1  | 1 (Альфа)   | оц/ст       | 32 (оц.сталь)       | 47

4  | 1 (Альфа)   | нерж.       | 33 (нерж.304)       | 8
5  | null        | НО          | 149 (НО норм.откр.) | 31
6  | null        | нж           | 33 (нерж.304)       | 5
```

### 5.4 `supplier_patterns` — Автогенерированные паттерны

Паттерны генерируются автоматически когда система замечает похожую структуру в 3+ подтверждённых соответствиях. Активируются после проверки администратором.

| Поле | Тип | Описание |
|---|---|---|
| id | int PK | Идентификатор |
| supplier_id | int FK→suppliers | Поставщик |
| element_id | int FK→elements | Элемент которому соответствует паттерн |
| pattern_template | varchar | Шаблон с плейсхолдерами {d}=число, {f}=дробь, {s}=текст |
| property_map | json | Маппинг групп паттерна на property_id |
| hits | int | Сколько раз сработал |
| confirmations | int | Сколько подтверждений накоплено |
| status | enum | `pending` = ожидает активации, `active` = активен, `disabled` = отключён |
| created_at | timestamp | Дата создания |

**Пример данных:**
```
id | supplier_id | element_id       | pattern_template                        | property_map                                          | hits | conf. | status
1  | 1 (Альфа)   | 1 (Возд.прям.)   | Воздуховод {d}х{d}х{d} оц/ст т.{f}     | {1:ширина, 2:высота, 3:длина, 4:толщина, mat:оц.сталь}| 47   | 12    | active
2  | 2 (Петров)  | 1 (Возд.прям.)   | Воздух. прямоуг. {d}*{d} L={d} ст.оц.  | {1:ширина, 2:высота, 3:длина, mat:оц.сталь}           | 23   | 7     | active
3  | 1 (Альфа)   | 3 (Винт метр.)   | Болт М{d}х{d} DIN {d}                   | {1:диам_резьбы, 2:длина, 3:станд_крепежа}             | 12   | 4     | active
4  | 3 (Климат)  | 8 (Вентилятор)   | Вент.канальный {s} {s} {d}              | {1:производитель, 2:модель, 3:диаметр}                | 2    | 2     | pending
```

**Статусы:**
```
pending  → накоплено < 3 подтверждений или ожидает проверки администратора
active   → работает автоматически при импорте
disabled → отключён вручную (например паттерн давал ошибки)
```

### 5.5 Процесс импорта прайса в UI

```
Шаг 1 — Загрузка файла
  Пользователь выбирает поставщика и загружает xlsx/csv

Шаг 2 — Парсинг и мэппинг (автоматически)
  Система проходит по каждой строке через 3 уровня поиска
  Формирует список с результатами и уверенностью

Шаг 3 — Экран проверки
  Слева: строки из прайса
  Справа: найденные соответствия (или "не найдено")
  Цвет строки:
    Зелёный  — 100% (точное совпадение из mappings)
    Жёлтый   — 80-99% (паттерн сработал)
    Оранжевый— 50-79% (нечёткий поиск, несколько вариантов)
    Красный  — не найдено

Шаг 4 — Пользователь проверяет и подтверждает
  Зелёные — принимает все сразу
  Жёлтые  — проверяет и принимает/отклоняет
  Оранжевые — выбирает из вариантов или создаёт новую позицию
  Красные  — создаёт новую позицию

Шаг 5 — Сохранение
  Все подтверждения → supplier_mappings
  Новые синонимы   → supplier_synonyms
  Новые паттерны   → supplier_patterns (статус pending)
```

## 6. Будущие расширения (не в текущем ТЗ)

- `composite_items` — комплекты из нескольких позиций
- `compatibility_rules` — правила совместимости элементов
- Медиафайлы — фото и чертежи к позициям
- Единица продажи — в чём продаётся позиция (шт, пм, м²)
- `element_property_conditions` — зависимые свойства. Пока не нужно — любой подобный случай решается созданием отдельного типа элемента. Добавить если появится элемент где разделение на подтипы нецелесообразно.
- AI-мэппинг — подключение Claude API для распознавания нестандартных строк поставщиков. Добавить поверх существующего алгоритма когда базовый механизм будет реализован и протестирован.
