# Зависимые свойства — отложенное решение + влияние на навигацию

---

## РЕШЕНИЕ (текущее)

```
СЕЙЧАС:
  Зависимые свойства НЕ вводим.
  Условные допы (изоляция, привод) = необязательные свойства.
  Выбор что заполнять — на администраторе при заведении элемента.
  Администратор понимает предметную область и заводит корректно.

ПОЧЕМУ не группы:
  Группа объединяет набор свойств, но НЕ выражает зависимость
  от ЗНАЧЕНИЯ (Тип изоляции=Минвата → одно, =Каучук → другое).
  Группа = "включить набор", а нужно "показать по значению".

БУДУЩЕЕ (если понадобится):
  Зависимые свойства через условие show_if/require_if
  (Способ Г — два поля в element_properties).
  Вводить только когда реальная боль от дублирования элементов.
```

## ВЛИЯНИЕ ЗАВИСИМОСТЕЙ НА ОСТАЛЬНУЮ СИСТЕМУ

Это и есть причина почему лучше пока не вводить — зависимости касаются НЕ только формы ввода. Они расходятся по всей системе.

### 1. Шаблон названия

```
БЕЗ зависимостей (сейчас):
  Шаблон линейный, все токены всегда есть.
  "Воздуховод [сечение] [Ш]х[В]х[Д] [металл] [изоляция] [толщина]"
  Если изоляция пустая → нужен механизм "пустой токен не выводить"

С зависимостями:
  Токен изоляции есть ТОЛЬКО когда изоляция выбрана.
  Шаблон должен знать: этот токен условный.
  → шаблон становится "ветвящимся", не линейным
  → сложнее предпросмотр (что показать пока не выбрано?)

ВЫВОД: даже сейчас нужен механизм "пустое значение → токен пропускается".
       С зависимостями — шаблон условный, сложнее.
```

### 2. Фасетная навигация

```
БЕЗ зависимостей:
  Фасет "Тип изоляции" есть у всех воздуховодов.
  Значения: Без изоляции (40), Минвата (12), Каучук (8)
  Чётко, предсказуемо.

С зависимостями:
  Свойство "Толщина изоляции" есть только у части позиций.
  Фасет "Толщина" применим не ко всем → "разреженный фасет"
  → при фильтрации "Толщина = 50" выпадают позиции БЕЗ изоляции
  → пользователь путается: "почему воздуховодов стало меньше?"

ВЫВОД: зависимые свойства создают РАЗРЕЖЕННЫЕ фасеты
       (свойство есть не у всех) — навигация менее предсказуема.
```

### 3. Обратная навигация (от свойства)

```
БЕЗ:
  "Что имеет Толщину изоляции?" → все утеплённые, чётко.

С зависимостями:
  То же, но свойство то появляется то нет в зависимости от Типа изоляции.
  "Покажи всё с Толщиной" → а где Тип изоляции=Без, толщины нет,
  хотя свойство в шаблоне элемента есть.
  → расхождение "свойство определено" vs "свойство заполнено"
```

### 4. Фильтрация и поиск

```
БЕЗ:
  WHERE Толщина = 50 — однозначно.

С зависимостями:
  WHERE Толщина = 50 — но у части позиций толщины нет вообще
  (не NULL, а отсутствует как строка в position_properties).
  → запросы должны учитывать "свойство может отсутствовать"
  → LEFT JOIN вместо JOIN, обработка отсутствия
  → агрегаты (средняя толщина) считаются по подмножеству
```

### 5. Валидация

```
БЕЗ:
  Обязательное — всегда проверяем.
  Необязательное — не проверяем.

С зависимостями (require_if):
  "Толщина обязательна ЕСЛИ изоляция выбрана"
  → валидатор должен вычислить условие перед проверкой
  → порядок заполнения важен (сначала изоляция, потом толщина)
  → при изменении изоляции на "Без" — толщину обнулить?
```

### 6. Создание позиции (форма)

```
БЕЗ:
  Статичная форма, все поля сразу.

С зависимостями:
  Динамическая форма — поля появляются/прячутся при вводе.
  → больше фронтенд-логики
  → пересчёт при каждом изменении управляющего свойства
```

### 7. Мэппинг поставщиков

```
БЕЗ:
  Строка поставщика → позиция с фиксированным набором свойств.

С зависимостями:
  Набор свойств позиции зависит от значений.
  → паттерны мэппинга должны учитывать вариативность
  → "Воздуховод утеплённый" и "без" — разный набор для распознавания
```

## СВОДКА ВЛИЯНИЯ

| Подсистема | Без зависимостей | С зависимостями |
|---|---|---|
| Шаблон имени | линейный + "пустой токен" | условный, ветвящийся |
| Фасеты | плотные, предсказуемые | разреженные (не у всех) |
| Обратная навигация | чёткая | расхождение опред./заполнено |
| Фильтрация | прямой WHERE | LEFT JOIN, обработка отсутствия |
| Валидация | статичная | вычисление условий |
| Форма позиции | статичная | динамическая |
| Мэппинг | фиксир. набор | вариативный набор |

## ГЛАВНЫЙ ВЫВОД

```
Зависимые свойства — это НЕ локальная фича формы ввода.
Они расходятся по 7 подсистемам, делая каждую сложнее
и менее предсказуемой.

Экономия −8 элементов НЕ окупает усложнение 7 подсистем.

Поэтому: оставляем плоскую модель (обязательные + необязательные),
выбор на администраторе. Зависимости — только если появится
реальная массовая боль, и тогда через простое show_if/require_if,
осознавая влияние на навигацию.
```

## Что всё равно нужно сейчас (независимо от зависимостей)

```
Механизм "пустое значение не выводится в имя":
  reference "Без изоляции" с пустым symbol → токен пропускается.
  Нужен и без зависимостей — для необязательных свойств
  которые не заполнили.
  → флаг у reference или правило "пустой токен → skip"
```
