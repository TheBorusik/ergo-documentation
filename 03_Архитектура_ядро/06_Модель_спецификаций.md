# Спецификация и связь с КП (переосмысление)

---

## СМЕНА МОДЕЛИ

```
БЫЛО:
  КП-блок → раскрывается в позиции
  (блок первичен, структура от КП)

СТАЛО:
  Проектировщик создаёт ПОДЗАДАЧИ-спецификации (как ему удобно)
  добавляет позиции, у каждой указывает связь с КП-блоком
  (спецификация первична, КП = разметка/сверка)
```

---

## ПОЧЕМУ ТАК

```
Реальность: спецификацию делают ЧАСТЯМИ
  - по помещениям
  - по системам (воздуховоды, холод)
  - по срокам (что срочно — раньше)
  не ждут весь проект целиком

Идеально бы: вся спецификация готова → согласовать → снабжение
Реально: часто частями, из-за сроков

→ проектировщик сам нарезает работу на подзадачи-спецификации
→ КП остаётся для сверки (видно какие позиции появились, статусы)
```

---

## МЕХАНИЗМ — те же подзадачи

```
ПРОЕКТИРОВАНИЕ (тот же универсальный механизм подзадач):
  проектировщик создаёт:
    "Спецификация помещение 1" (subtask, stage=Проектирование)
    "Спецификация воздуховоды этаж 2" (subtask)
  наполняет позициями
  завершает подзадачу → спецификация готова
  передаёт в снабжение
```

---

## СВЯЗЬ С КП — атрибут позиции

```
Проектировщик добавляет позицию в спецификацию:
  Воздуховод 500х300 ×14
    → выбирает КП-блок: "Приточная установка П1"

project_positions:
  id, project_id, position_id, quantity,
  kp_item_id   ← к какому КП-блоку относится (выбор проектировщика)
  subtask_id   ← в какой спецификации создана
  supplier_id, purchase_status, warehouse_id...

→ позиция знает И свой КП-блок, И свою спецификацию
```

---

## ДВА СРЕЗА НА ПОЗИЦИИ

```
СРЕЗ 1 — по спецификациям (как работал проектировщик):
  Спецификация помещение 1 → Воздуховод×14, Клапан×2
  Спецификация воздуховоды → Воздуховод×20, Отвод×8

СРЕЗ 2 — по КП (сверка с заказчиком):
  Приточная П1 → Воздуховод×14, Клапан×2 (+статусы)
  Материалы → ...

Одни project_positions, группировка по subtask_id или kp_item_id.
```

---

## СОГЛАСОВАНИЕ СПЕЦИФИКАЦИИ

```
Спецификация (или весь проект) согласуется с объектом
перед снабжением.

Подзадача-спецификация завершена → готова к согласованию →
согласована → позиции можно в снабжение.

(возможен статус подзадачи: черновик → готова → согласована)
```

---

## ОБНОВЛЕНИЕ МОДЕЛИ

```
project_positions (единая, конвейер):
  id, project_id, position_id, quantity,
  kp_item_id    FK→kp_items (связь с КП, выбирает проектировщик)
  subtask_id    FK→subtasks (в какой спецификации создана)
  supplier_id, purchase_status, warehouse_id, created_at

subtasks (включая спецификации):
  stage=Проектирование → подзадача-спецификация
  stage=Снабжение → подзадача закупки
  stage=Работы → подзадача работ
  один механизм, разные этапы

kp_items (остаются для сверки):
  позиции ссылаются через kp_item_id
  КП видит свои позиции и их статусы
```

## РАСКРЫТИЕ "МАТЕРИАЛОВ" (закрытие вопроса ⚠️5)

```
"Воздуховоды оц 720м" из КП (группа Материалы):
  → это kp_item
  → проектировщик при создании спецификации добавляет
    конкретные воздуховоды и привязывает к этому kp_item
  → 720м раскрывается в реальные позиции
    (Воздуховод 500х300 ×14, Воздуховод 400х200 ×20...)
    сумма метража ≈ 720м (плановое из КП)

→ материалы раскрываются ТАК ЖЕ как установки,
  через привязку позиций к kp_item.
  КП-метраж (720) = план, факт = сумма реальных позиций
```

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Привязка к КП — практически обязательна

```
kp_item_id заполняется почти всегда, включая добавленные позиции.

Причина новых позиций (пример из практики):
  проектировщик заложил прямую трассу воздуховодов →
  на объекте оказался столб → нужно обойти →
  заказывают доп. элементы (отводы и т.д.) →
  новые позиции (source=added) →
  привязываются к тому же КП-блоку (Воздуховоды)

Технически: kp_item_id nullable
UI: подталкивает выбрать группу при создании позиции
Без привязки → позиция в отдельном списке "без группы" (видно)
```

### ② Один kp_item на позицию

```
Связь позиция → kp_item: один-к-одному.

Теоретически воздуховод мог бы идти и в П1, и в П2 (одинаковый размер),
НО на практике воздуховоды привязывают к разделу "Воздуховоды",
а не к конкретной установке.
→ случай "одна позиция в разных kp_item" не возникает
→ связь простая, kp_item_id одно поле
```

### ③ Спецификация → снабжение целиком, дробится там

```
ПРОЕКТИРОВАНИЕ:
  спецификация-подзадача завершена →
  передаётся в снабжение ЦЕЛИКОМ (вся подзадача)

СНАБЖЕНИЕ:
  получает спецификацию (список позиций)
  САМ создаёт подзадачи закупки и дробит:
    "Воздуховоды → Поставщик А"
    "Клапаны → Поставщик Б"

Разделение логики нарезки:
  проектировщик режет по СВОЕЙ логике (помещения, системы)
  снабженец пересобирает по СВОЕЙ (поставщики)
  одни позиции, разные группировки на разных этапах
```

### Как технически "передаётся в снабжение"

```
Подзадача-спецификация (stage=Проектирование, status=done)
  → её позиции становятся доступны снабжению

Снабженец создаёт новые подзадачи (stage=Снабжение)
  subtask_positions ссылаются на те же project_positions
  (группировка для закупки, не копия)

project_positions остаются едины (конвейер):
  были в спецификации (subtask проектирования)
  теперь в подзадаче закупки (subtask снабжения)
  → позиция помнит обе через связи subtask_positions
  → ИЛИ position.subtask_id обновляется на текущую активную?

⚠️ уточнить: subtask_id на позиции — это "где создана" (проектирование)
   или "где сейчас" (текущий этап)?
   Вероятно нужна связь many-to-many через subtask_positions
   (позиция была в спецификации, теперь в закупке — две связи)
```

## ОБНОВЛЁННАЯ СВЯЗЬ позиция ↔ подзадачи

```
project_positions — конвейер, едины
  kp_item_id     — связь с КП (один, практически всегда)
  (убираем subtask_id с позиции — связь через таблицу)

subtask_positions — many-to-many позиция ↔ подзадача
  id, subtask_id, project_position_id
  → позиция в спецификации (subtask проектирования)
  → та же позиция в закупке (subtask снабжения)
  → две записи, видна история прохождения по подзадачам

Вопрос: нужно ли ограничить — позиция в один момент в одной
активной подзадаче на этап? (в проектировании одна спец,
в снабжении одна закупка)
```
