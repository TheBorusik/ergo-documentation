# Архитектура конвейера — задачи в этапах, позиции текут сквозь

## ФУНДАМЕНТАЛЬНОЕ РЕШЕНИЕ

```
Задачи НЕ перемещаются между этапами.
Задача принадлежит своему этапу (фиксированный stage).
Позиция течёт сквозь этапы, меняя СТАТУС.
Результат задачи = изменение статуса позиции для следующего этапа.
```

---

## КАК РАБОТАЕТ КАЖДЫЙ ЭТАП

```
Этап = автономный обработчик:
  ВХОД:    берёт позиции в своём входном статусе
  РАБОТА:  создаёт свои задачи (подзадачи этапа)
  ВЫХОД:   результат задач меняет статус позиций → следующий этап
```

```
ПРОЕКТИРОВАНИЕ
  вход:   проект выигран
  работа: задачи-спецификации ("Спец. помещение 1")
          создают позиции, привязывают к КП
  выход:  позиции "спецификация готова"

СНАБЖЕНИЕ
  вход:   позиции "готовы к закупке"
  работа: задачи закупки ("Воздуховоды → Поставщик А")
          процесс: запрос→цены→согласование→оплата→доставка
  выход:  позиции "доставлена"

МОНТАЖ
  вход:   позиции "доставлена"
  работа: задачи работ ("Монтаж воздуховодов")
  выход:  позиции "установлена"

ДОКУМЕНТАЦИЯ
  вход:   позиции "установлена"
  работа: задачи документов (паспорта)
  выход:  позиции "задокументирована"

СДАЧА
  вход:   всё задокументировано
  выход:  проект завершён
```

---

## ПОЗИЦИЯ — единственное что течёт

```
project_positions (конвейер):
  status (конвейерный):
    created → spec_ready → in_purchase → purchased →
    delivered → mounting → installed → documented

  статус = где позиция в конвейере = какой этап с ней работает

Этап фильтрует позиции по входному статусу:
  Снабжение видит позиции status=spec_ready
  Работы видят status=delivered
```

---

## ЗАДАЧИ — в своих этапах, не двигаются

```
subtasks:
  stage_id    — ФИКСИРОВАН (задача принадлежит этапу)
  не меняется, задача не переезжает

Задачи проектирования остаются в проектировании (завершённые).
Задачи снабжения — в снабжении.
История чистая: видно какие задачи были на каждом этапе.
```

---

## СВЯЗЬ задача ↔ позиция (many-to-many + количество)

```
subtask_positions:
  id, subtask_id, project_position_id, quantity
  → quantity = сколько ИЗ позиции в этой задаче (дробление!)

Одна позиция → много задач разных этапов:
  Воздуховод ×14:
    задача проектир. "Спец. помещ.1" (создала, ×14)
    задача снабж. "Воздуховоды А" (×7)
    задача снабж. "Воздуховоды Б" (×7)  ← дробление по поставщикам
    задача работ "Монтаж" (×14)

→ видна история прохождения позиции по всем задачам
```

---

## ДРОБЛЕНИЕ ПО КОЛИЧЕСТВУ (решено)

```
Пример: часть воздуховодов на складе, часть заказать.
  Воздуховод ×14:
    ×4 на складе (статус: на складе)
    ×10 заказать (задача снабжения, quantity=10)

subtask_positions.quantity дробит позицию между задачами.
Сумма quantity по задачам этапа ≤ общее количество позиции.

Поставщики: часть у А, часть у Б — разные задачи, разные quantity.
Склад: часть со склада, часть заказать — аналогично.
```

---

## ПРЕИМУЩЕСТВА ЭТОЙ АРХИТЕКТУРЫ

```
✓ Задача = работа одного этапа (не размазана по этапам)
✓ stage_id фиксирован — нет миграции задач
✓ Позиция — единственный "движущийся" объект (чистый конвейер)
✓ Этапы автономны: вход(статус)→работа(задачи)→выход(статус)
✓ Дробление по количеству естественно (quantity в связи)
✓ История прозрачна: задачи каждого этапа на месте
✓ Параллельность: разные позиции в разных статусах одновременно
  (часть проекта в снабжении, часть в монтаже — само собой)
```

---

## СВЯЗЬ С КП (сохраняется)

```
Позиция привязана к kp_item (один, практически всегда).
КП видит свои позиции и их конвейерный статус.
Срез по КП = группировка позиций по kp_item_id.
Срез по задачам = группировка по subtask_positions.
```

---

## ОБНОВЛЁННАЯ МОДЕЛЬ

```
project_positions (конвейер — единый источник истины)
  id, project_id
  position_id    FK→element_positions (номенклатура)
  kp_item_id     FK→kp_items (связь с КП)
  quantity       общее количество
  status         конвейерный статус (created→...→installed)
  supplier_id    финальный поставщик (после выбора)
  warehouse_id   текущий склад (по умолч. склад проекта)
  created_at

subtasks (задачи этапов — не двигаются)
  id, project_id, parent_id, stage_id (ФИКСИРОВАН),
  name, status (new/in_progress/done/paused),
  stage_status (внутренний для этапа),
  responsible_id, created_at

subtask_positions (связь + дробление)
  id, subtask_id, project_position_id, quantity
```

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Статус позиции — автоматически от статуса задачи

```
Задача меняет статус → её позиции автоматически меняют статус.

"Воздуховоды Поставщик А" → "доставлена"
  → все позиции этой задачи → "доставлена" (авто)

Причины:
  ✓ позиций в задаче много — вручную неудобно
  ✓ монотонные действия = ошибки, автомат надёжнее
  ✓ снабженец работает с задачей, позиции следуют

Правило: смена stage_status задачи → маппинг на статус позиций.
  задача done/delivered → позиции delivered
  задача purchased → позиции purchased
```

### ② Только вперёд — без откатов

```
Конвейер ОДНОНАПРАВЛЕННЫЙ. Откатов назад нет.

Позиция не используется на объекте →
  не возврат в закупку, а КОНЕЧНЫЕ состояния:
    - утилизирована (брак/излишек)
    - на склад (остаток, перемещение)

Это терминальные статусы, не откат.

Статусы-ответвления (терминальные):
  ...→ installed (установлена) ✓ норма
  ...→ utilized (утилизирована)
  ...→ to_warehouse (остаток на склад)
```

### ⚠️ Нюанс — авто-статус при дроблении позиции

```
Проблема: позиция ×14 раздроблена:
  ×10 в задаче "Поставщик А"
  ×4 в задаче "со склада"

Задача А → "доставлена". Позиция целиком "доставлена"?
Но ×4 ещё на складе (другой статус).

РЕШЕНИЕ — статус на УРОВНЕ связи, не всей позиции:
  subtask_positions хранит свой статус для своей части
  ИЛИ
  при дроблении создаётся под-позиция (отдельная строка)

Вариант А: статус в subtask_positions
  ×10 (задача А) — доставлена
  ×4 (склад) — на складе
  общий статус позиции = агрегат (частично доставлена?)

Вариант Б: дробление = новые позиции
  Воздуховод ×10 (своя строка, свой статус)
  Воздуховод ×4 (своя строка)
  → каждая целиком в одной задаче, статус простой

→ УТОЧНИТЬ: статус на связи (агрегат) или дробление на строки?
  Вариант Б проще для конвейера (позиция целиком в одном статусе),
  но плодит строки.
  Вариант А компактнее, но статус позиции составной.
```

### Обновление модели — авто-статус

```
Правило синхронизации (бизнес-логика):
  при изменении subtasks.stage_status →
    обновить status у связанных project_positions
    (через subtask_positions)

Маппинг статусов задача→позиция:
  задача снабжения "delivered" → позиции "delivered"
  задача работ "installed" → позиции "installed"

Терминальные статусы позиции:
  installed | utilized | to_warehouse
  (только вперёд, без возврата)
```

---

## РЕШЕНИЕ ПО ДРОБЛЕНИЮ (финал) — Вариант Б

```
Дробление позиции → создаются ОТДЕЛЬНЫЕ позиции (строки).

Причина: у частей разная судьба:
  - разный поставщик
  - разная доставка (часть со склада снабженец сам,
    часть везёт поставщик)
→ это реально разные позиции, не одна составная

Воздуховод ×14 → снабженец дробит →
  Воздуховод ×10 (поставщик А, доставка поставщиком)
  Воздуховод ×4  (со склада, доставка снабженцем)

Каждая позиция:
  ✓ целиком в одной задаче этапа
  ✓ простой статус (вся позиция в одном статусе конвейера)
  ✓ авто-смена статуса работает чисто
  ✓ свой поставщик, свой склад, своя доставка
```

### Связь раздробленных позиций

```
project_positions:
  parent_position_id  FK→project_positions (из какой раздроблена, NULL если исходная)

Воздуховод ×14 (исходная, parent=NULL) — после дробления "архивна" или удаляется?
  → лучше: исходная заменяется дочерними
  Воздуховод ×10 (parent = исходная или просто отдельная)
  Воздуховод ×4  (parent = ...)

ИЛИ проще: дробление = редактирование количества исходной + новая строка
  Воздуховод ×14 → меняем на ×10, создаём новую ×4
  обе с одним kp_item_id (связь с КП сохраняется)

→ обе позиции помнят общий kp_item (раздел КП), это их связь
→ parent_position_id опционально для истории
```

### Итог по дроблению

```
Дробим когда у частей разная судьба (поставщик/склад/доставка).
Результат — отдельные project_positions строки.
Связь через общий kp_item_id (всегда) + parent_position_id (опц, история).
Каждая едет по конвейеру самостоятельно, статус простой и авто.
```

## ИТОГОВАЯ КОНВЕЙЕРНАЯ МОДЕЛЬ (сводка)

```
project_positions (единый источник, конвейер):
  id, project_id, position_id (номенклатура),
  kp_item_id (связь с КП, практически всегда),
  parent_position_id (если раздроблена, опц.),
  quantity,
  status (конвейерный: created→spec_ready→in_purchase→
          purchased→delivered→mounting→installed
          + терминальные: utilized, to_warehouse),
  supplier_id (финальный, после выбора),
  warehouse_id (текущий склад, по умолч. склад проекта),
  created_at

subtasks (задачи этапов, не двигаются):
  id, project_id, parent_id, stage_id (фиксирован),
  name, status, stage_status, responsible_id, created_at

subtask_positions (связь, БЕЗ дробления количества теперь):
  id, subtask_id, project_position_id
  → позиция целиком в задаче (дробление = отдельные позиции)
  → quantity НЕ нужен здесь (позиция уже нужного размера)

ПРАВИЛА:
  - задача меняет статус → её позиции авто меняют статус
  - конвейер только вперёд (откатов нет)
  - дробление = новые позиции (разная судьба)
  - терминальные статусы: installed / utilized / to_warehouse
```

Заметка: раз дробление = отдельные позиции, то subtask_positions
больше НЕ нужен quantity — позиция уже нужного размера, входит целиком.
Это упрощает связь до чистого many-to-one (позиция в одной активной
задаче этапа) или many-to-many (история по этапам).
