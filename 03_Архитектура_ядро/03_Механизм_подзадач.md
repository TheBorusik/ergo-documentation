# Универсальный механизм подзадач
## Единый паттерн "получил → раздробил → передал дальше"

---

## Главная идея

Каждый этап работает по ОДНОМУ механизму:
```
1. ПОЛУЧАЕТ подзадачу от предыдущего этапа
2. Создаёт СВОИ подзадачи (дробит как удобно)
3. ПЕРЕДАЁТ дочерние подзадачи на следующий этап
```

Один универсальный механизм вместо разных типов подзадач для каждого этапа.

---

## Поток

```
ПРОЕКТИРОВАНИЕ
  создаёт "Приточная установка П1" + список номенклатур
  [передать в снабжение]
        │
        ▼
СНАБЖЕНИЕ  получает "Приточная установка П1"
  дробит на свои подзадачи:
    "Воздуховоды П1", "Материалы П1", "Оборудование П1"
  закупает каждую
  [передать в работы]
        │
        ▼
РАБОТЫ  получает "Воздуховоды П1"
  дробит:
    "Монтаж воздуховодов П1", "Изоляция воздуховодов П1"
  выполняет
  [передать дальше]
```

---

## Рекурсивное дерево подзадач

```
"Приточная установка П1"  (проектирование)
  ├─ "Воздуховоды П1"  (снабжение)
  │    ├─ "Монтаж воздуховодов П1"  (работы)
  │    └─ "Изоляция воздуховодов П1"  (работы)
  ├─ "Материалы П1"  (снабжение)
  │    └─ "Монтаж материалов П1"  (работы)
  └─ "Оборудование П1"  (снабжение)
       └─ "Установка оборудования П1"  (работы)

Каждая подзадача знает:
  - откуда пришла (parent_id)
  - на каком этапе живёт (stage_id)
```

---

## Модель данных

```
subtasks (самоссылающееся дерево)
  id
  project_id          FK→projects
  parent_id           FK→subtasks (откуда пришла, null для корневых)
  stage_id            FK→project_stages (на каком этапе живёт)
  name                "Воздуховоды П1"
  status              микро-статус (новая/в работе/готова/передана)
  responsible_id      FK→users (ответственный)
  created_at

Содержимое зависит от этапа подзадачи:

subtask_positions (для снабжения/проектирования — номенклатуры)
  id, subtask_id, position_id (FK→element_positions), quantity

subtask_works (для работ — виды работ)
  id, subtask_id, work_id (FK→works), volume
```

---

## Передача между этапами

```
"Передать дальше" = породить дочернюю подзадачу на следующем этапе.

Подзадача А (снабжение) ──[передать]──► создаётся подзадача Б
                                         parent_id = А
                                         stage_id = Монтаж

Дробление: одна родительская → несколько дочерних
  "Воздуховоды П1" → "Монтаж" + "Изоляция" (две дочерние)
```

---

## Опережающий поток (сохраняется)

```
Проект ещё на этапе Проектирования,
но подзадачи "Воздуховоды П1" уже переданы в Снабжение и работают.

Подзадача живёт на СВОЁМ этапе (stage_id),
независимо от текущего этапа проекта целиком.

→ части бегут вперёд, не дожидаясь всего проекта
```

---

## Преимущества единого механизма

```
✓ Один паттерн на все этапы — проще кодить и понимать
✓ Рекурсивное дерево — естественно отражает дробление
✓ Гибкость — каждый этап дробит как ему нужно
✓ Прослеживаемость — видно всю цепочку от установки до конкретной работы
✓ Опережающий поток — подзадачи независимы от этапа проекта
✓ Масштабируемость — любая глубина вложенности
```

## Что показать в UI (предварительно)

```
На каждом этапе — список ПОЛУЧЕННЫХ подзадач (входящие от предыдущего)
  + возможность создать свои дочерние (дробление)
  + кнопка "передать дальше" по готовности

Дерево подзадач проекта — общая картина
  видно где какая часть находится
```

## Открытые вопросы

1. Микро-статусы подзадачи — единый набор для всех этапов
   или свой на каждом этапе?
   (снабжение: новая/в закупке/закуплена;
    работы: новая/в работе/выполнена)

2. Можно ли передать подзадачу частично?
   (закупили 5 из 10 → передать эти 5 в работы, остальное потом?)

3. Корневые подзадачи (на проектировании) — это укрупнённые
   блоки из КП? ("Приточная установка П1" из КП → подзадача)

4. Связь с наборами — подзадача = набор? Думаю да, это одно.

5. Содержимое передаётся вниз? (родитель "Воздуховоды" имеет
   позиции — дочерняя "Монтаж воздуховодов" видит их? или только работы?)

6. На этапе работ подзадача содержит работы. Откуда берутся виды
   работ — справочник? (нужен модуль "Справочник работ")

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Два уровня статусов

```
ОБЩИЙ статус (единый для всех подзадач, любой этап):
  new        — Новая
  in_progress— В работе
  done       — Выполнена
  paused     — Остановлена (пауза/проблема)
  → используется для обзора, канбана, фильтрации

ВНУТРЕННИЙ статус (свой набор на каждом этапе):
  Снабжение: new → purchasing → purchased
  Работы:    new → mounting → commissioning → handed
  → детализирует что именно на этапе

В модели:
  subtasks.status         — общий (new/in_progress/done/paused)
  subtasks.stage_status   — внутренний (зависит от stage_id)
```

### ② Без частичной передачи — только дробление

```
Передаётся всегда ЦЕЛАЯ подзадача (выполненная).
Нужно передать часть → раздробить:
  "Воздуховоды П1" закуплено 5 из 10 →
    дробим: "Воздуховоды П1 партия 1" (5, done) → в работы
            "Воздуховоды П1 партия 2" (5, in_progress)

→ нет частичной передачи, есть дробление. Меньше путаницы.
```

### ③ Всё привязано к родителю

```
parent_id обязателен для всех подзадач КРОМЕ корневых.
Корневые подзадачи создаются на этапе Проектирования.
Цепочка всегда прослеживается до корня:
  работа → снабжение → проектирование (корень)

subtasks.parent_id — NOT NULL кроме корневых (stage=Проектирование)
```

### Обновлённая модель

```
subtasks
  id
  project_id      FK→projects
  parent_id       FK→subtasks (NULL только для корневых на проектировании)
  stage_id        FK→project_stages
  name
  status          enum: new | in_progress | done | paused  (ОБЩИЙ)
  stage_status    enum: зависит от этапа (ВНУТРЕННИЙ)
  responsible_id  FK→users
  created_at

subtask_positions   — номенклатуры (проектирование/снабжение)
  id, subtask_id, position_id, quantity

subtask_works       — работы (этап работ)
  id, subtask_id, work_id, volume
```
