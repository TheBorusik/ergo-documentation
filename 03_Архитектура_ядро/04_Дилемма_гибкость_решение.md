# Разрешение дилеммы: партии vs позиции (гибкость)

## ДИЛЕММА (найдена пользователем)

```
"ПАРТИИ" (задача едет целиком)     "ПОЗИЦИИ" (гибкость)
✓ наглядно (помещение в снабж.)    ✓ гибко (готовое — дальше)
✓ понятно к чему задачи            ✓ доработки, добавления
✗ ждём всю партию (воздуховоды     ✓ объединять (2 помещения → 1 доставка)
  готовы, оборуд. нет — стоим)     ✗ теряется "где помещение"?
✗ негибко                          ✗ как снабжение поймёт что готово?
```

РЕШЕНИЕ: гибкость важнее (реальность непредсказуема).
Проблемы гибкого подхода решаются (ниже).

═══════════════════════════════════════════════════
КЛЮЧ — задачи НЕ едут, позиции текут
═══════════════════════════════════════════════════
```
Это наше фундаментальное решение (конвейер):
  задача НЕ едет по этапам
  задача создаёт/обрабатывает позиции
  позиции текут по конвейеру, меняя статус

Проектировщик создал спецификацию (задача) →
позиции получили статус "готова к закупке" →
задача ОСТАЁТСЯ в проектировании,
позиции живут дальше сами.
```

═══════════════════════════════════════════════════
СВЯЗЬ С ПОМЕЩЕНИЕМ — на позиции, не на задаче
═══════════════════════════════════════════════════
```
Позиция помнит откуда пришла:
  project_positions:
    spec_subtask_id  → из какой спецификации (помещение 1)
    kp_item_id       → к чему в КП (Приточная П1)
    status           → конвейерный статус

Связь с помещением НЕ теряется — она на позиции.
На любом этапе видно: эта позиция из помещения 1.
Даже если снабжение объединило с помещением 2.
```

═══════════════════════════════════════════════════
КАК СНАБЖЕНИЕ ПОНИМАЕТ ЧТО ГОТОВО
═══════════════════════════════════════════════════
```
Проектировщик проставляет позициям "готова к закупке"
(может ЧАСТЯМИ — не всю спецификацию сразу).

Снабжение видит ПУЛ готовых позиций (со всех спецификаций):
  Воздуховод ×14 (помещ.1) — готова
  Воздуховод ×20 (помещ.2) — готова
  Клапан ×2 (помещ.1) — готова
  Оборудование (помещ.1) — ещё проектируется (не в пуле)

Снабженец группирует СВОБОДНО:
  "все воздуховоды" (пом.1+пом.2) → один заказ, одна доставка
  "клапаны" → другой заказ

→ гибкость: объединение из разных помещений
→ связь с помещением у каждой позиции сохранена
```

═══════════════════════════════════════════════════
РЕШЕНИЕ ПРОБЛЕМЫ "ВОЗДУХОВОДЫ ГОТОВЫ, ОБОРУДОВАНИЕ НЕТ"
═══════════════════════════════════════════════════
```
Проблема (при партиях):
  воздуховоды приехали, оборудование нет,
  задача "помещение 1" не завершена → ждём монтаж

Решение (при позициях):
  воздуховоды (позиции) доставлены → доступны монтажу СРАЗУ
  оборудование (позиции) ещё в закупке → едет параллельно
  НЕ ждём. Монтируем воздуховоды, оборудование параллельно.

Позиции независимы. Нет застрявшей задачи.
```

═══════════════════════════════════════════════════
НАГЛЯДНОСТЬ — срез по помещению в "Позициях"
═══════════════════════════════════════════════════
```
Вместо "помещение едет по доске" → срез позиций по спецификации:

Экран "Позиции", группировка [По спецификации]:
  📁 Помещение 1
     Воздуховод ×14   🔨 монтаж
     Клапан ×2        🛒 закупка
     Оборудование ×1  📐 проектируется
  📁 Помещение 2
     Воздуховод ×20   🛒 закупка

Видно состояние помещения ДЕТАЛЬНО (по позициям):
что смонтировано, что закупается, что проектируется.
Нагляднее чем "помещение в снабжении" — видно ЧТО где.

Прогресс помещения = агрегат статусов его позиций
  (5 из 10 позиций установлено → помещение 50%).
```

═══════════════════════════════════════════════════
ИТОГ — как разрешается дилемма
═══════════════════════════════════════════════════
```
✓ Задачи НЕ едут (в этапах) — конвейер
✓ Позиции текут — единица движения
✓ Связь с помещением на позиции (spec_subtask_id) — не теряется
✓ Снабжение видит пул готовых → группирует свободно
✓ Объединение помещений в заказ — гибко
✓ Независимость позиций — нет застрявших партий
✓ Наглядность — срез по спецификации в "Позициях"

ГИБКОСТЬ + СВЯЗЬ + НАГЛЯДНОСТЬ одновременно.
Ключ: единица движения — ПОЗИЦИЯ, а помещение — это ЯРЛЫК на ней.
```

═══════════════════════════════════════════════════
ЧТО ЭТО ЗНАЧИТ ДЛЯ UI (уточнение разделов)
═══════════════════════════════════════════════════
```
Разделы этапов (Проектирование/Снабжение/Работы/Документы):
  показывают ЗАДАЧИ этого этапа (список)
  задачи НЕ едут между разделами — каждый создаёт свои

Раздел "Позиции" — ГЛАВНЫЙ для наглядности:
  все позиции, группировка по помещению/КП/статусу
  видно где что течёт

Готовность для этапа:
  проектировщик помечает позиции "готовы к закупке"
  снабжение в своём разделе при создании задачи
    выбирает из пула готовых (любые, из разных помещений)

Индикатор: счётчик готовых позиций у раздела Снабжение
  "12 позиций готовы к закупке"
```

═══════════════════════════════════════════════════
ОБНОВЛЕНИЕ МОДЕЛИ
═══════════════════════════════════════════════════
```
project_positions:
  spec_subtask_id  → из какой спецификации (связь с помещением)
  kp_item_id       → связь с КП
  status           → конвейерный (управляет готовностью к этапам)
  current_subtask_id → текущая занятость (эксклюзивность)

Статусы конвейера — точки готовности:
  spec_ready    → готова к закупке (снабжение берёт)
  delivered     → готова к монтажу (работы берут)
  installed     → готова к документации
  documented    → готова к сдаче

Проектировщик управляет переходом created→spec_ready
  (помечает "готова к закупке", может частями).
```
