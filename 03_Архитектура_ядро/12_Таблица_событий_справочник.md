ЛЕГЕНДА КОЛОНОК
═══════════════════════════════════════════════════
```
event_type  — константа-тип (что произошло)
entity      — entity_type: чего касается (что писать в entity_type)
from→to     — from_status / to_status (для смены статуса; "—" если нет)
reason      — заполняется ли причина (✓ текст = обязательно текстом)
detail      — ЧТО класть в detail (описание/данные события)

★ = критичное событие (точно сохранять)
? = причина под вопросом (может нужна)

ВСЕГДА заполняются (не в таблице, для всех):
  project_id, entity_id, changed_by, changed_at
```

═══════════════════════════════════════════════════
ИТОГО ПО ЗАПОЛНЕНИЮ
═══════════════════════════════════════════════════
```
from→to заполняется:  события смены статуса (позиции, этапы, задачи)
reason заполняется:   переоткрытие этапа, отклонение согласования
                      (везде где "откат/доработка" — нужна причина)
detail заполняется:   почти всегда (человекочитаемое описание)
                      поставщик/цена/склад/название/было→стало

Единый паттерн: каждое действие → одна запись event_log
с заполнением нужных колонок по этой таблице.
```
