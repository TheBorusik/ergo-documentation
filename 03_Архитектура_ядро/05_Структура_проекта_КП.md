# Единая структура проекта — КП как скелет, этапы наполняют деталями

---

## Главный принцип

```
КП (продажа) = СКЕЛЕТ проекта (верхнеуровневые блоки + цены заказчику)
Этапы = НАПОЛНЕНИЕ скелета деталями

Проект — единое дерево, которое растёт деталями по мере прохождения этапов.
Зайдя в проект, видишь ВСЁ со статусами каждой позиции.
```

---

## Связка двух идей (объединение)

```
КП-блок раскрывается в детали  +  подзадачи дробятся и движутся
                    ╲                    ╱
                     ╲                  ╱
              КП-блок = КОРНЕВАЯ подзадача

"Приточная установка П1" из КП = корневая подзадача проекта,
которую проектирование наполняет позициями,
а снабжение/работы дробят на дочерние.
```

---

## Три уровня структуры

```
УРОВЕНЬ 0 — КП (создаётся на продаже)
  Корневые подзадачи = блоки КП:
    Приточная установка П1     + цена заказчику
    Приточная установка П2     + цена
    Вытяжная установка В1      + цена
    Воздухораспределительная сеть
    Работы
    ...

УРОВЕНЬ 1 — наполнение (проектирование)
  Каждый блок раскрывается в номенклатуры:
    Приточная установка П1:
      Воздуховод 500х300 ×14
      Клапан ×2
      Вентилятор ×1

УРОВЕНЬ 2 — движение (снабжение → работы)
  Блоки дробятся на рабочие подзадачи:
    Приточная П1 → "Воздуховоды П1" (снабжение)
                 → "Оборудование П1" (снабжение)
                      → "Монтаж воздуховодов П1" (работы)
```

---

## Жизнь одной сущности "Приточная установка П1"

```
ПРОДАЖА:
  появилась в КП с ценой заказчику
  → корневая подзадача (stage=Проектирование, ждёт наполнения)

ПРОЕКТИРОВАНИЕ:
  наполнилась позициями (Воздуховод ×14, Клапан ×2...)
  → subtask_positions заполнены

СНАБЖЕНИЕ:
  раздроблена на "Воздуховоды П1", "Оборудование П1"
  → дочерние подзадачи (stage=Снабжение)
  позиции закупаются, виден статус каждой

МОНТАЖ:
  по закупленным частям → работы
  → дочерние подзадачи (stage=Работы)

ВСЁ ЭТО — одна ветка дерева проекта, растущая деталями.
```

## Что видно в проекте (единый список)

```
Зашёл в проект → дерево со статусами:

📁 Приточная установка П1          [снабжение 60%]
   ├─ Воздуховод 500х300 ×14       ✓ закуплено
   ├─ Клапан ×2                    🛒 в закупке
   └─ Вентилятор ×1                ⏳ не заказано
📁 Приточная установка П2          [проектирование]
   └─ (наполняется)
📁 Работы                          [монтаж]
   └─ Монтаж воздуховодов П1       🔨 в работе

В любой момент — какая позиция в каком статусе.
```

## Уточнение модели — КП-блок это подзадача

```
subtasks (корневые = блоки КП):
  "Приточная установка П1"
    parent_id = NULL (корневая)
    stage_id = ... (текущий этап этой ветки)
    is_kp_item = true (это блок из КП)
    customer_price = 495636 (цена заказчику из КП)

subtask_positions (наполнение проектированием):
  Приточная П1 → Воздуховод ×14, Клапан ×2...

Дочерние подзадачи (снабжение/работы):
  parent_id = Приточная П1
```

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Типы блоков — справочник (расширяемый)

```
block_types (справочник, расширяемый):
  Установка/оборудование → наполняется номенклатурами
  Сеть                   → номенклатуры
  Материалы              → номенклатуры
  Работы                 → работы
  Документация           → документы (возможно)
  ... могут добавляться новые

Тип определяет ЧЕМ наполняется блок (номенклатуры / работы / документы).
Не жёсткий enum, а справочник — заранее не знаем все типы.
```

### ② Работы — привязаны к части, видны общим списком

```
Работы СОЗДАЮТСЯ привязанными к подзадаче (parent_id):
  "Монтаж воздуховодов П1" → parent "Воздуховоды П1"

ОТОБРАЖАЮТСЯ двумя срезами (данные одни):
  СРЕЗ 1 — по установкам (дерево):
    Приточная П1 → работы по ней
  СРЕЗ 2 — блок "Работы" (плоский список ВСЕХ работ проекта):
    все подзадачи типа "работа" одним списком
    (соответствует строке "Работы" в КП)

Технически: блок "Работы" = представление-фильтр
  (все subtasks где тип содержимого = работы),
  а не отдельная сущность. Данные = подзадачи работ.
```

### ③ Блоки могут добавляться по ходу (не только из КП)

```
subtasks (корневые блоки) получают источник:
  source: 'kp'    — из коммерческого предложения (продажа)
  source: 'added' — добавлен по ходу (проектирование и далее)

→ видно что было в КП, что добавили потом
→ контроль бюджета: добавленное вне КП = отклонение от КП
```

### Обновление модели корневых подзадач (блоков)

```
subtasks (корневые = блоки):
  id, project_id
  parent_id = NULL (корневой блок)
  block_type_id   FK→block_types (установка/сеть/материалы/работы/документация)
  source          enum: kp | added
  name            "Приточная установка П1"
  customer_price  цена заказчику (из КП, для kp-блоков)
  status, stage_id, ...

block_types (справочник)
  id, name, content_kind (positions | works | documents)
  content_kind — чем наполняется (номенклатуры/работы/документы)
```
