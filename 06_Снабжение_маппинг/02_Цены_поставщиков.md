# Конвейер позиций + Цены поставщиков

---

## КОНЦЕПЦИЯ — позиция как конвейер (труба)

```
Позиция = единый источник истины.
Каждый этап НАПОЛНЯЕТ её данными по мере прохождения.

Проектирование → создана (kp_item, номенклатура, кол-во)
Снабжение      → + поставщик, цена, статус закупки, доставка, склад
Монтаж         → + статус работ (установлена/реализована)
Документация   → + паспорт/документ

Один источник, нет копий, нет рассинхрона.
```

### Как не "раздуть" таблицу позиции
```
ОСНОВНОЕ — в project_positions:
  текущий статус, поставщик, склад, количество

ДЕТАЛИ ЭТАПОВ — в связанных таблицах (ссылка на позицию):
  deliveries (доставка) → position_id
  works/montage (работы) → position_id
  → позиция хранит ИТОГ/ТЕКУЩЕЕ, детали в связях
```

### История переходов (если нужна хронология)
```
position_history (опционально)
  id, position_id, field, old_value, new_value, changed_by, changed_at
  → кто и когда менял статус позиции
  → позиция = текущее состояние, history = лог
```

### Минусы конвейера — управляемые
```
⚠️ Толстая таблица → детали в связанные таблицы
⚠️ История → отдельный position_history
⚠️ Позиция в разных этапах → это и есть конвейер, не проблема
ПЛЮСЫ перевешивают: один источник, нет рассинхрона, вся картина в одном месте
```

## ЦЕНЫ ПОСТАВЩИКОВ — отдельная подсистема

### Факты (со слов снабжения)
```
- Цены меняются ~3 раза в год
- Снабженец ВСЕГДА запрашивает свежую перед заказом
- Хранится последняя известная цена по позиции поставщика
- Несколько поставщиков на одну нашу позицию
- Выбор поставщика по цене → запрос свежей → обновление
```

### Процесс запроса цен
```
1. Снабженец отправляет НАШ список позиций поставщику
2. Поставщик присылает СВОЙ список (соответствует нашему по порядку)
3. Снабженец проставляет соответствия (легко — списки совпадают)
4. Сохраняется:
   - соответствие: наша позиция ↔ позиция поставщика (название) + цена
   - маппинг (обучение): текст → наша позиция
5. Цена обновляется (новый запрос = новая цена)
```

### Две вещи сохраняются раздельно
```
СООТВЕТСТВИЕ ПОЗИЦИЙ (постоянное):
  наша позиция ↔ позиция поставщика (с названием поставщика)
  → для будущего: видеть какая у поставщика позиция соответствует нашей
  → у одной нашей позиции — соответствия с разными поставщиками

МАППИНГ (обучение распознаванию):
  raw_text поставщика → наша позиция
  → supplier_mappings (модуль номенклатуры)
  → автораспознавание прайсов
```

## МОДЕЛЬ ДАННЫХ — цены

```
supplier_position_links (соответствие наша ↔ поставщика)
  id
  our_position_id    FK→element_positions (наша эталонная номенклатура)
  supplier_id        FK→contractors
  supplier_name      "Возд. оцинк. 500x300" (как у поставщика)
  supplier_article   артикул поставщика (если есть)
  created_at
  → постоянная связь, у нашей позиции много поставщиков

supplier_prices (история цен)
  id
  link_id            FK→supplier_position_links
  price              цена
  currency
  quoted_at          когда получена
  request_id         FK→supplier_requests (из какого запроса)
  is_current         актуальная ли (последняя)
  → история: цена менялась 3 раза в год, храним все, флаг текущей
```

### Связь с маппингом (модуль номенклатуры)
```
supplier_position_links перекликается с supplier_mappings:
  supplier_mappings: raw_text → position (распознавание)
  supplier_position_links: позиция ↔ поставщик + название + артикул

Возможно объединить или связать:
  link = подтверждённое соответствие (постоянное, с поставщиком)
  mapping = обучающие данные (тексты для автораспознавания)
  → link порождается при подтверждении, mapping копит варианты текста
```

## КАК СНАБЖЕНЕЦ ВЫБИРАЕТ ПОСТАВЩИКА ПО ЦЕНЕ

```
Наша позиция "Воздуховод 500х300"
  ← supplier_position_links:
      Поставщик А: "Возд. оц 500х300" — последняя цена 1200 (2 мес назад)
      Поставщик Б: "Воздуховод оцинк." — последняя цена 1150 (4 мес назад)
      Поставщик В: "ВЗ-500-300" — последняя цена 1300 (1 мес назад)

Снабженец видит последние цены → выбирает Б (дешевле) →
запрашивает свежую (вдруг изменилась) →
обновляет supplier_prices (новая запись, is_current)
```

## ОБНОВЛЕНИЕ РЕШЕНИЙ ПО НЕСТЫКОВКАМ

```
⚠️6 supplier_id → C (request=кому слали, позиция=финальный) ✓
⚠️7 статус → C (подзадача=процесс, позиция=наличие) + конвейер ✓
   позиция наполняется статусами по этапам (труба)
⚠️8 маппинг/цены → РАЗДЕЛЕНО:
   supplier_position_links (соответствие+название)
   supplier_prices (история цен)
   supplier_mappings (обучение распознаванию)
   три вещи, связаны, но разные роли
⚠️9 склад → виртуальный склад проекта ПО УМОЛЧАНИЮ
   позиция меняет склад → указывается явно (перемещение/остатки)
```
