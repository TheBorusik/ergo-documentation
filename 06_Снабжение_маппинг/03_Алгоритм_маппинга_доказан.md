# Алгоритм маппинга прайсов — на основе реальных примеров

## ДОКАЗАНО НА ПРАКТИКЕ

Тест: наш запрос (Excel, 9 позиций) ↔ ответ Центавр (PDF, 8 позиций)
```
8 из 9 сматчились автоматически:
  7 позиций — 100% (после нормализации)
  1 позиция (саморез) — 64% fuzzy (поставщик переформулировал)
  1 позиция (стул) — НЕ найдена (её нет у поставщика — норма)
```

## КЛЮЧЕВОЕ ОТКРЫТИЕ

```
Поставщик отвечает НА наш запрос → копирует наши названия.
→ названия почти идентичны, разница в мелочах:
    "1 шт. в уп." vs "1шт в уп"  (пробелы, точки)
    регистр, запятые/точки в числах

→ после НОРМАЛИЗАЦИИ большинство = точное совпадение
```

## АЛГОРИТМ МАППИНГА (многоуровневый)

### Шаг 0 — Нормализация строки
```
normalize(s):
  - lower case
  - убрать лишние пробелы (множественные → один, trim)
  - убрать/унифицировать пунктуацию (. , () кавычки)
  - унифицировать числа (5.5 = 5,5 → 5.5)
  - убрать единицы упаковки для сравнения ("1шт в уп")?
    (осторожно — может нести смысл)

"Шинорейка 20 (3м) УГФ 2,5мм 1 шт. в уп."
  → "шинорейка 20 3м угф 2,5мм 1шт в уп"
```

### Шаг 1 — EXACT по артикулу (если есть)
```
Если у поставщика есть артикул И у нас есть его артикул →
  точное совпадение артикулов → мгновенный матч
(в этих примерах артикулов нет, но у оборудования есть:
 ВЕРОСА-500-193..., МАКК 320-1102)
```

### Шаг 2 — EXACT по нормализованному названию
```
normalize(наше) == normalize(поставщика) → матч 100%
Покрывает большинство (поставщик копирует наш запрос).
```

### Шаг 3 — По истории маппинга (обучение)
```
Раньше уже мапили "стул в бытовку" на позицию X у этого поставщика →
supplier_mappings помнит → автоматом.
```

### Шаг 4 — FUZZY (нечёткое)
```
SequenceMatcher / расстояние Левенштейна / токены.
Порог ~0.6:
  "саморез кровельный удлиненое сверло" ↔
  "саморез кровельный с увеличенным сверлом" → 64% → предложить
Показываем снабженцу кандидатов, он подтверждает.
```

### Шаг 5 — Ручной выбор
```
Не нашлось → снабженец вручную выбирает нашу позицию
(или отмечает "нет у поставщика").
Выбор → запоминается (обучение на будущее).
```

## ЧТО СОХРАНЯЕМ ДЛЯ БУДУЩЕГО МАППИНГА

### supplier_position_links (постоянное соответствие)
```
id
our_position_id     наша позиция (element_position)
supplier_id         поставщик (Центавр)
supplier_name       "Шинорейка 20 (3м) УГФ 2,5мм 1шт в уп" (как у них)
supplier_article    артикул поставщика (если есть, тут нет)
created_at, confirmed_by

→ в след. раз: этот поставщик, эта строка → сразу наша позиция
```

### supplier_mappings (обучение распознаванию)
```
id
supplier_id
raw_text            нормализованный текст поставщика
raw_text_original   оригинал (для показа)
position_id         наша позиция
match_type          exact | fuzzy | manual
confidence
times_used          счётчик (сколько раз подтверждён)
last_used_at

→ накапливает варианты написания одной позиции
→ чем чаще подтверждают, тем увереннее автоматом
```

### supplier_prices (история цен)
```
id
link_id             FK→supplier_position_links
price               127.00
currency            RUB
quoted_at           дата счёта (19.06.2026)
source_document     файл-счёт
is_current
```

## ПОЛНЫЙ ПОТОК (на примере Центавр)

```
1. Отправили запрос (Excel) Центавру — 9 позиций
2. Центавр прислал счёт (PDF) — 8 позиций с ценами
3. Парсинг PDF (C# PdfPig) → 8 строк
4. Маппинг каждой строки:
   Шаг 2 (exact норм.): 7 позиций → 100%
   Шаг 4 (fuzzy): саморез → 64% → снабженец подтверждает
   "стул" из нашего запроса — нет в ответе (ок, у другого купим)
5. Сохранение:
   - supplier_position_links: 8 соответствий Центавр
   - supplier_mappings: 8 записей (обучение)
   - supplier_prices: 8 цен от 19.06.2026
6. След. раз запрос Центавру → эти позиции распознаются сразу
   (по supplier_position_links), запросим только свежую цену
```

## ВАЖНЫЕ ДЕТАЛИ ИЗ ПРИМЕРОВ

### Единица и упаковка
```
Наш запрос: "уп" (упаковка), кол-во 30 уп
Ответ: 30 шт по 127 (но "1шт в уп" = упаковка)
       Уголок: наш 2 уп → их 200 шт (100шт в уп × 2)

⚠️ Единицы РАСХОДЯТСЯ: мы в упаковках, они в штуках!
   "Уголок 100шт в уп" наш 2 (уп) → их 200 (шт)
   → нужно учитывать фасовку при сверке количества
   → цена у них за ШТ (15р), не за упаковку
```

### Разные поставщики — разные счета
```
Газзачас: газы (кислород, азот) + доставка отдельной строкой
Центавр: монтажные материалы (шинорейка, траверса, саморез)
→ один заказ мог разойтись по разным поставщикам
→ каждый прислал свой счёт (PDF)
→ подтверждает: несколько файлов/поставщиков на запрос
```

### Доставка как строка
```
Газзачас: "Доставка ... 1 рейс 4508" — отдельная позиция счёта
→ доставка это тоже строка, но НЕ наша номенклатура
→ маппить на спец-позицию "Доставка" или отдельный учёт?
```

## РЕАЛИЗАЦИЯ FUZZY — на сервере (C#)

```
Fuzzy-сравнение на БЭКЕ (C#), единообразно со всем маппингом.

Причины:
  - весь маппинг (exact/fuzzy/обучение) в одном месте
  - fuzzy работает с данными из БД (история, supplier_mappings)
  - обучение (запоминание) — на сервере
  - предфильтрация через SQL перед fuzzy (производительность)

Библиотеки C#:
  FuzzySharp — порт FuzzyWuzzy, Fuzz.Ratio() → 0-100
  F23.StringSimilarity — Левенштейн, Джаро-Винклер и др.

Поток:
  СЕРВЕР: парсинг → нормализация → exact → fuzzy (среди
          предфильтрованных кандидатов) → кандидаты с %
  ФРОНТ:  показать (100% авто / 64% "подтвердить?")
  СЕРВЕР: сохранить подтверждённое (обучение)

Оптимизация:
  грубый фильтр SQL (категория, первое слово) →
  fuzzy только среди подходящих (не все 5000 позиций)
```
