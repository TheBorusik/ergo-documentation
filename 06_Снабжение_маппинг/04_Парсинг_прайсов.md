# Модуль парсинга прайсов поставщиков

## ЗАДАЧА
Поставщики присылают прайсы в РАЗНЫХ форматах (PDF, Excel, Word, фото).
Нужно извлечь позиции (название, цена, кол-во, артикул) и смапить на наши.

---

## СТРАТЕГИЯ — многоуровневый механизм

```
Уровень 1 — АВТОПАРСИНГ (эвристики + извлечение таблиц)
  система сама находит таблицу и колонки
      ↓ если не уверена или новый поставщик
Уровень 2 — ВИЗУАЛЬНАЯ РАЗМЕТКА (пользователь выделяет)
  файл на экране, пользователь размечает колонки мышкой
      ↓ разметка сохраняется
Уровень 3 — ШАБЛОН ПОСТАВЩИКА (запоминание)
  следующий файл того же поставщика → авторазметка по шаблону
      ↓
Уровень 4 — AI-ПАРСИНГ (сложные случаи)
  нестандартный формат → Claude извлекает структуру
```

---

## УРОВЕНЬ 1 — Автопарсинг

### Извлечение по типу файла
```
PDF  → pdfplumber (таблицы) / pdftotext (текст)
Excel → openpyxl / pandas (готовая таблица)
Word → python-docx (таблицы)
CSV  → прямой парсинг
Фото/скан → OCR (tesseract) → текст
```

### Эвристики распознавания колонок
```
Колонка ЦЕНА:
  числа с разделителями (494 289,18 / 1 053 171.90)
  рядом валюта (руб, RUB, ₽)

Колонка КОЛИЧЕСТВО:
  малые целые/дробные числа
  рядом единица (шт, м, кг, компл)

Колонка НАЗВАНИЕ:
  самый длинный текст в строке
  содержит буквы + артикулы

Колонка АРТИКУЛ:
  паттерн код (ВЕРОСА-500-193, ГЕРМИК-П-400)
  буквы-цифры-дефисы

Колонка № / примечание:
  короткие (1,2,3 / П1,П2)
```

### Результат автопарсинга
```
Система показывает распознанное:
  ✓ уверенно (зелёным)
  ? под вопросом (жёлтым — проверь)
Пользователь подтверждает или переходит к разметке.
```

## УРОВЕНЬ 2 — Визуальная разметка (ядро UX)

### Как работает
```
1. Файл рендерится на экране:
   PDF → изображение страниц / HTML
   Excel → таблица (сетка)
   Word → таблица

2. Пользователь ВЫДЕЛЯЕТ:
   - область таблицы (рамкой)
   - кликает на колонку → назначает роль:
       [Название] [Цена] [Количество] [Единица] [Артикул] [Примечание]

3. Система размечает по назначению колонок:
   колонка 2 = Название, колонка 5 = Цена...

4. Предпросмотр извлечённых позиций → подтверждение
```

### UI визуальной разметки
```
┌─ Прайс ВЕЗА.pdf ────────────────────────────┐
│  № │ Товар          │Кол│Ед│ Цена    │Прим  │
│  1 │ Кондиционер... │ 1 │шт│494289.18│ П1   │
│  2 │ Шкаф...        │ 1 │шт│ 78138.00│ П1   │
│    ▲                 ▲      ▲          ▲     │
│  [№] [Название]  [Кол][Ед][Цена]  [Примеч] │← клик назначает роль
└──────────────────────────────────────────────┘
  Назначь роли колонок кликом на заголовок
  [Извлечь позиции →]
```

### Для Excel/таблиц — проще
```
Таблица уже структурна → пользователь кликает заголовки колонок
→ назначает роли → готово
```

### Для PDF без чёткой таблицы
```
Выделение области + указание разделителей
ИЛИ сразу Уровень 4 (AI)
```

## УРОВЕНЬ 3 — Шаблон поставщика (запоминание)

### Сохранение разметки
```
supplier_parse_templates
  id
  supplier_id       FK→contractors
  file_type         pdf | excel | word
  column_mapping    JSON: какая колонка = какая роль
                    {name: 2, price: 5, qty: 3, unit: 4, note: 7}
  table_area        координаты/лист таблицы (если нужно)
  header_row        номер строки заголовка
  created_at, updated_at
```

### Применение
```
Пришёл файл от ВЕЗА →
  система находит шаблон ВЕЗА →
  применяет разметку автоматически →
  показывает результат (проверь) →
  подтверждение
```

### Обучение
```
Поставщик поменял формат → разметка не сошлась →
пользователь переразмечает → шаблон обновляется.
```

## УРОВЕНЬ 4 — AI-парсинг (сложные случаи)

```
Нестандартный формат / скан / хаос →
  файл (текст/изображение) → Claude API →
  "Извлеки позиции: название, цена, кол-во, артикул. Верни JSON" →
  структурированный результат → проверка пользователем

Применять когда:
  - автопарсинг не справился
  - визуальная разметка невозможна (сложный layout)
  - разовый нестандартный поставщик
```

## СВЯЗЬ С МАППИНГОМ (наш модуль номенклатуры)

```
После извлечения позиций поставщика →
маппинг на наши позиции (трёхуровневый: exact→pattern→fuzzy):

Прайс ВЕЗА (извлечено):        Наши позиции:
  "Кондиционер ВЕРОСА-500..."  → ? "Приточная установка П1"
  "Клапан ГЕРМИК-П-400*400"    → ? "Клапан противопожарный 400х400"

  1. exact — точное совпадение по прошлым маппингам
  2. pattern — по паттернам (артикул, размеры)
  3. fuzzy — нечёткое (похожие названия)
  → снабженец подтверждает

Подтверждённое → supplier_position_links + supplier_prices
```

## ПОЛНЫЙ ПОТОК

```
1. Снабженец загружает файл прайса (PDF/Excel/Word)
2. Система определяет тип → пробует автопарсинг (Ур.1)
   ИЛИ находит шаблон поставщика (Ур.3)
3. Если не вышло → визуальная разметка (Ур.2)
   ИЛИ AI (Ур.4)
4. Извлечены позиции поставщика (название, цена, кол-во)
5. Автомаппинг на наши позиции (наш модуль)
6. Снабженец подтверждает соответствия
7. Сохраняется:
   - supplier_position_links (соответствие + название поставщика)
   - supplier_prices (цены + дата)
   - supplier_parse_template (разметка на будущее)
8. Цены доступны для выбора поставщика
```

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Форматы — любые, приоритет Excel/PDF/Word
```
Приоритет реализации: Excel → PDF → Word → остальное
Excel проще всего (структурная таблица)
Сканы/фото — редко, OCR по необходимости
```

### ② Маппинг по артикулу И названию
```
Артикулы у поставщиков непредсказуемые →
мапим по ОБОИМ полям:
  - артикул (если распознан) — точное совпадение
  - название — нечёткое совпадение
Оба сохраняются в supplier_position_links.

price_import_lines: raw_name + raw_article (оба)
маппинг пробует: артикул → потом название → потом fuzzy
```

### ③ AI-парсинг — вероятно основной в будущем
```
Импорт прайса — НЕ частая операция (несколько раз при закупке).
→ можно позволить AI (Claude) даже как основной способ.
Стоимость оправдана редкостью операции + экономией времени.

Стратегия:
  сейчас — правила + визуальная разметка + AI для сложных
  будущее — AI как основной, правила как fallback/удешевление
```

### ④ Каталог vs прайс — двухфазный подход (если есть каталог)
```
ГИПОТЕЗА (проверить на практике):
  У некоторых поставщиков есть ОБЩИЙ КАТАЛОГ (без цен).

Двухфазный подход:
  Фаза 1: загрузить каталог поставщика (без цен) ОДИН раз
          → установить соответствия наша ↔ поставщика заранее
          → supplier_position_links наполняется
  Фаза 2: прайс с ценами → сопоставление с готовым каталогом
          → маппинг почти авто (соответствия есть)
          → подставляем только цены

Разделение природы данных:
  Каталог (редко меняется) → ЧТО есть у поставщика (соответствия)
  Прайс (3 раза/год) → ПОЧЁМ (только цены)

НО: не у всех есть каталог →
  система работает и БЕЗ каталога (прайс парсится напрямую,
  соответствия устанавливаются по ходу).

→ каталог = ускоритель, не обязательное условие
```

### Обновление модели — каталог
```
supplier_catalogs (опциональный каталог поставщика)
  id, supplier_id, imported_at
  → загружается один раз, без цен

supplier_catalog_items
  id, catalog_id, supplier_name, supplier_article,
  matched_position_id (соответствие нашей позиции)
  → соответствия наша ↔ поставщика (заранее)

При импорте ПРАЙСА:
  если есть каталог → сопоставляем с ним (быстро, по артикулу/названию)
  если нет → маппинг напрямую (трёхуровневый)

supplier_position_links может наполняться:
  - из каталога (Фаза 1)
  - из прайса напрямую (без каталога)
```

### Итоговая стратегия парсинга
```
ПРИОРИТЕТ СПОСОБОВ (по мере развития):
  1. Есть каталог поставщика → прайс мапится по каталогу (легко)
  2. Есть шаблон разметки → авторазметка формата
  3. AI-парсинг (Claude) → универсально, для важных/сложных
  4. Визуальная разметка → ручной контроль
  5. Автопарсинг эвристиками → быстрый черновик

Все ведут к: извлечённые позиции + цены → маппинг → supplier_prices
```

## ТЕХНИЧЕСКАЯ РЕАЛИЗАЦИЯ — показ и разметка форматов

### Ключевой принцип
```
Для разметки колонок нужны ДАННЫЕ, не просто картинка.
→ конвертируем каждый формат в структуру (таблицу/HTML),
  затем рисуем сами и накладываем выделение колонок.
НЕ используем "вьюеры-картинки" (они запирают данные).
```

### По форматам

```
EXCEL (.xlsx/.xls) — проще всего
  библиотека: SheetJS (xlsx) — читает в браузере
  файл → массив ячеек [[...],[...]] → рисуем HTML <table>
  пользователь кликает заголовок → назначает роль колонки
  данные УЖЕ структурны, парсить не нужно

PDF — двухсоставно
  фон: pdf.js — рендер страницы (пользователь видит оригинал)
  данные: pdfplumber (Python, бэкенд) — извлечение таблиц
          (проверено на ВЕЗА — таблица извлеклась чисто)
  показываем распознанную таблицу, пользователь корректирует границы

WORD (.docx)
  библиотека: mammoth.js или docx-preview → HTML
  находим <table> в результате → размечаем колонки

CSV — тривиально
  прямой парсинг → таблица

СКАН/ФОТО — OCR
  tesseract → текст → эвристики/AI
  (редкий случай)
```

### Готовые вьюеры — когда уместны
```
react-doc-viewer / Nutrient / Apryse — показать документ "как есть"
Уместно для:
  - просмотр приложенных договоров, чертежей (не парсим)
  - предпросмотр без извлечения
НЕ для разметки колонок (данные заперты в рендере).
```

### Архитектура парсинга (клиент + сервер)
```
КЛИЕНТ (браузер):
  Excel → SheetJS (в браузере, без сервера)
  CSV → парсинг в браузере
  PDF → pdf.js рендер (показать)
  Word → mammoth (показать + таблицы)

СЕРВЕР (бэкенд Python/Node):
  PDF таблицы → pdfplumber (точное извлечение)
  OCR → tesseract
  AI-парсинг → Claude API
  сложные форматы → серверная обработка
```

### Поток разметки в UI
```
1. Загрузка файла → определение типа
2. Конвертация в таблицу (SheetJS/pdfplumber/mammoth)
3. Рендер как интерактивная HTML-таблица
4. Пользователь кликает заголовки → назначает роли
   [Название][Цена][Кол-во][Ед][Артикул][Примечание]
5. Извлечение позиций по разметке
6. Сохранение шаблона (supplier_parse_templates)
7. → маппинг на наши позиции
```

### Библиотеки (стек)
```
Frontend (React):
  xlsx (SheetJS) — Excel в браузере
  pdfjs-dist — рендер PDF
  mammoth / docx-preview — Word в HTML
  papaparse — CSV

Backend (Python, если нужен):
  pdfplumber — таблицы из PDF (лучшее для нашего кейса)
  openpyxl — Excel на сервере
  python-docx — Word
  pytesseract — OCR
  anthropic — AI-парсинг
```

## УТОЧНЕНИЕ СТЕКА — Backend на C#

### Парсинг на C# (не на вебе)
```
Бэкенд C# → весь парсинг там (библиотеки не хуже Python):

Excel:  ClosedXML / EPPlus / NPOI
PDF:    PdfPig / iText7 (извлечение таблиц)
Word:   Open XML SDK (DocumentFormat.OpenXml) / NPOI
OCR:    Tesseract .NET
AI:     Anthropic .NET SDK / HTTP к Claude API
```

### Распределение фронт/бэк
```
FRONTEND (React) — тонкий, показ + разметка:
  - показать файл (pdf.js рендер страницы, или превью)
  - показать распознанную таблицу (пришла с бэка)
  - пользователь кликает колонки → назначает роли
  - отправляет разметку на бэк

BACKEND (C#) — вся обработка:
  - принять файл
  - распарсить (ClosedXML / PdfPig / OpenXML)
  - применить разметку колонок
  - извлечь позиции
  - маппинг на наши позиции
  - сохранить цены (supplier_prices)
```

### Поток (уточнённый под C#)
```
1. React: загрузка файла → POST на C# бэк
2. C#: парсинг → распознанная таблица → JSON → React
3. React: показ таблицы → пользователь размечает колонки
4. React: разметка (какая колонка = роль) → POST на C#
5. C#: извлечение позиций по разметке →
       маппинг (трёхуровневый) →
       сохранение supplier_position_links + supplier_prices
6. C#: сохранение шаблона разметки (supplier_parse_templates)
```

### Решение — ВСЁ на сервере (единообразно)
```
Весь парсинг на C# бэке. Фронт НЕ парсит.
Причина: единообразие важнее экономии на предпросмотре.
  - один код парсинга (C#), не дублируется в JS
  - одинаковое поведение для всех форматов
  - проще поддержка и отладка

Фронт (React):
  - показывает файл (pdf.js только РЕНДЕР картинки страницы)
  - показывает таблицу, пришедшую с бэка
  - разметка колонок (клики)
  НЕ извлекает данные сам.
```

### AI-парсинг на C#
```
Anthropic предоставляет .NET SDK (или прямой HTTP к API).
C# отправляет текст/файл → Claude → структурированный JSON →
обработка на бэке.
→ AI-парсинг тоже на бэке, единообразно с остальным.
```

### Итого — стек проекта
```
Frontend: React (показ файла, разметка колонок, UI)
Backend:  C# (.NET) — ВСЯ логика, парсинг, маппинг
БД:       PostgreSQL
Парсинг:  C# библиотеки (ClosedXML/EPPlus, PdfPig/iText7,
          Open XML SDK, Tesseract .NET) — ВСЁ на сервере
AI:       Anthropic .NET SDK (парсинг сложных прайсов)

Принцип: фронт тонкий (UX), весь парсинг и логика на C#.
```

## УТОЧНЕНИЕ — несколько файлов в одном ответе

### Реальность
```
Поставщик может прислать НЕСКОЛЬКО файлов на один запрос цен:
  оборудование.xlsx + материалы.pdf + трассы.xlsx
  (разбивка по типам, заводам, или как удобно поставщику)

Форматы файлов могут отличаться внутри одного ответа
  (один Excel, другой PDF).
```

### Уровни модели
```
supplier_request (один запрос поставщику)
  └─ supplier_quote (ответ — контейнер)
       ├─ price_import_file 1 (оборудование.xlsx)
       │    └─ lines (позиции файла 1)
       ├─ price_import_file 2 (материалы.pdf)
       │    └─ lines (позиции файла 2)
       └─ price_import_file 3 (трассы.xlsx)
            └─ lines (позиции файла 3)

Каждый файл:
  - парсится отдельно (свой формат, своя разметка)
  - свой шаблон разметки (может быть разный)
Позиции всех файлов → сливаются в общий список → маппинг.
```

### Обновлённая модель данных
```
supplier_quotes (ответ на запрос — контейнер)
  id, request_id, received_at, status

price_import_files (файлы ответа — НЕСКОЛЬКО на quote)
  id, quote_id (FK→supplier_quotes),
  file_path, file_type (excel/pdf/word),
  parse_status, template_id, uploaded_at

price_import_lines (позиции из файла)
  id, file_id (FK→price_import_files),
  raw_name, raw_article, price, quantity, unit, note,
  matched_position_id, match_confidence
```

### UI — загрузка нескольких файлов
```
Экран ответа поставщика:
  [+ Добавить файл] — можно загрузить несколько
  каждый файл:
    - показывается отдельно
    - размечается (своя разметка колонок)
    - парсится → позиции
  Все позиции → общий список → маппинг на наши

Загрузка: drag-and-drop нескольких файлов сразу
  или по одному через [+ Добавить файл]
```

### Поток (уточнённый)
```
1. Снабженец получил ответ поставщика (несколько файлов)
2. Загружает ВСЕ файлы в quote (один ответ)
3. Каждый файл: парсинг (C#) → разметка если нужно → позиции
4. Позиции всех файлов сливаются в общий список
5. Маппинг на наши позиции
6. Подтверждение → цены сохранены
```
