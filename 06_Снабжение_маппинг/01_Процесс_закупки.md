# Процесс закупки (снабжение) — детальный

---

## Полный жизненный цикл подзадачи снабжения

```
1. ПОЛУЧЕНА
   снабжению приходит задача = список позиций (от проектирования)

2. РАЗБИВКА
   снабженец разбивает на подзадачи (любое кол-во)
   назначает поставщика каждой (или оставляет существующего)
   → одна задача → N подзадач по разным поставщикам

3. ОТПРАВЛЕНА ПОСТАВЩИКУ
   список позиций → поставщику (по почте/email)

4. ПОЛУЧЕНЫ ЦЕНЫ
   поставщик присылает список с ценами

5. МАППИНГ (соответствия)
   снабженец сопоставляет наши номенклатуры ↔ позиции поставщика
   → это модуль маппинга!

6. НА СОГЛАСОВАНИИ
   отправляет менеджеру на согласование цены

7. СОГЛАСОВАНА
   менеджер согласовал цену

8. НА ОПЛАТЕ
   отправлено в бухгалтерию

9. ОПЛАЧЕНА
   бухгалтерия оплатила → ответила "готово"

10. СРОКИ УСТАНОВЛЕНЫ
    снабженец проставил сроки готовности (от поставщика)

11. ГОТОВА У ПОСТАВЩИКА
    позиции изготовлены/готовы

12. В ДОСТАВКЕ
    договорился о доставке на объект

13. ДОСТАВЛЕНА
    позиции на объекте → статус "доставлено"

14. ГОТОВО (закрыта)
    подзадача закрыта со статусом "готово"
```

---

## Внутренние статусы подзадачи снабжения

```
stage_status для этапа Снабжение:
  received      — получена
  assigned      — распределена (выбран поставщик)
  sent          — отправлена поставщику (запрос цен)
  priced        — получены цены
  mapping       — сопоставление номенклатур
  approval      — на согласовании (менеджер)
  approved      — согласована
  payment       — на оплате (бухгалтерия)
  paid          — оплачена
  production    — в производстве (сроки от поставщика)
  ready         — готова у поставщика
  shipping      — в доставке
  delivered     — доставлена
  done          — готово (закрыта)
```

Общий статус (subtasks.status) при этом:
```
new → in_progress (всё время работы) → done (закрыта)
paused — если застряла (например поставщик не отвечает)
```

## Роли в процессе

```
Снабженец    — ведёт подзадачу через все статусы
Менеджер     — согласует цену (статус approval → approved)
Бухгалтерия  — оплачивает (статус payment → paid)
```

→ нужны роли users: снабженец, менеджер, бухгалтер

## Новые сущности

```
supplier_request (запрос поставщику)
  id, subtask_id, supplier_id, sent_at, status
  → отправленный список позиций

supplier_quote (ответ с ценами)
  id, request_id, received_at
  quote_lines: позиция поставщика + цена

quote_mapping (соответствия — модуль маппинга)
  наша позиция ↔ строка прайса поставщика
  (используем supplier_mappings из модуля номенклатуры!)

approval (согласование)
  id, subtask_id, manager_id, approved_at, status

payment (оплата)
  id, subtask_id, accountant_id, paid_at, amount

delivery (доставка)
  id, subtask_id, ready_date, delivered_at,
  warehouse_id (куда доставлено)
```

## Связь с маппингом (модуль номенклатуры)

```
Шаг 5 (маппинг) использует supplier_mappings из модуля номенклатуры:
  raw_text (строка прайса поставщика) → position_id (наша)

Цепочка трёх уровней (exact → pattern → fuzzy) которую проектировали —
именно здесь работает! Снабженец сопоставляет, система учится.
```

---

## Параллельность подзадач

```
Одна задача снабжения → несколько подзадач параллельно:
  Подзадача 1: Воздуховоды → Поставщик А (статус: оплачена)
  Подзадача 2: Клапаны → Поставщик Б (статус: на согласовании)
  Подзадача 3: Крепёж → Поставщик В (статус: доставлена)

Каждая идёт по своей цепочке статусов независимо.
```

---

## Связь со статусом позиции в проекте

```
Когда подзадача снабжения доставлена/закрыта →
позиции в ней получают статус "закуплено/доставлено" →
в ПРОЕКТЕ видно что эти позиции готовы.

project_positions.purchase_status обновляется
из статуса подзадачи снабжения.

Позиция одна → статус виден и в проекте, и в снабжении.
```

## Вопросы для уточнения

1. Запрос ОДНОМУ поставщику или можно нескольким сразу
   (сравнить цены)?

2. Согласование менеджером — всей подзадачи или построчно?
   (может согласовать часть, часть отклонить?)

3. Если менеджер НЕ согласовал — назад к выбору поставщика?

4. Оплата — полная сразу или частичная (аванс)?

5. Доставка — вся подзадача разом или частями?
   (раньше говорил про частичную приёмку — она тут?)

6. Сроки — на подзадачу одни или на каждую позицию свои?

7. Маппинг цен — цена попадает в позицию? (для план/факт)

8. "Отправить поставщику по почте" — система генерит письмо/файл
   или снабженец сам шлёт, а в системе только отмечает?
```

## УТОЧНЕНИЯ (раунд 1) — зафиксировано

### ① Запрос одному поставщику (можно повторить другому)

```
Основной поток: подзадача → запрос ОДНОМУ поставщику.
Но тот же список можно отправить другому (сравнить/если не устроило).

supplier_request — может быть НЕСКОЛЬКО на подзадачу:
  request 1 → Поставщик А (цены не устроили)
  request 2 → Поставщик Б (выбран, идём дальше)

→ выбирается один, по нему продолжается процесс
```

### ② Частичная доставка — НЕТ (пока)

```
Доставка = заказанное у поставщика целиком, разом.
Частичную откладываем (под вопросом на будущее).

delivery — одна на подзадачу, всё доставляется сразу.
```

### ③ Отправка поставщику — шаблон + email из системы

```
Система генерирует ФАЙЛ по шаблону (список позиций для запроса).
Снабженец выбирает почту поставщика → отправляет прямо из системы.

supplier_request:
  id, subtask_id, supplier_id
  document        — сгенерированный файл (по шаблону)
  email_to        — куда отправлено
  sent_at
  status          — отправлен / получен ответ / отклонён

→ нужен модуль ШАБЛОНОВ ДОКУМЕНТОВ + отправка email
→ шаблон запроса: список позиций + реквизиты
```

### Обновление: supplier_request (несколько на подзадачу)

```
supplier_request
  id
  subtask_id      FK→subtasks
  supplier_id     FK→contractors
  document        сгенерированный файл запроса
  email_to        почта поставщика
  sent_at
  status          sent | answered | rejected | selected
  is_selected     выбран ли этот поставщик (по нему идём дальше)

supplier_quote (ответ на конкретный request)
  id, request_id, received_at
  document        файл ответа поставщика (если есть)
  quote_lines:    строки с ценами
```
