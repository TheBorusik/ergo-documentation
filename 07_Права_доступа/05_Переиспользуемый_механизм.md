# Переиспользуемый механизм прав (платформа: 1 бек, N фронтов)

Цель: сделать механизм прав ОДИН РАЗ → применять в любом
новом приложении на общем беке. Не изобретать заново.

═══════════════════════════════════════════════════

═══════════════════════════════════════════════════
```
Одно ЯДРО-БЕК (постоянное) + много ФРОНТОВ (меняются):
  ERP, CRM, ещё приложения... — все на одном беке

Механизм прав нужен ПЕРЕИСПОЛЬЗУЕМЫЙ:
  сделал раз → подключил к любому фронту → работает

→ здесь реестр в БД ОПРАВДАН (стоимость амортизируется

```

═══════════════════════════════════════════════════
АРХИТЕКТУРА — две части
═══════════════════════════════════════════════════
```
БЭК (ядро, делается ОДИН РАЗ):
  таблицы: permissions, ui_components, привязка к app

ФРОНТ (библиотека, делается ОДИН РАЗ → в любой проект):
  <Can> компонент, Provider, хук
  → npm-пакет / общий модуль, подключается в новый фронт
```

═══════════════════════════════════════════════════
БЭК — модель (мультиприложение)
═══════════════════════════════════════════════════
```
permissions (справочник прав — уже есть)
  code: "ERGO.Supply.Approve"
  description

roles (роли — уже есть)
role_permissions (роль → права — уже есть)

users (роль пользователя — уже есть)

── НОВОЕ (реестр UI) ──

apps (приложения на этом беке)
  id: "erp" | "crm" | ...
  name

ui_components (реестр компонентов, по приложениям)
  id
  app_id: FK→apps
  component_key: "button.approve"
  description: "Кнопка согласования"  (для админки)
  category: menu|button|column|section

ui_component_permissions (компонент → нужные права)
  component_id: FK
  permission_code: FK→permissions

ui_component_config
  component_id: FK
  require: any|all
  mode: hide|disable
```

═══════════════════════════════════════════════════
БЭК — API (универсальный)
═══════════════════════════════════════════════════
```
GET /api/me
  → { user, role, permissions: ["ERGO.Supply.Request", ...] }
  права текущего пользователя (из роли)

GET /api/ui-permissions?app={appId}
  → карта компонентов приложения:
    {
      "button.approve": {
        permissions: ["ERGO.Supply.Approve"],
        require: "any", mode: "disable"
      },
      "menu.users": { permissions: ["ERGO.Admin.Users"],
                      require: "any", mode: "hide" },
      ...
    }
  только компоненты запрошенного приложения (appId)

→ один бек обслуживает права ВСЕХ фронтов
```

═══════════════════════════════════════════════════
ФРОНТ — переиспользуемая библиотека
═══════════════════════════════════════════════════
```jsx
// === Один раз написать, класть в любой проект ===

// 1. Provider — загружает права + карту при старте
<PermissionsProvider app="erp">
  <App />
</PermissionsProvider>

// внутри: GET /api/me + GET /api/ui-permissions?app=erp
// хранит: userPermissions[] + componentMap{}

// 2. Хук
const { can, componentMap } = usePermissions();

// 3. Компонент <Can> — два режима:

// режим A — явные права (простой)
<Can permissions={["ERGO.Supply.Approve"]}>
  <button>Согласовать</button>
</Can>

// режим B — по ключу из реестра (управляется с бэка)
<Can componentKey="button.approve">
  <button>Согласовать</button>
</Can>
// берёт права из componentMap["button.approve"] → сверяет

// Реализация поддерживает ОБА:
function Can({ permissions, componentKey, require, mode,
              children }) {
  const { userPermissions, componentMap } = usePermissions();

  // если componentKey — берём конфиг из карты (с бэка)
  const config = componentKey
    ? componentMap[componentKey]
    : { permissions, require, mode };

  const perms = config?.permissions || [];
  const req = config?.require || "any";
  const md = config?.mode || "hide";

  const has = req === "all"
    ? perms.every(p => userPermissions.includes(p))
    : perms.some(p => userPermissions.includes(p));

  if (has) return children;
  if (md === "disable")
    return React.cloneElement(children, { disabled: true });
  return null;
}
```

## ① Именование ключей — app.screen.element (полный путь)
```
Формат: app.screen.element — ТОЧНОЕ расположение элемента.

Почему полный путь:
  "button.approve" НЕ уникально (кнопок согласования много)
  → нужно описать ГДЕ именно элемент

Примеры:
  erp.supply_task.approve_button    — кнопка согл. в задаче снабжения
  erp.project_card.finance_section  — блок финансов в карточке
  erp.sidebar.users_menu            — пункт меню пользователей
  erp.positions_table.price_column  — колонка цены в таблице
  crm.lead_card.convert_button      — кнопка конвертации (CRM)

Правило: две одинаковые по смыслу кнопки на разных экранах
  = РАЗНЫЕ ключи (разное расположение).

Структура: {app}.{screen}.{element}
  app    — приложение (erp, crm)
  screen — экран/раздел (supply_task, project_card, sidebar)
  element — конкретный элемент (approve_button, price_column)
```

## ② Админка — редактор UI-прав (доработать существующую)
```
У ПОЛЬЗОВАТЕЛЯ УЖЕ ЕСТЬ админка → ДОБАВИТЬ редактор UI-прав:

Функционал редактора:
  выбор приложения (app) →
  список его компонентов (ui_components этого app) →
  для каждого: назначить требуемые права + require + mode

→ не строить с нуля, расширить существующую админку
→ управление правами UI всех приложений в одном месте
```

## ③ Изоляция — никакого шаринга между приложениями
```
app_id ЖЁСТКО разделяет компоненты:
  erp.* — только ERP
  crm.* — только CRM
  компоненты НЕ пересекаются

GET /api/ui-permissions?app=erp → только erp.* компоненты

→ каждое приложение изолировано, свои права
→ чисто, нет путаницы общих компонентов
```

## Обновлённая модель (с учётом решений)
```
ui_components
  id
  app_id: FK→apps           (изоляция по приложению)
  component_key: string     ("erp.supply_task.approve_button")
                            UNIQUE в рамках app_id
  description: string       ("Кнопка согласования в задаче снабжения")
  screen: string?           (для группировки в админке: "supply_task")
  category: menu|button|column|section

ui_component_permissions
  component_id, permission_code

ui_component_config
  component_id, require (any|all), mode (hide|disable)

Админка редактора:
  GROUP BY screen → удобная навигация по компонентам экрана
  фильтр по app_id → права конкретного приложения
```

## Итоговый поток (платформа)
```
РАЗРАБОТКА нового фронта:
  1. верстаешь экраны, оборачиваешь элементы:
     <Can componentKey="erp.supply_task.approve_button">
  2. в админке заводишь эти компоненты (app=erp) + права
  3. фронт при старте: GET /api/ui-permissions?app=erp → карта
  4. <Can> применяет права из карты

ИЗМЕНЕНИЕ прав:
  админка → выбрал app → компонент → сменил права/mode
  → фронт получил новую карту → подстроился (без деплоя)

НОВОЕ приложение:
  подключил библиотеку → app={новый} → завёл компоненты в админке
```
