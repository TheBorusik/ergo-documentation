# UI управляемый с бэка — чтобы не менять код фронта

Проблема: даже с permissions на фронте хардкод (кнопка→право).
Цель: изменил роль/право/структуру на бэке → фронт подстроился без правок.

═══════════════════════════════════════════════════
ТРИ УРОВНЯ (от простого к server-driven)
═══════════════════════════════════════════════════
```
Уровень 1 — permissions + хардкод связки (обычный)
Уровень 2 — бэк отдаёт ЧТО показывать (действия с enabled)
Уровень 3 — server-driven UI (вся структура экрана с бэка)
```

═══════════════════════════════════════════════════
УРОВЕНЬ 1 — permissions + хардкод (базовый)
═══════════════════════════════════════════════════
```
Фронт получает права, но связка "кнопка→право" в коде фронта:
  { can('approval.approve') && <button>Согласовать</button> }

✓ просто
✗ добавил действие → надо править код фронта
✗ хардкод какая кнопка какое право
```

═══════════════════════════════════════════════════
УРОВЕНЬ 2 — бэк отдаёт ДЕЙСТВИЯ (рекомендую для старта) ★
═══════════════════════════════════════════════════
```
Бэк для экрана возвращает СПИСОК действий с их доступностью.
Бэк УЖЕ решил что доступно (по правам пользователя).

GET /api/subtasks/{id}  →
{
  ...данные задачи...,
  actions: [
    { id: "request",  label: "Отправить запрос", enabled: true },
    { id: "approve",  label: "Согласовать",       enabled: false },
    { id: "pay",      label: "Оплатить",          enabled: false }
  ]
}

Фронт рисует что пришло (не знает про права):
  actions.map(a =>
    <button disabled={!a.enabled} onClick={() => doAction(a.id)}>
      {a.label}
    </button>)

✓ добавил действие/право на бэке → фронт нарисовал сам
✓ фронт не знает связок кнопка→право
✓ бэк — единственный источник правды
✗ фронт всё же знает КАК рисовать кнопки (но не КАКИЕ)

→ золотая середина: структура экрана на фронте,
  доступность действий — с бэка
```

═══════════════════════════════════════════════════
УРОВЕНЬ 3 — SERVER-DRIVEN UI (вся структура с бэка)
═══════════════════════════════════════════════════
```
Бэк описывает ВЕСЬ экран как данные (JSON).
Фронт — движок: читает JSON → рендерит из компонентов.

GET /api/screens/supply_task/{id}  →
{
  screen: "supply_task",
  title: "Воздуховоды П1",
  sections: [
    { type: "stepper",
      steps: [{label:"Запрос",done:true}, {label:"Цены",active:true}] },
    { type: "positions",
      items: [{name:"Воздуховод ×14", price:1200}] },
    { type: "actions",
      buttons: [
        { id:"approve", label:"Согласовать", visible:false },
        { id:"pay", label:"Оплатить", visible:false }
      ]}
  ]
}

Фронт — реестр компонентов (stepper, positions, actions...):
  sections.map(s => renderComponent(s.type, s))

✓ меняешь структуру/права/порядок на бэке → фронт перестроился
✓ фронт вообще не знает конкретных экранов
✓ можно A/B, разные UI для ролей — всё с бэка
✗ СЛОЖНО: нужен движок рендера, реестр компонентов
✗ фронт менее гибкий в сложной интерактивности
✗ больше данных по сети

→ мощно, но overkill для старта. Применяют крупные (Airbnb, и т.п.)
```

═══════════════════════════════════════════════════
РЕКОМЕНДАЦИЯ ДЛЯ НАШЕГО ПРОЕКТА
═══════════════════════════════════════════════════
```
УРОВЕНЬ 2 — оптимально:
  структура экранов на фронте (React-компоненты, как обычно)
  ДЕЙСТВИЯ (кнопки, доступность) — с бэка

Почему:
  ✓ добавил роль/право → бэк вернёт другой набор actions →
    фронт нарисует без правок
  ✓ вся логика прав на бэке (у тебя уже permissions есть)
  ✓ фронт не хардкодит связки
  ✓ разумная сложность (не движок рендера)

Как:
  каждый GET деталей (задача, позиция) возвращает + actions[]
  бэк вычисляет actions по правам текущего пользователя
  фронт рисует actions единообразно (компонент ActionBar)

Уровень 3 — если понадобится радикальная гибкость (вряд ли).
Уровень 1 — если где-то простые статичные экраны.
```

═══════════════════════════════════════════════════
КАК БЭК ВЫЧИСЛЯЕТ actions (Уровень 2)
═══════════════════════════════════════════════════
```
GET /api/subtasks/{id}:

1. subtasks → данные задачи + stage_status
2. (текущий пользователь + его permissions — из токена)
3. Вычислить доступные действия:
   для каждого возможного действия этапа:
     enabled = user.hasPermission(action.permission)
               AND action.validForStatus(stage_status)
   → комбинация: право + уместность по статусу

Пример (задача в статусе "цены получены"):
  request:  enabled=false (уже отправлен)
  approve:  enabled = user has approval.approve (менеджер)
  pay:      enabled=false (ещё не согласовано)

→ actions учитывают И право, И текущий статус задачи
```

═══════════════════════════════════════════════════
ГИБРИД (практичный)
═══════════════════════════════════════════════════
```
Разные части — разный уровень:

Простые справочники, статичные формы → Уровень 1
  (обычные компоненты + can())

Экраны задач (действия зависят от роли+статуса) → Уровень 2
  (бэк отдаёт actions[])

Меню/навигация → permissions с бэка (скрытие разделов)

→ не обязательно один уровень везде.
  Действия этапов — Уровень 2 (там польза максимальна).
```

═══════════════════════════════════════════════════
МОДЕЛЬ ДЕЙСТВИЯ (Action)
═══════════════════════════════════════════════════
```
Action (в ответе API) {
  id: string          — идентификатор ("approve")
  label: string       — подпись кнопки ("Согласовать")
  enabled: bool       — доступно (право + статус)
  reason: string?     — почему недоступно (подсказка)
  variant: string?    — стиль (primary/danger)
  confirm: bool?      — требует подтверждения
}

Фронт рисует ActionBar из Action[]:
  <button disabled={!a.enabled} title={a.reason}>
    {a.label}
  </button>
  onClick → POST /api/subtasks/{id}/actions/{a.id}
    (единый эндпоинт действий ИЛИ конкретные)
```
