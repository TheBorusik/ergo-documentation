# Права доступа — механизм (роли + разрешения)

Как реализовать права: бэк проверяет, фронт скрывает.

═══════════════════════════════════════════════════
ГЛАВНЫЙ ПРИНЦИП
═══════════════════════════════════════════════════
```
БЭК (C#) — РЕАЛЬНАЯ защита. Проверяет КАЖДЫЙ запрос.
ФРОНТ — УДОБСТВО. Скрывает кнопки (которые всё равно не сработают).

Фронт НЕ защита: DevTools → вызов API напрямую.
Поэтому бэк проверяет всегда, независимо от фронта.
```

═══════════════════════════════════════════════════
МОДЕЛЬ — роли и разрешения (RBAC)
═══════════════════════════════════════════════════
```
RBAC = Role-Based Access Control (роль → набор прав)

РОЛЬ = набор ПРАВ (permissions).

Роли (уже есть при заведении пользователя):
  manager    — менеджер
  engineer   — инженер/проектировщик
  supply     — снабженец
  accountant — бухгалтер
  foreman    — прораб
  employee   — сотрудник (документация)
  admin      — администратор

Права (что можно делать) — гранулярные:
  project.create, project.win, project.complete
  kp.edit
  spec.create, position.create, position.ready, position.delete
  supply.create_task, supply.request, supply.map_prices, supply.deliver
  approval.approve       ← менеджер
  payment.pay            ← бухгалтер
  works.create, works.complete, act.create
  docs.attach, docs.complete
  handover.send, handover.accept
  admin.users, admin.dictionaries

Связь роль → права:
  engineer:   [spec.create, position.create, position.ready,
               position.delete, kp.edit]
  supply:     [supply.create_task, supply.request,
               supply.map_prices, supply.deliver]
  manager:    [project.create, project.win, kp.edit,
               approval.approve, handover.send]
  accountant: [payment.pay]
  foreman:    [works.create, works.complete, act.create]
  employee:   [docs.attach, docs.complete]
  admin:      [все + admin.*]
```

═══════════════════════════════════════════════════
ВАРИАНТЫ ДЛЯ ФРОНТА (как узнаёт что показывать)
═══════════════════════════════════════════════════
```
ВАРИАНТ 1 — правила в коде фронта (по роли)
  фронт: if (role === 'supply') показать
  ✗ дублирование правил (фронт+бэк), негибко

ВАРИАНТ 2 — фронт запрашивает ПРАВА (РЕКОМЕНДУЕТСЯ) ★
  при логине бэк отдаёт список прав:
    permissions: ["supply.request", "supply.map_prices"...]
  фронт: if (has('supply.request')) показать кнопку
  ✓ права централизованы на бэке
  ✓ фронт не знает ролей, только права
  ✓ изменил роль → фронт автоподстроился

ВАРИАНТ 3 — спрашивать по каждому действию
  GET /api/can?action=... перед каждой кнопкой
  ✗ много запросов, медленно

→ ВАРИАНТ 2 (роли на бэке → права фронту)
```

═══════════════════════════════════════════════════
ПОЛНЫЙ ПОТОК (Вариант 2)
═══════════════════════════════════════════════════
```
1. Логин → бэк проверяет учётку

2. Бэк собирает права:
   роль пользователя → её permissions

3. Отдаёт фронту (в JWT-токене или /api/me):
   {
     user: { id, name },
     role: "supply",
     permissions: ["supply.request", "supply.map_prices"...]
   }

4. Фронт хранит permissions (в памяти/сторе)

5. Показ UI по правам:
   hasPermission('approval.approve') → [Согласовать] видна
   иначе → кнопки нет

6. Пользователь жмёт действие → запрос на бэк

7. Бэк ПРОВЕРЯЕТ право снова (middleware/атрибут):
   [RequirePermission("approval.approve")]
   нет права → 403 Forbidden

→ фронт удобство, бэк защита
```

═══════════════════════════════════════════════════
РЕАЛИЗАЦИЯ НА БЭКЕ (C#)
═══════════════════════════════════════════════════
```
JWT-токен при логине содержит роль/права:
  claims: { userId, role, permissions[] }

Проверка на каждом эндпоинте — атрибут:
  [RequirePermission("approval.approve")]
  public IActionResult Approve(...) { ... }

Middleware читает токен → проверяет наличие права →
  есть → выполняет
  нет → 403

ИЛИ проверка в коде:
  if (!user.HasPermission("payment.pay")) return Forbid();
```

═══════════════════════════════════════════════════
РЕАЛИЗАЦИЯ НА ФРОНТЕ (React)
═══════════════════════════════════════════════════
```
При логине получили permissions → в стор (Context/Redux).

Хелпер:
  const can = (perm) => permissions.includes(perm);

Показ по праву:
  { can('approval.approve') && <button>Согласовать</button> }
  { can('payment.pay') && <button>Оплатить</button> }

Скрытие разделов:
  { can('admin.users') && <MenuItem>Пользователи</MenuItem> }

→ фронт прячет недоступное, но это лишь UX.
```

═══════════════════════════════════════════════════
МОДЕЛЬ ДАННЫХ
═══════════════════════════════════════════════════
```
users (уже есть)
  id, name, email, role (enum), is_active

ВАРИАНТ A — права зашиты в код (роль→права в конфиге)
  роли и их права — константы в коде бэка
  не меняются без деплоя
  ✓ просто, для фиксированного набора ролей

ВАРИАНТ B — права в БД (гибко, редактируемо)
  roles (id, name)
  permissions (id, code)  — справочник прав
  role_permissions (role_id, permission_id)  — связь
  ✓ можно менять права ролей без деплоя (админка)
  ✗ сложнее

Для начала — ВАРИАНТ A (роль→права в коде).
Позже B если нужна гибкая настройка.
```

═══════════════════════════════════════════════════
УРОВНИ ПРОВЕРКИ (что учитывать)
═══════════════════════════════════════════════════
```
1. ДЕЙСТВИЕ (может ли роль делать это вообще)
   снабженец не может approval.approve

2. КОНТЕКСТ (может ли с ЭТИМ объектом) — сложнее
   - только свои задачи? (снабженец видит все или свои?)
   - только свой проект?
   - этап проекта (нельзя закупку если проектирование не начато?)

Для начала — уровень 1 (действие по роли).
Уровень 2 (контекст) — если нужен (свои/чужие задачи).
```

═══════════════════════════════════════════════════
ВИДИМОСТЬ vs ДЕЙСТВИЯ
═══════════════════════════════════════════════════
```
Два типа прав:
  ВИДЕТЬ (read) — какие разделы/данные показывать
    бухгалтер видит снабжение (для оплаты)?
    прораб видит цены закупки?

  ДЕЙСТВОВАТЬ (write) — что может менять
    только менеджер согласует
    только бухгалтер оплачивает

Можно раздельно: project.view, approval.approve
Часто: видят многие, действуют избранные.
```

═══════════════════════════════════════════════════
ВОПРОСЫ
═══════════════════════════════════════════════════
```
1. Права в коде (A) или БД с админкой (B)?
2. Контекст нужен (свои/чужие задачи) или хватит роли?
3. Видимость: кто что ВИДИТ (не только действует)?
   - бухгалтер видит только оплаты или весь проект?
   - прораб видит цены?
4. Один пользователь — одна роль или несколько?
5. Админ — полный доступ ко всему?
6. Есть ли "супер-действия" (отмена, откат) только для админа?
```
