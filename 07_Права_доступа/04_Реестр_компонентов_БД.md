# Реестр компонентов в БД — управление правами UI с бэка

Идея: таблица "компонент → нужные права" на бэке.
Убирает хардкод связок с фронта. Взвешенный разбор.

═══════════════════════════════════════════════════
ИДЕЯ
═══════════════════════════════════════════════════
```
Таблица на бэке: какой компонент какие права требует.
Фронт запрашивает карту → применяет.

component_permissions:
  component_key       | required_permissions       | mode
  ────────────────────────────────────────────────────────
  "menu.users"        | ["ERGO.Admin.Users"]       | hide
  "button.approve"    | ["ERGO.Supply.Approve"]    | disable
  "column.price"      | ["ERGO.Prices.View"]       | hide
  "section.finance"   | ["ERGO.Finance.View"]      | hide

→ меняешь права компонента в БД → фронт подстроился
→ даже связку компонент→право не хардкодишь
```

═══════════════════════════════════════════════════
КАК РАБОТАЕТ
═══════════════════════════════════════════════════
```
1. Фронт при старте:
   GET /api/ui-permissions
   → карта: { "button.approve": {perms:["ERGO.Supply.Approve"],
                                  mode:"disable"}, ... }
   GET /api/me
   → права пользователя: ["ERGO.Supply.Request", ...]

2. Компонент по ключу:
   <Can componentKey="button.approve">
     <button>Согласовать</button>
   </Can>
   → ищет в карте права для ключа → сверяет → показ/скрытие/блок

3. Изменил в таблице (права/mode компонента):
   → фронт получил новую карту → применил
   → код фронта НЕ трогали
```

═══════════════════════════════════════════════════
ЧЕСТНО — ПЛЮСЫ И ПОДВОДНЫЕ КАМНИ
═══════════════════════════════════════════════════
```
ПЛЮСЫ:
  ✓ Ноль хардкода "компонент→право" на фронте
  ✓ Меняешь права компонентов с бэка (без деплоя)
  ✓ Переименовал право → поменял в таблице, фронт цел
  ✓ Централизованное управление (можно админку)
  ✓ mode (hide/disable) тоже из таблицы

ПОДВОДНЫЕ КАМНИ:
  ✗ Фронт ВСЁ РАВНО знает КЛЮЧИ компонентов
    (<Can componentKey="button.approve"> — ключ в коде)
    хардкод сдвинулся с "право" на "ключ"
  ✗ Кто-то ВЕДЁТ таблицу (новый компонент → завести запись)
  ✗ Рассинхрон: ключ есть на фронте, нет в таблице →
    поведение по умолчанию? (показать/скрыть?)
  ✗ Лишний запрос + зависимость от карты при старте
```

═══════════════════════════════════════════════════
КЛЮЧЕВОЙ АНАЛИЗ — что реально выигрываем
═══════════════════════════════════════════════════
```
БЕЗ таблицы (враппер с явными правами):
  <Can permissions={["ERGO.Supply.Approve"]}>
  → право В КОДЕ фронта

С таблицей (реестр):
  <Can componentKey="button.approve">
  → ключ в коде фронта, право в БД

В ОБОИХ что-то в коде фронта (право ИЛИ ключ).
Разница — ЧТО меняется без деплоя фронта:

Выигрыш таблицы:
  ✓ переименовал право → в таблице, фронт цел
  ✓ изменил какие права нужны компоненту → в таблице
  ✓ изменил mode компонента → в таблице

НЕ выигрыш:
  ✗ новый компонент → всё равно код фронта (ключ + запись в БД)
  ✗ ключи компонентов — те же "магические строки"
```

═══════════════════════════════════════════════════
КОГДА ЧТО ВЫБРАТЬ
═══════════════════════════════════════════════════
```
ТАБЛИЦА (реестр в БД) — если:
  права/связки меняются ЧАСТО
  переименования прав, реструктуризации ролей
  нужна тонкая настройка "кто что видит" без деплоя
  планируется админка управления доступом UI

ВРАППЕР С ЯВНЫМИ ПРАВАМИ — если:
  права относительно СТАБИЛЬНЫ (роли устоялись)
  проще: право в коде — меняется редко, не беда
  меньше движущихся частей (нет карты, запроса, ведения таблицы)

→ таблица = гибкость ценой сложности
  враппер = простота ценой правок при смене прав
```

═══════════════════════════════════════════════════
ГИБРИДНЫЙ ПОДХОД (практичный компромисс)
═══════════════════════════════════════════════════
```
Можно начать с ВРАППЕРА (явные права в коде),
структуру заложить под РЕЕСТР на будущее:

  <Can permissions={["ERGO.Supply.Approve"]}>  ← сейчас

Если понадобится гибкость → добавить карту:
  <Can componentKey="button.approve">  ← потом
  (компонент <Can> поддержит оба режима)

→ не переусложнять сразу, но оставить путь к реестру.
```

═══════════════════════════════════════════════════
МОДЕЛЬ ТАБЛИЦЫ (если реестр)
═══════════════════════════════════════════════════
```
ui_components (реестр компонентов)
  id
  component_key: string    (уникальный: "button.approve")
  description: string      (для админки: "Кнопка согласования")
  category: string?        (menu | button | column | section)

ui_component_permissions (связь)
  id
  component_id: FK
  permission_code: string  ("ERGO.Supply.Approve")

ui_component_config
  component_id: FK
  require: enum (any|all)
  mode: enum (hide|disable)

API:
  GET /api/ui-permissions
    → { "button.approve": { perms:[...], require, mode }, ... }
    (вся карта для фронта, кэшируется)

Наполнение: собрать все возможные component_key из системы
  (можно скриптом просканировать фронт, или вести вручную).
```

═══════════════════════════════════════════════════
РЕКОМЕНДАЦИЯ
═══════════════════════════════════════════════════
```
СТАРТ: враппер <Can> с явными правами (файл 03)
  просто, работает, права строками с бэка

ЕСЛИ появится потребность в частой смене прав/связок:
  → мигрировать на реестр (эта схема)
  <Can> поддержит оба режима (permissions ИЛИ componentKey)

Не строить реестр СРАЗУ (преждевременное усложнение),
но заложить <Can> так, чтобы реестр добавился легко.

→ прагматично: простое сейчас, путь к гибкому оставлен.
```
