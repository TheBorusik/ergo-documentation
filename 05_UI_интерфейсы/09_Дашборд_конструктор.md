# ДАШБОРД-КОНСТРУКТОР — полная документация

> ⚠️ ЖИВОЙ ДОКУМЕНТ — каталог виджетов (раздел 8) расширяется.
> Новый виджет → добавить в каталог (ключ, название, размер, право).
> Появится новая метрика/список/диаграмма для показа → сюда.

Главный экран как настраиваемый дашборд из виджетов.
Каждый сотрудник собирает свой (строки + виджеты), сохраняется на сервере.

Единый формат (как этапы): назначение → интерфейс → действия →
правила → API → серверная логика → таблицы.

Всё в одном файле: назначение, интерфейс, каталог виджетов,
API, серверная логика, модели. Не разбито по файлам.

1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Главный экран = настраиваемый дашборд.
Вместо фиксированного (мы решаем что кому) — КОНСТРУКТОР:
  сотрудник сам собирает из виджетов, раскладка на сервере.

Зачем (особенно первое приложение):
  ✓ не гадаем что нужно роли — человек соберёт сам
  ✓ работа у всех разная → дашборд под себя
  ✓ шаблоны дают быстрый старт
  ✓ растёт с пониманием пользователей

Компоненты:
  ПАЛИТРА виджетов (каталог ~49, файл 11)
  ДАШБОРД пользователя (его раскладка)
  ШАБЛОНЫ (стартовые наборы)
```

## 2.2 Режим настройки (строки-контейнеры)

┌──────────────────────────────────────────────────────┐

├──────────────────────────────────────────────────────┤
│ ┌─ СТРОКА 1 · высота авто ──────────────────── [✕] ┐ │
│ │ [W1]⇄ [W2]⇄ [W3]⇄ · · · · свободно            │ │
│ └────────────────────────────────────────────────┘ │
│ ┌─ СТРОКА 2 ─────────────────────────────────── [✕] ┐ │

3. ДЕЙСТВИЯ ПОЛЬЗОВАТЕЛЯ
═══════════════════════════════════════════════════
```
ПРОСМОТР:
  ① видит свой дашборд (или дефолтный шаблон)
  ② взаимодействует с виджетами (фильтры, выбор проекта в config)

НАСТРОЙКА:
  ③ [Настроить] → режим редактирования
  ④ [+ Добавить строку] → пустая строка
  ⑤ [+ Виджет] в строку → модалка (превью) → выбрал → встал
  ⑥ двигает виджет по строке (шаг колонок)
  ⑦ расширяет/сужает виджет (соседи сдвигаются)
  ⑧ удаляет виджет/строку
  ⑨ [Сохранить] → раскладка на сервер

ШАБЛОНЫ:
  ⑩ [Шаблон] → выбрать готовый → применить
  ⑪ сохранить свою раскладку как шаблон (публичный)
```

═══════════════════════════════════════════════════
4. ПРАВИЛА
═══════════════════════════════════════════════════
```
СЕТКА:
  - строки сверху вниз (контейнеры)
  - внутри строки — сетка 24 колонки (дефолт, настраиваемо 12|24|48)
  - высота строки авто (по самому высокому виджету)

ВИДЖЕТЫ:
  - каждый с мин/макс размером (админ задал)
  - в строке стоят слева, справа может быть свободно
  - расширение → соседи сдвигаются вправо (заполняя место)
  - предел: свободные колонки строки + макс размер виджета
  - виджет под правом (нет права → не в палитре)

РАЗМЕЩЕНИЕ:
  - добавляешь виджет (не выделяешь область)
  - виджет встаёт с дефолт размером
  - дальше двигаешь/расширяешь

ШАБЛОНЫ:
  - админ создаёт базовые (под роли)
  - потом любой создаёт свой публичный

СОХРАНЕНИЕ:
  - раскладка на сервере (user_dashboards)
  - при входе грузится своя (или дефолт по роли)
```

5. API (детально)
═══════════════════════════════════════════════════
```
GET /api/dashboard/widgets
  → каталог виджетов (с учётом прав пользователя)
  Ответ: { items: DashboardWidget[] }
    (виджеты под правом фильтруются)

GET /api/dashboard/my
  → раскладка текущего пользователя
  Ответ: { layout: {...} } (или дефолтный шаблон если нет)

PUT /api/dashboard/my
  Тело: { layout: {...} }
  → сохранить раскладку
  Ответ: 200

GET /api/dashboard/templates
  → список шаблонов (публичные + под роль)
  Ответ: { items: DashboardTemplate[] }

POST /api/dashboard/apply-template
  Тело: { template_id }
  → применить шаблон (заменить раскладку)
  Ответ: { layout: {...} }

POST /api/dashboard/templates
  Тело: { name, layout, is_public }
  → сохранить свою раскладку как шаблон
  Ответ: DashboardTemplate

GET /api/dashboard/widget-data/{widget_key}?params
  → данные конкретного виджета (лениво)
  Ответ: зависит от виджета (число / список / данные диаграммы)
  params: config виджета (период, project_id, threshold_days)
```

5Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════

───────────────────────────────────────────────────
GET /api/dashboard/widgets — каталог (по правам)
───────────────────────────────────────────────────
```
1. dashboard_widgets → SELECT где is_enabled=true
                     → все активные виджеты

2. (права пользователя из токена)
   фильтр: оставить виджеты где required_permission
     пусто ИЛИ ∈ права пользователя
   → только доступные виджеты

Возврат: DashboardWidget[] (с мин/макс/превью/дефолт)

[C#] фильтрация по правам в коде (не SQL — права в токене).
```

───────────────────────────────────────────────────
GET /api/dashboard/my — раскладка пользователя
───────────────────────────────────────────────────
```
1. user_dashboards → SELECT где user_id={me}
                   → его раскладка (layout JSONB)

2. ЕСЛИ нет:
   dashboard_templates → SELECT где role_hint={роль юзера}
                       LIMIT 1
                       → дефолтный шаблон по роли

Возврат: { layout }
```

───────────────────────────────────────────────────
PUT /api/dashboard/my — сохранить раскладку
───────────────────────────────────────────────────
```
1. user_dashboards → SELECT где user_id={me}
                   → есть ли запись

2. user_dashboards → UPDATE layout={...} (если есть)
                   ИЛИ INSERT (user_id, layout) (если нет)

Возврат: 200
[Валидация layout: виджеты существуют, размеры в пределах мин/макс]
```

───────────────────────────────────────────────────
GET /api/dashboard/widget-data/{key} — данные виджета
───────────────────────────────────────────────────
```
Роутинг по widget_key (каждый виджет — свой запрос):

metric.projects_active:
  projects → SELECT COUNT где status='active'

metric.tasks_my:
  subtasks → SELECT COUNT где responsible_id={me}
    AND status IN (new, in_progress)

list.my_tasks:
  subtasks → SELECT где responsible_id={me} AND активные
  projects → SELECT названия (группировка по проекту)
  → сгруппированный список

feed.all:
  event_log → SELECT где ... ORDER BY changed_at DESC LIMIT N
  projects → названия (для группировки)
  → лента, группировка по проекту

chart.projects_by_stage:
  project_stages → SELECT COUNT GROUP BY stage_type
  → данные для диаграммы

attention.summary:
  event_log → последнее событие каждой сущности (давность)
  projects → сроки (deadline)
  → вычислить триггеры (просрочка, без движения, застряло)
    по порогам (config.threshold_days)

→ каждый виджет = свой лёгкий запрос, грузится лениво

[SQL] где агрегация (COUNT/GROUP BY) — одним запросом.
Каждый виджет самодостаточен (свои данные).
```

  id

  layout: JSONB
  created_by: FK→User
  is_public: bool
  role_hint: string?

Layout (JSONB структура):

    grid_columns: 24,

      { height: "auto", widgets: [

          { widget:"metric.tasks_my", col:4, w:4, h:1 }
