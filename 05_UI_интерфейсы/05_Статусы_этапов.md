# Статусы этапов (общий уровень) — наглядность на доске

Каждый ЭТАП имеет свой общий статус (не только задачи внутри).
Видно на доске и в разделах — где что.

═══════════════════════════════════════════════════
ЗАЧЕМ
═══════════════════════════════════════════════════
```
Было: статусы только у задач (подзадач)
Стало: + общий статус у ЭТАПА

→ на доске сразу видно: этап ждёт / в работе / завершён
→ наглядная картина всего проекта
```

═══════════════════════════════════════════════════
ЕДИНАЯ СХЕМА СТАТУСОВ ЭТАПА
═══════════════════════════════════════════════════
```
⚪ ОЖИДАНИЕ  — ждёт предыдущий этап
   Проектирование ждёт КП (Продажу)
   Снабжение ждёт Проектирование
   Работы ждут Снабжение
   и т.д.

🔵 В РАБОТЕ  — этап активен (идёт работа)

🟢 ЗАВЕРШЁН  — этап полностью сделан
   (ответственный нажал "Завершить этап")

Единая цветовая схема для ВСЕХ этапов.
```

═══════════════════════════════════════════════════
НАЧАЛЬНЫЕ СТАТУСЫ (цепочка ожидания)
═══════════════════════════════════════════════════
```
Продажа:        в работе (первый, сразу активен)
Проектирование: ожидание (ждёт КП / выигрыш)
Снабжение:      ожидание (ждёт Проектирование)
Работы:         ожидание (ждёт Снабжение)
Документация:   ожидание (ждёт Работы)
Сдача:          ожидание (ждёт Документацию)

Каждый этап ждёт предыдущий.
```

═══════════════════════════════════════════════════
ГДЕ ПОКАЗЫВАТЬ
═══════════════════════════════════════════════════
```
1. ДОСКА — статус над колонкой этапа:
   ┌─Проектир.──┬─Снабжение──┬─Монтаж────┬─Докумен.──┐
   │ 🟢 Завершён│ 🔵 В работе│ ⚪ Ожидание│ ⚪ Ожидание│
   │ [задачи]   │ [задачи]   │            │            │
   └────────────┴────────────┴────────────┴────────────┘

2. РАЗДЕЛЫ (список слева) — индикатор у этапа:
   ▸ Доска
   ▸ Проектирование 🟢
   ▸ Снабжение 🔵
   ▸ Работы ⚪
   ▸ Документы ⚪
```

═══════════════════════════════════════════════════
ПЕРЕХОДЫ СТАТУСА ЭТАПА
═══════════════════════════════════════════════════
```
ОЖИДАНИЕ → В РАБОТЕ:
  ВОПРОС (уточнить):
    A) авто когда предыдущий этап завершён
    B) авто когда появилась первая задача этапа
    C) вручную "Начать этап"

В РАБОТЕ → ЗАВЕРШЁН:
  вручную — ответственный жмёт "Завершить этап"
  (проектировщик закончил всё → "Завершить проектирование")
  → статус этапа 🟢, видно на доске
```

═══════════════════════════════════════════════════
ВАЖНО — статус этапа vs опережающий поток
═══════════════════════════════════════════════════
```
ТОНКОСТЬ: позиции текут независимо (опережающий поток).
  Проектирование "завершено" →
  но на объекте столб → нужны доп. позиции →
  проектировщик добавляет в "завершённый" этап

РЕШЕНИЕ (предложение — Вариант A):
  Статус этапа = ИНДИКАТОР состояния, НЕ жёсткая блокировка.

  "Завершён" = ответственный считает что закончил (наглядность).
  Если надо добавить ещё:
    - можно добавить (этап вернётся в "в работе"?)
    - ИЛИ добавить не меняя статус (мелкая доработка)

  → статус помогает видеть картину, не мешает реальности

ВОПРОС: при добавлении в завершённый этап —
  статус возвращается в "в работе" или остаётся "завершён"?
```

═══════════════════════════════════════════════════
СВЯЗЬ СО СТАТУСАМИ ЗАДАЧ (два уровня)
═══════════════════════════════════════════════════
```
ДВА УРОВНЯ статусов:
  ЭТАП (общий): ожидание / в работе / завершён
  ЗАДАЧА (внутри): свои статусы этапа
    (снабжение: 14 статусов, проектирование: new→done...)

Этап "в работе" — есть активные задачи.
Этап "завершён" — ответственный пометил (все задачи сделаны).

Статус этапа НЕ вычисляется автоматом из задач жёстко
(из-за опережающего потока), а ставится ответственным
+ помогает наглядности.
```

═══════════════════════════════════════════════════
МОДЕЛЬ
═══════════════════════════════════════════════════
```
project_stages (уже есть — 6 этапов проекта)
  + status: enum (waiting | in_progress | completed)
  + started_at, completed_at
  + completed_by (кто завершил)

Каждый проект имеет 6 записей project_stages
(по одной на этап), у каждой свой статус.

API:
  PATCH /api/projects/{id}/stages/{stageId}
    Тело: { status: "in_progress" | "completed" }
    Ответ: ProjectStage
    (завершить этап → completed, completed_by, completed_at)

  GET /api/projects/{id}/stages
    Ответ: { items: ProjectStage[] }
    (для доски и индикаторов — статусы всех этапов)
```

═══════════════════════════════════════════════════
ЦВЕТОВАЯ СХЕМА (единая)
═══════════════════════════════════════════════════
```
⚪ Ожидание  — серый (неактивен, ждёт)
🔵 В работе  — синий (активен)
🟢 Завершён  — зелёный (готов)

Та же схема везде: доска, разделы, обзор.
Консистентно для всех 6 этапов.
```

═══════════════════════════════════════════════════
РЕШЕНИЯ (зафиксировано)
═══════════════════════════════════════════════════
```
① ОЖИДАНИЕ→В РАБОТЕ — ВРУЧНУЮ ("Начать этап"), пока.
   Потом по практике посмотрим (может авто удобнее).

② СМЕНА СТАТУСА с причиной:
   Завершён → Доработка (в работе) — с ПРИЧИНОЙ текстом
   ("обход столба", "заказчик изменил ТЗ")
   → видно сколько раз дорабатывали и почему
   → всё пишется в УНИФИЦИРОВАННУЮ историю (event_log)
   см. 03_Архитектура_ядро/09_Унифицированная_история.md

③ Завершить при незавершённом предыдущем — МОЖНО.
   Статусы этапов независимы (не блокируют жёстко).
   Редкий случай, с переоткрытием не критично.

СТАТУС ЭТАПА = наглядный ИНДИКАТОР, не жёсткая блокировка.
```

═══════════════════════════════════════════════════
УНИФИЦИРОВАННАЯ ИСТОРИЯ (важно)
═══════════════════════════════════════════════════
```
Все события (позиции, этапы, задачи, проект) → ОДИН журнал event_log.
  Начали проектирование → завершили → переоткрыли (причина)
  Позиция создана → закуплена → доставлена
  → timeline проекта, история этапа, история позиции = срезы

Детали: 03_Архитектура_ядро/09_Унифицированная_история.md
position_history заменяется на event_log (все сущности).
```
