# Раздел Справочники (системные) + универсальная таблица

Решение 2026-07-22 (обновлено в тот же день: универсальная таблица).
Системные справочники — меню СИСТЕМА → Справочники; номенклатурные —
в своём разделе (05/16).

═══════════════════════════════════════════════════
★ УНИВЕРСАЛЬНАЯ ТАБЛИЦА ПРОСТЫХ СПРАВОЧНИКОВ
═══════════════════════════════════════════════════
```
Простые списки (2-10 значений «название+активность») встречаются
часто — таблица под каждый не имеет смысла. Решение: ОДНА таблица.

dictionary_values
  id          PK
  dict_type   varchar   — ключ справочника (work_types, units, …)
  value       varchar   — отображаемое значение
  code        varchar?  — короткий код (интеграции/артикулы), опционален
  sort_order  int       — порядок в списках (= порядок строк при Save)
  is_active   bool      — деактивация вместо удаления
  is_system   bool      — системное: нельзя удалить/переименовать из UI
  created_at, updated_at

Паттерн известен как OTLT (One True Lookup Table). В чистой
реляционной теории — антипаттерн (нет FK/констрейнтов на конкретный
список), но у НАС целостность и так на уровне приложения (SAL
GenericCRUD: линейные запросы, проверки в коде) — минус не работает.

ГИБРИДНОЕ ПРАВИЛО:
  простые списки (value+code+active)      → dictionary_values
  сложные справочники (свои поля/связи:   → ОТДЕЛЬНЫЕ таблицы
    склады project_id, контрагенты,          и свои методы/разделы
    номенклатурные)

Ссылки на значения: FK на dictionary_values.id (например
subtask_works.work_id); принадлежность типу проверяет код
(dict_type='work_types') — как все наши проверки.
```

═══════════════════════════════════════════════════
УСТРОЙСТВО РАЗДЕЛА
═══════════════════════════════════════════════════
```
КОНТЕЙНЕР вкладок (единый паттерн 05/11 п.11):
  [Список · pinned] + вкладки открытых справочников
  URL: /dictionaries(/{dictKey})

Вкладка справочника — значения ЧЕРЕЗ ЧЕРНОВИК (единый механизм):
  правки/добавления/удаления локально, изменённое подсвечивается
  (lib/changed), dirty ● на вкладке, SaveGuard, [Сохранить] шлёт
  ВЕСЬ черновик — diff считает СЕРВЕР (паттерн Kp.Save).

Строка: Название | Код | Активно | 🗑
  is_system → название текстом с 🔒, корзинки нет.
  Используемое значение удалить нельзя (сервер вернёт ошибку) —
  деактивировать. sort_order = порядок строк при сохранении.
```

═══════════════════════════════════════════════════
WIRE-МЕТОДЫ (универсальные, ✓ реализованы)
═══════════════════════════════════════════════════
```
ErgoArea.Dictionaries.Query { Type, All? } → { Items: DictionaryValue[] }
  All=true — включая неактивные (экран справочника);
  без All — только активные (селекты в задачах/формах).

ErgoArea.Dictionaries.Save { Type, Items } → Query(All)
  Id < 0 — новые. Diff на сервере:
  1. dictionary_values → SELECT где dict_type={Type}
  2. отсутствующие в Items:
       is_system → ошибка;
       USAGE-проверка типа (work_types → subtask_works.work_id) →
         используется → ошибка «деактивируйте»;
       DELETE
  3. каждая присланная: Id<0 → INSERT (sort_order = индекс);
       иначе UPDATE (is_system переименовать нельзя; updated_at)
  4. event_log → 'dictionary_saved'

★ Прежние ErgoArea.Works.Query/Save УПРАЗДНЕНЫ — виды работ ходят
  через Dictionaries.* с Type='work_types' (04/03 обновлён).
```
