# Механика навигации: роутинг, вкладки, загрузка данных

Предварительный свод всего что разобрали устно.
Основа для реализации фронта (React + React Router + React Query).

═══════════════════════════════════════════════════
1. ДВА РАЗНЫХ ПОНЯТИЯ — не путать
═══════════════════════════════════════════════════
```
URL-РОУТ (адресная строка) — ГДЕ пользователь:
  /project/1/supply/task/42
  весь путь; для ссылок, кнопки "назад", F5, восстановления

API-ЭНДПОИНТ (запрос данных) — ЧТО загрузить:
  GET /api/subtask/42
  по сущности; id уникален, серверу хватает его

Роутер читает URL → разбирает → каждый уровень грузит свой эндпоинт.
URL длинный (навигация), API короткие (данные).
```

═══════════════════════════════════════════════════
2. КТО СТРОИТ URL — ФРОНТ
═══════════════════════════════════════════════════
```
БЭК отдаёт ДАННЫЕ (type + id + контекст):
  { type:"subtask", id:42, project_id:1, section:"supply" }

ФРОНТ строит URL по своим шаблонам (switch по type):
  subtask     → /project/{project_id}/{section}/task/{id}
  position    → /project/{project_id}/{section}/position/{id}
  project     → /project/{id}
  contractor  → /contractors/{id}
  order       → /orders/{id}
  nomenclature→ /nomenclature/{id}

→ navigate(url) → роутер разворачивает.

URL = навигация приложения = забота фронта.
Поменяли схему URL → меняем только фронт, бэк не трогаем.
```

═══════════════════════════════════════════════════
3. РОУТЕР — готовый механизм (React Router v6/v7)
═══════════════════════════════════════════════════
```
Ручного парсинга строки НЕТ. Объявляем шаблоны с :параметрами:

<Route path="project/:projectId" element={<ProjectLayout/>}>
  <Route path=":section" element={<SectionLayout/>}>
    <Route path="task/:taskId" element={<TaskPage/>} />
  </Route>
</Route>

Роутер сам:
  достаёт значения → useParams() → { projectId, section, taskId }
  монтирует дерево компонентов (nested routes)
  вставляет вложенный уровень в <Outlet/> родителя

Готовые механизмы:
  useParams()   — параметры из URL
  <Outlet/>     — точка вставки вложенного уровня
  <Link to=>    — переход без перезагрузки
  useNavigate() — программный переход
```

═══════════════════════════════════════════════════
4. ГДЕ ЖИВУТ ОТКРЫТЫЕ ВКЛАДКИ — ★ПОДХОД B (выбран)
═══════════════════════════════════════════════════
```
ПОДХОД A — вкладки в URL:
  URL описывает одну активную ветку.
  Новый URL → старая вкладка ЗАМЕНЯЕТСЯ.
  Вкладки не накапливаются. Проще, но не наш случай.

★ПОДХОД B — список вкладок в СОСТОЯНИИ (выбран):
  openTabs = [Документы, Снабжение, ...]   ← state приложения
  URL = указатель на АКТИВНУЮ вкладку

  Переход на новую вкладку:
    1. нет в openTabs → добавить
    2. уже есть → просто активировать (не дублировать)
    3. активную записать в state + URL

  → Документы остаётся + добавляется Снабжение. ВКЛАДКИ НАКАПЛИВАЮТСЯ.
  Как в браузере: открытые табы = состояние, адрес = активный таб.

Прототип erp_prototype_hybrid.html уже так устроен
  (openGlobal/openInLevel добавляют в состояние).

Нюанс: при F5 URL восстановит активную вкладку,
  остальные открытые — только если сохранять список
  (localStorage/сессия) — решить при реализации.
```

═══════════════════════════════════════════════════
5. ЧТО ГРУЗИТСЯ — РОЛЬ КОМПОНЕНТА В ДЕРЕВЕ ★
═══════════════════════════════════════════════════
```
При открытии URL /project/1/supply/task/42 — ДВА запроса:

① РОДИТЕЛЬ (ProjectLayout) — грузит ВСЕГДА:
   GET /project/1 → { name, sections:[{key,title,count}...] }
   зачем: шапка проекта + лента вкладок (имена/бейджи разделов)
   без него нечем нарисовать вкладки-дети

② ЛИСТ = последний сегмент URL (TaskPage) — АКТИВЕН:
   GET /subtask/42 → содержимое задачи

③ СРЕДНИЕ (SectionLayout "Снабжение") — НЕ грузят:
   isActive=false → содержимое не грузим
   их имя+бейдж уже пришли в списке разделов проекта

Правило: родитель-обёртка грузит "оглавление" (список детей с мета),
  лист URL грузит детали. Средние молчат до активации.
```

═══════════════════════════════════════════════════
6. ДВА УРОВНЯ ДАННЫХ ★ (важное различие)
═══════════════════════════════════════════════════
```
МЕТА ВКЛАДКИ (имя, бейдж, счётчик, иконка, статус):
  приходит в СПИСКЕ РОДИТЕЛЯ — дёшево, уже загружено
  GET /project/1 → sections:[{ key:"supply", title:"Снабжение", count:3 }]

СОДЕРЖИМОЕ ВКЛАДКИ (полные задачи, детали, позиции):
  грузится ОТДЕЛЬНО, только когда вкладка активна
  GET /section/supply/tasks → [задачи с деталями]

→ бейдж "3" на вкладке Снабжение ЕСТЬ сразу (из списка проекта),
  а сами 3 задачи не загружены пока не откроешь.

Типовой паттерн: список даёт заголовки+счётчики для навигации,
  детали подгружаются при открытии.
```

═══════════════════════════════════════════════════
7. КАК ВКЛАДКА ПОНИМАЕТ ЧТО НЕ ГРУЗИТЬ
═══════════════════════════════════════════════════
```
Роутер монтирует ветку, НО загрузку запускает КОМПОНЕНТ по условию:

  useEffect(() => {
    if (isActive) загрузить содержимое;
  }, [isActive])

Задача активна → true → грузит.
Снабжение не активно → false → НЕ грузит.
Переключился на снабжение → isActive стал true → загрузило.

Решает не роутер, а сам компонент по своему isActive.
```

═══════════════════════════════════════════════════
8. КЭШ — убирает повторную загрузку ★
═══════════════════════════════════════════════════
```
ПРОБЛЕМА: переключаюсь между вкладками → каждый раз загрузка?
РЕШЕНИЕ: кэш (React Query / SWR — готовые библиотеки).

Первый клик на вкладку → запрос + сохранить в кэш
Вернулся на неё      → ИЗ КЭША, 0 запросов
Снова ушёл/вернулся  → из кэша, 0 запросов

Устарело (прошло время / изменил данные):
  показать старое СРАЗУ + тихо обновить фоном (без мелькания)

Из коробки: кэш по ключу, stale-while-revalidate, инвалидация.
Писать вручную не нужно.
```

═══════════════════════════════════════════════════
9. ОТКРЫТИЕ ИЗ ГЛОБАЛЬНОГО ПОИСКА
═══════════════════════════════════════════════════
```
search/open НЕ НУЖЕН. Поиск — просто источник данных.

Клик на результат:
  1. фронт по type строит URL (см. раздел 2)
  2. navigate(url)
  3. роутер разворачивает вкладки (структура сразу)
  4. компоненты грузят своё (родитель + активный лист)

Результат: открылись 3 вкладки (проект→снабжение→задача),
  активна задача, грузится только она + проект (для мета).
  Промежуточные — каркасы, наполнятся при клике, дальше кэш.
```

═══════════════════════════════════════════════════
10. МЕЛЬКАНИЕ — не свойство вложенности
═══════════════════════════════════════════════════
```
Мелькание возникает при НАИВНОЙ реализации
  (рендерить каждый промежуточный шаг по очереди).

Убирается:
  ✓ активна СРАЗУ конечная вкладка (промежуточные лениво)
  ✓ скелетоны вместо белых экранов
  ✓ кэш (повторные переходы мгновенные)

Ручная навигация (клик за кликом) — запрос на шаг НОРМАЛЬНО:
  пользователь идёт по одному, ждёт один экран.
```

11. ИТОГОВАЯ СХЕМА (одной картиной)
═══════════════════════════════════════════════════
```
БЭК: данные (type + id + контекст)
  ↓
ФРОНТ: строит URL по type → navigate()
  ↓
РОУТЕР: читает URL → useParams → монтирует дерево (Outlet)
  ↓
СОСТОЯНИЕ: openTabs накапливаются (подход B), URL = активная
  ↓
КОМПОНЕНТЫ: родитель грузит мета-список, лист грузит детали,
             средние молчат (isActive=false)
  ↓
КЭШ (React Query): повторные переходы 0 запросов
```

13. КОНТЕЙНЕРНЫЕ РАЗДЕЛЫ (решение 2026-07-22, реализовано)
═══════════════════════════════════════════════════
```
ГЛОБАЛЬНАЯ лента вкладок содержит ТОЛЬКО разделы меню.
Сущности открываются ВНУТРИ своего раздела-контейнера
(внутренний уровень вкладок, первый — закреплённый список):

  Проекты      → [Список·pinned] + вкладки проектов
                 URL /projects | /project/{id}(/board|/{section}(/task/{id}))
  Продажи      → [Сделки·pinned] + вкладки сделок
                 сделка — тоже контейнер: [Детали·pinned][КП·pinned]
                 URL /sales | /sales/sale/{id}(/kp)
  Номенклатура → [Каталог·pinned][Элементы][Справочники] (05/16)
  Справочники  → [Список·pinned] + вкладки справочников (05/17)
                 URL /dictionaries(/{dictKey})
  Раздел works → [Задачи·pinned][Акты·pinned] (04/03)

Раньше проект был ГЛОБАЛЬНОЙ вкладкой — перенесён внутрь раздела;
старые localStorage-деревья мигрируются при загрузке (root proj-N
переезжают в children menu-projects со всем поддеревом).

Правило на будущее: новая сущность с вкладками → внутрь своего
раздела, глобальную ленту не засорять.
```
