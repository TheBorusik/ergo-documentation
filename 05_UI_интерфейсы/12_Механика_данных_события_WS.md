# Механика данных: WS-протокол, события, кэш, черновики вкладок

Как фронт получает данные и реагирует на изменения с сервера.
Дополняет 05/11 (роутинг и навигация).

═══════════════════════════════════════════════════
1. ТРАНСПОРТ — свой WS-протокол
═══════════════════════════════════════════════════
```
СВОЯ БИБЛИОТЕКА (уже есть):
  request/response — отправляет запрос, маппит с ответом по requestId,
  возвращает promise → в коде обычный async/await
  + сервер шлёт СОБЫТИЯ (push) об изменениях

  api.request('subtask.get', { id: 42 })  → promise
  api.on('subtask.created', handler)      → подписка на событие
```

═══════════════════════════════════════════════════
2. КЭШ — React Query поверх WS ★
═══════════════════════════════════════════════════
```
React Query НЕ привязан к HTTP. queryFn — просто async-функция.
Наш api.request возвращает promise → подходит как есть:

  useQuery({
    queryKey: ['subtask', 42],
    queryFn: () => api.request('subtask.get', { id: 42 })
  })

→ кэш, дедупликация запросов, ленивая загрузка вкладок работают
  как с HTTP. Ничего адаптировать не нужно.

Зачем: переключение между открытыми вкладками = 0 запросов
  (данные из кэша, см. 05/11 раздел 8).
```

═══════════════════════════════════════════════════
3. СОБЫТИЯ → ИНВАЛИДАЦИЯ ★ (ключевой механизм)
═══════════════════════════════════════════════════
```
ПРИНЦИП: компоненты НЕ подписываются на события.
  Один глобальный слой подписок говорит "эти данные протухли",
  React Query сам перезапрашивает то что на экране.

  api.on('subtask.created', ({ project_id, section }) => {
    queryClient.invalidateQueries(['section-tasks', project_id, section])
  })

  api.on('subtask.updated', (task) => {
    queryClient.setQueryData(['subtask', task.id], task)
    queryClient.invalidateQueries(['section-tasks', task.project_id, task.section])
  })

ЧТО ПРОИСХОДИТ ДАЛЬШЕ (React Query знает что на экране):
  вкладка открыта и активна  → перезапрос + перерисовка
  вкладка открыта, неактивна → пометить устаревшим, догрузит при переходе
  вкладка закрыта            → ничего не делает

→ не пишем "если я в компоненте задач, обнови список".
  Компоненты чистые: только отображают данные.
```

## Почему это важно именно для нашей ERP
```
КОНВЕЙЕР: позиция меняет статус в снабжении →
  видно в закупках, дашборде, карточке проекта, списке позиций.

Без инвалидации: ловить событие в 5 местах, дублировать логику.
С инвалидацией: сервер сказал position.status_changed →
  все затронутые ключи протухли → экраны обновятся сами.

~77 событий (03/12) × конвейер = дублирование недопустимо.
```

## Переподключение
```
  api.on('reconnect', () => queryClient.invalidateQueries())

→ всё видимое перезапросится, невидимое пометится устаревшим.
  Одна строка закрывает "что после обрыва связи".
```

═══════════════════════════════════════════════════
4. КЛАССИФИКАЦИЯ СОБЫТИЙ ★ (4 типа поведения)
═══════════════════════════════════════════════════
```
НЕ все события одинаковые. Каждое относится к одному из типов:

① ДАННЫЕ ПРОТУХЛИ → invalidateQueries (глобально)
   subtask.created, position.status_changed, order.paid
   признак: данные с сервера, видят несколько экранов
   компонент НЕ знает о событии

② АДРЕСОВАНО ЧЕЛОВЕКУ → уведомление (push БУДИТ, содержимое из БД)
   subtask.assigned, дедлайн близко, требуется согласование
   признак: человек должен узнать
   было (неточно): notifications.push('текст')  ← только всплывашка, теряется
   стало:
     сервер: INSERT notifications (БД, модель в 01/01) + push "новое уведомление"
     фронт:  push → invalidateQueries(['ErgoArea','Notifications'])
             → счётчик колокольчика + список перезапросились из БД
     колокольчик берёт данные из БД (непрочитанные + история),
       НЕ из текста push-события.
   причина: офлайн/закрытая вкладка/история/is_read — push этого не даёт.

③ ПРОТОКОЛ ОПЕРАЦИИ → обработчик в api-библиотеке
   нужен код подтверждения, 2FA, требуется согласие
   признак: диалог клиент-сервер в рамках одной операции
   ★ не в компоненте — в самой библиотеке (см. раздел 5)

④ ЭФЕМЕРНОЕ ДЛЯ ОКНА → локальная подписка в компоненте
   parse.progress (полоска парсинга счёта),
   присутствие ("Иван смотрит эту задачу"), живой лог операции
   признак: состояние ОДНОГО окна, в кэше ему делать нечего

ПРАВИЛО РАЗЛИЧЕНИЯ:
  данные с сервера + видят другие экраны → ① инвалидация
  эфемерное состояние одного окна        → ④ локальная подписка

Одно событие может быть и ① и ②:
  api.on('subtask.assigned', (e) => {
    queryClient.invalidateQueries(['my-tasks'])          // ① данные
    if (e.responsible_id === me.id)
      notifications.push('Вам назначена задача')          // ② человеку
  })
```

5. ПРОТОКОЛ ОПЕРАЦИИ (код подтверждения) — в библиотеке
═══════════════════════════════════════════════════
```
СЦЕНАРИЙ: отправили запрос → сервер: "нужен код подтверждения" →
  показать диалог ввода → код верный → первый запрос получает ответ.

★ ОБРАБАТЫВАТЬ В API-ОБЁРТКЕ, не в компоненте:

  async request(method, params) {
    const res = await ws.send(method, params)
    if (res.needs_confirmation) {
      const code = await confirmDialog.ask(res.challenge_id)  // глобальный диалог
      return ws.send('confirm', { challenge_id: res.challenge_id, code })
    }
    return res
  }

ПОЧЕМУ не в компоненте:
  - запрос висит в промисе, компонент мог закрыться пока вводят код
  - диалог глобальный (поверх любого экрана)
  - добавили подтверждение ещё для 3 операций → компоненты не трогаем

Компонент пишет обычное:
  await api.request('order.pay', {...})
  и НЕ знает что был код. Диалог всплывёт сам, промис дождётся.
```

6. ЧЕРНОВИКИ И НЕСОХРАНЁННОЕ (dirty tabs) ★
═══════════════════════════════════════════════════
```
СЦЕНАРИЙ (как в VS Code / прежнем приложении пользователя):
  добавил позиции в задачу → кнопка "Сохранить" →
  точка на вкладке (есть изменения) →
  закрываешь вкладку → "Есть несохранённые данные, сохранить?"

МОДЕЛЬ ВКЛАДКИ:
  tab = {
    id, title, kind,
    children?: level,   (вложенные табы, см. 05/11)
    isDirty: bool,      ← есть несохранённые изменения
    draft: {...}        ← локальный черновик (не отправлен)
  }

ПОТОК:
  редактируешь → пишем в draft, isDirty = true → точка на вкладке
  "Сохранить" → api.request('subtask.update', draft)
              → успех: isDirty = false, обновить кэш
  закрыть грязную → модалка "есть несохранённые"

★ РАЗДЕЛЕНИЕ (как файл на диске vs буфер редактора):
  серверные данные → кэш (React Query)
  черновик         → состояние вкладки (draft)
  не пересекаются.
```

7. ИТОГОВЫЙ СТЕК
═══════════════════════════════════════════════════
```
api (свой WS)     — request/response (promise) + события (push)
React Query       — кэш серверных данных (queryFn = api.request)
Слой подписок     — событие → invalidateQueries / уведомление (глобально)
Zustand/Context   — вкладки (openTabs, active, isDirty, draft)

РАЗДЕЛЕНИЕ ОТВЕТСТВЕННОСТИ:
  серверные данные   → React Query
  вкладки, черновики → свой стор
  синхронизация      → события → инвалидация
  протокол операций  → api-обёртка
  Уведомления        → БД (notifications), push будит счётчик колокольчика
  История событий    → БД (event_log), экран "История" (pull, не push)
```

═══════════════════════════════════════════════════
7Б. ЭКРАН "ИСТОРИЯ" — поверх event_log (pull)
═══════════════════════════════════════════════════
```
"Просмотр всех событий" = экран поверх event_log:
  источник — event_log (БД), НЕ push
  GET ErgoArea.EventLog.Query { project_id?, entity?, from?, to? }
  показывает ленту: что, кто, когда, из→в (по каталогу 03/12)
  места: история проекта, история задачи, общий аудит (под правом)
  это PULL (запросил и смотришь), не реактивный push

ТРИ РАЗНЫЕ ВЕЩИ (не путать):
  event_log (БД)     — ВСЁ что произошло (аудит, для всех) → экран История
  notifications (БД) — адресное человеку + is_read → колокольчик (05/00)
  push-события (WS)  — сигнал "обновись", эфемерные, не хранятся
```
