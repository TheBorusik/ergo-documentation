# Глобальный поиск — детальный разбор

Единое поле поиска в шапке по всей системе.
Формат как этапы: назначение → интерфейс → API →
серверная логика (по таблицам) → правила.

═══════════════════════════════════════════════════
1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Одно поле в шапке → искать по ВСЕМ сущностям сразу:
  проекты, позиции, задачи, поставщики/заказчики, закупки (order).

Пользователь вводит текст → видит сгруппированные результаты
по типам → клик открывает сущность (вкладкой).

Цель: быстрый переход к чему угодно без навигации по меню.
```

При вводе (от 2-3 символов) — выпадающая панель:

4. API
═══════════════════════════════════════════════════
```
GET /api/search?q={query}&limit=5
  → быстрый поиск для выпадашки (топ-N каждого типа)
  Ответ:
    {
      projects:    ProjectSearchItem[],
      positions:   PositionSearchItem[],
      subtasks:    SubtaskSearchItem[],
      contractors: ContractorSearchItem[],
      orders:      OrderSearchItem[]
    }
  каждый item: { id, title, subtitle(контекст), type }

GET /api/search/full?q={query}&type={type?}&page={n}
  → страница всех результатов (если нажал "показать все")
  фильтр по типу опционально
```

═══════════════════════════════════════════════════
5. СЕРВЕРНАЯ ЛОГИКА (по таблицам)
═══════════════════════════════════════════════════

───────────────────────────────────────────────────
GET /api/search?q={query} — быстрый поиск (выпадашка)
───────────────────────────────────────────────────
```
0. НОРМАЛИЗАЦИЯ запроса (C#)
   q → lower, trim, убрать лишние пробелы
   (тот же нормализатор что в маппинге — переиспользуем)

Параллельно по каждой сущности (топ-5 каждой):

1. projects        → SELECT где name ILIKE %q% OR code ILIKE %q%
                     + JOIN contractors (customer_name ILIKE %q%)
                     LIMIT 5
                     → фильтр по правам (какие проекты видит юзер)

2. element_positions → SELECT где name ILIKE %q% OR article ILIKE %q%
                     LIMIT 5  → для позиций и номенклатуры

3. project_positions → SELECT где position_id IN (шаг2)
                     + JOIN projects (контекст проекта)
                     LIMIT 5  → позиции в проектах

4. subtasks        → SELECT где name ILIKE %q%
                     + JOIN projects (контекст)
                     LIMIT 5
                     → фильтр по правам

5. contractors     → SELECT где name ILIKE %q% OR inn ILIKE %q%
                     LIMIT 5

6. orders          → SELECT где number ILIKE %q%
                     + JOIN contractors (supplier)
                     LIMIT 5

СБОРКА: сгруппированный ответ по типам

[SQL] ILIKE %q% — простой префиксно-подстрочный поиск.
  Для старта достаточно. Индексы на name/code/article/number.
[C#] нормализация запроса + сборка групп.
[Права] фильтрация проектов/задач по доступу (как везде).
```

───────────────────────────────────────────────────
УЛУЧШЕНИЕ (этап 2) — умный поиск
───────────────────────────────────────────────────
```
ILIKE %q% — простой, но:
  ✗ не находит с опечатками
  ✗ не ранжирует по релевантности

Этап 2 (если нужно):
  - полнотекстовый поиск PostgreSQL (tsvector/tsquery)
    ранжирование по релевантности (ts_rank)
  - ИЛИ fuzzy (pg_trgm, similarity) — опечатки
    (у нас уже есть fuzzy-подход из маппинга)
  - подсветка совпадений

Для старта: ILIKE достаточно (быстро, просто).
Умный поиск — когда данных много и точность важна.
```

═══════════════════════════════════════════════════
6. МОДЕЛИ (для результатов)
═══════════════════════════════════════════════════
```
SearchResult {
  projects: SearchItem[]
  positions: SearchItem[]
  subtasks: SearchItem[]
  contractors: SearchItem[]
  orders: SearchItem[]
}

SearchItem {
  id: int
  type: enum (project|position|subtask|contractor|order|nomenclature)
  title: string        (название/номер)
  subtitle: string?    (контекст для показа: проект, этап, ИНН...)
  # поля контекста — РАЗНЫЕ по type (для построения URL на фронте):
  project_id?: int     (у вложенных: subtask, position)
  section?: string     (у вложенных: supply|works|docs)
}

★ type — КЛЮЧЕВОЕ поле. Глобальный поиск находит разное
  (задачу в проекте, поставщика, закупку, проект...), и фронт
  по type понимает: в какой раздел вести, вложенный или плоский,
  какие поля контекста читать.

Контекст РАЗНЫЙ по типу:
  subtask/position → нужны project_id + section (вложенные)
  project → только id (сам проект)
  contractor/order/nomenclature → только id (плоские разделы)

ФРОНТ строит URL по type (switch на тип):
  subtask     → /project/{project_id}/{section}/task/{id}
  position    → /project/{project_id}/{section}/position/{id}
  project     → /project/{id}
  contractor  → /contractors/{id}
  order       → /orders/{id}
  nomenclature→ /nomenclature/{id}

Бэк отдаёт данные (type + id + контекст).
Фронт по type собирает URL и делает navigate().
Роутер разворачивает вкладки, компоненты грузят своё (+ кэш).
(search/open НЕ нужен — поиск это источник данных, URL строит фронт.)

→ фронт группирует по type, клик открывает по route/id.
```

═══════════════════════════════════════════════════
6Б. ОТКРЫТИЕ РЕЗУЛЬТАТА ВО ВЛОЖЕННОЙ НАВИГАЦИИ ★
═══════════════════════════════════════════════════
```
ПРОБЛЕМА: поиск даёт "плоскую" ссылку на задачу,
  а навигация вложенная (гибрид: проект → раздел → задача).
  Клик на задачу в поиске → надо ВОССТАНОВИТЬ всю цепочку вкладок.

Задача "Воздуховоды П1" живёт:
  Проект "Завод сыра" (верхняя вкладка)
    └ Снабжение (вкладка внутри проекта)
        └ Воздуховоды П1 (задача)

Чтобы открыть — построить цепочку по порядку.
```

## Решение: результат несёт ПОЛНЫЙ путь
```
SearchItem для задачи содержит не только id, а весь путь:
  {
    type: 'subtask',
    id: 42,
    title: 'Воздуховоды П1',
    path: {
      project_id: 1,        (какой проект)
      section: 'supply',    (какой раздел)
      subtask_id: 42        (сама задача)
    }
  }

→ фронт знает ВСЮ цепочку, строит вкладки по порядку.
```

## Механика открытия (фронт)
```
Клик на задачу в поиске → openSearchResult(item):

1. openGlobal(проект)        — открыть/активировать вкладку проекта
   (если уже открыт — переиспользовать, не дублировать)

2. openInLevel(root, раздел) — внутри проекта открыть "Снабжение"
   (в рекурсивном уровне проекта)

3. openInLevel(section, задача) — в ленте снабжения открыть задачу

→ каждый шаг = та же функция что при ручном клике
  (openGlobal, openInLevel — уже есть в навигации!)
  Просто вызываем их последовательно по path.

Если что-то из цепочки уже открыто — переиспользуем
(не плодим дубли вкладок).
```

## Путь для каждого типа результата
```
Проект:
  path: { project_id } → openGlobal(проект), активен Обзор

Позиция:
  path: { project_id, section?, position_id }
  → открыть проект → раздел где позиция → выделить позицию
  (позиция может быть в разделе Позиции проекта)

Задача:
  path: { project_id, section, subtask_id }
  → проект → раздел → задача (3 уровня)

Поставщик:
  path: { contractor_id } → раздел Поставщики (плоско, не в проекте)

Закупка (order):
  path: { order_id } → раздел Закупки (плоско)

Номенклатура:
  path: { element_position_id } → раздел Номенклатура (плоско)
```

## Ключевое — переиспользуем функции навигации
```
НЕ выдумываем отдельный механизм открытия.
Те же openGlobal / openInLevel что при ручных кликах,
  вызванные последовательно по path результата.

Открытие из поиска = автоматизация ручной навигации:
  "открой проект, потом раздел, потом задачу" — программно.
```

## Что упрощает
```
- Часть пути может быть уже открыта → переиспользуем вкладки
- Плоские сущности (поставщик, order, номенклатура) —
  просто раздел меню, без вложенности (проще)
- Вложенность только у проектных (позиция, задача)
- path строит БЭК (он знает где что лежит), фронт исполняет
```

═══════════════════════════════════════════════════
6В. КАСКАД ЗАПРОСОВ ПРИ ОТКРЫТИИ — как избежать мелькания
═══════════════════════════════════════════════════
```
ПРОБЛЕМА: разворот пути = 3 запроса подряд
  (проект → раздел → задача). Наивно = 3 перерисовки, мелькание.

РЕШЕНИЕ — несколько приёмов, комбинируются:
```

## Приём 1 — открывать СРАЗУ конечную вкладку (не рендерить промежуточные)
```
Не показываем каждый шаг. Строим структуру вкладок в памяти,
активируем СРАЗУ конечную (задачу), грузим только её данные.

  path → создать вкладки [проект→раздел→задача] в состоянии
       → активной сделать ЗАДАЧУ
       → загрузить данные ТОЛЬКО задачи (1 запрос)
       → промежуточные (проект, раздел) — данные лениво,
         когда переключишься на них

→ 1 запрос (задача), не 3. Промежуточные вкладки —
  пустые каркасы, наполняются при клике на них.
  Нет мелькания (сразу конечный экран).
```

## Приём 2 — один запрос с полными данными пути
```
БЭК отдаёт ВСЁ разом для открытия из поиска:
  GET /api/search/open?type=subtask&id=42
  → { project: {...}, section: {...}, subtask: {...} }
    (проект + раздел + задача одним ответом)

→ 1 запрос вместо 3. Фронт строит вкладки из готовых данных.
  Нет каскада, нет мелькания.
  (чуть больше данных в одном ответе — но один roundtrip)
```

## Приём 3 — скелетоны вместо белых экранов
```
Пока грузится — показываем skeleton (серые плейсхолдеры),
не белый экран. Даже если каскад — не "мелькание пустотой",
а плавное появление контента.

→ каскад визуально сглажен (скелетон → данные).
```

## Приём 4 — переход показываем один раз (loading на финале)
```
Клик в поиске → индикатор загрузки НА ПОИСКЕ →
  когда всё готово → открыть сразу конечную вкладку.
Пользователь не видит промежуточные шаги —
  видит "загрузка" → финальный экран.
```

## РЕКОМЕНДАЦИЯ (комбинация)
```
Для открытия из поиска:
  Приём 2 (один запрос /search/open — данные всего пути)
  + Приём 1 (активна сразу конечная, промежуточные лениво)
  + скелетон на время загрузки

→ 1 запрос, сразу конечный экран, без мелькания.

Для РУЧНОЙ навигации (клик за кликом):
  каждый шаг = свой запрос — это НОРМАЛЬНО
  (пользователь сам идёт по одному, ждёт каждый экран,
   мелькания нет — он видит один переход за раз)
```

## Плоский vs вложенный — про запросы честно
```
ВЛОЖЕННЫЙ (гибрид):
  открытие из поиска — каскад ИЛИ один /open запрос (решаемо)
  ручная навигация — запрос на шаг (нормально, ждёшь один экран)

ПЛОСКИЙ:
  открытие из поиска — сразу задача, 1 запрос (проще)
  НО контекст (проект/раздел) не на экране →
    хлебные крошки, клик по ним = те же запросы (отложены)

Вывод: плоский НЕ убирает запросы, а откладывает их.
  Вложенный с приёмом /search/open — тоже 1 запрос на открытие.
  Разница не в количестве запросов, а в том ГДЕ контекст:
    вложенный — контекст в структуре вкладок (видно путь)
    плоский — контекст в крошках (компактнее, но менее явно)
```

## Итог по мельканию
```
Мелькание — НЕ обязательное свойство вложенности.
Оно возникает только при наивной реализации
  (рендерить каждый промежуточный шаг).

Убирается:
  ✓ один запрос /search/open (данные всего пути разом)
  ✓ активна сразу конечная вкладка (промежуточные лениво)
  ✓ скелетоны (не белые экраны)

→ открытие из поиска = один запрос → сразу задача, плавно.
  Вложенность не мешает, если открывать умно.
```

7. ПРАВИЛА
═══════════════════════════════════════════════════
```
- Поиск от 2-3 символов (не грузить на 1 символ)
- Дебаунс ввода (~300мс) — не запрос на каждую букву
- Топ-5 каждого типа в выпадашке (не перегружать)
- Права: результаты фильтруются по доступу пользователя
  (не показывать проекты/задачи которые не видит)
- Клик → открыть сущность вкладкой (в системе вкладок)
- "Показать все" → полная страница (если результатов много)
- Нормализация запроса (переиспользуем из маппинга)
- Пустой результат → "Ничего не найдено по «query»"
```

10. ВОПРОСЫ
═══════════════════════════════════════════════════
```
1. Искать номенклатуру (эталон) вместе с позициями или отдельно?
2. Поиск по позициям — по номенклатурному имени или и по проекту?
3. Нужен ли поиск по содержимому файлов (документы)? — вряд ли на старте
4. "Показать все" — нужна отдельная страница или хватит выпадашки?
5. История поиска / недавнее — нужно?
```
