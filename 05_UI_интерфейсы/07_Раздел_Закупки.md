# Закупки (Order) — самостоятельная сущность

Закупка (order) — отдельная сущность, НЕ привязана к задаче.
Позиции ссылаются на order через order_id.

═══════════════════════════════════════════════════
КЛЮЧЕВОЕ РЕШЕНИЕ (уточнено)
═══════════════════════════════════════════════════
```
Закупка (order) — САМОСТОЯТЕЛЬНАЯ сущность.
НЕ привязана к задаче снабжения (связь через задачу нелогична).

Задача снабжения — рабочий процесс (собрать, запросить цены).
Order (закупка) — финансовый факт (заказ поставщику, оплата).
Это РАЗНЫЕ действия.

Связь: ПОЗИЦИЯ знает свой order (position.order_id),
  а не задача знает закупку.
```

═══════════════════════════════════════════════════
ПОЧЕМУ order НЕ привязан к задаче
═══════════════════════════════════════════════════
```
Жёсткая связь order↔задача мешает реальности:
  - можно закупить позиции из РАЗНЫХ задач в ОДИН order
    (объединить в одну поставку у поставщика)
  - можно из ОДНОЙ задачи сделать НЕСКОЛЬКО order
    (разным поставщикам)

→ связь на уровне ПОЗИЦИИ (position.order_id) — гибко
  order собирает позиции обратной связью
```

═══════════════════════════════════════════════════
МОДЕЛЬ
═══════════════════════════════════════════════════
```
orders (закупки — самостоятельная сущность)
  id
  number: string           (номер закупки)
  supplier_id: FK→Contractor
  total_amount: decimal
  payment_status: enum (draft | approved | in_payment | paid)
  approved_by: FK→User?, approved_at: datetime?
  paid_by: FK→User?, paid_at: datetime?
  invoice_file: string?    (счёт)
  created_at, created_by

project_positions (+ связь с закупкой)
  ... существующие поля ...
  order_id: FK→Order?      ← в какой закупке куплена ← НОВОЕ

Связь:
  позиция → order_id (одна позиция в одной закупке)
  order → позиции (обратно: SELECT positions WHERE order_id={id})

Позиция несёт связи:
  kp_item_id      — из записи КП
  spec_subtask_id — из спецификации
  order_id        — в какой закупке ← новое
```

═══════════════════════════════════════════════════
СВЯЗЬ С ЗАДАЧЕЙ СНАБЖЕНИЯ (переход)
═══════════════════════════════════════════════════
```
Задача снабжения: выбор позиций, запрос цен, маппинг (рабочее)
  ↓
Создание ORDER (закупки):
  выбрали позиции + поставщика → оформили order
  позициям проставился order_id
  ↓
Order: согласование → оплата → доставка (финансовое)

Order может собрать позиции из разных задач.
Задача может породить несколько order.

ВОПРОС: где именно граница задача→order?
  (задача готовит цены → order оформляет заказ и оплату?)
```

═══════════════════════════════════════════════════
ФИНАНСОВЫЙ ПРОЦЕСС ORDER
═══════════════════════════════════════════════════
```
order.payment_status:
  draft      — черновик (собираются позиции)
  approved   — согласовано (менеджер)
  in_payment — в оплате (бухгалтерия)
  paid       — оплачено

Это финансовый статус ЗАКУПКИ (отдельно от статуса позиций).
Позиции при оплате order → purchased (конвейер).
```

═══════════════════════════════════════════════════
API
═══════════════════════════════════════════════════
```
GET /api/orders
  Параметры: supplier_id?, status?, period, project_id?
  → все закупки
  Ответ: { items: Order[], totals: {...} }

GET /api/orders/{id}
  → деталь + позиции (SELECT positions WHERE order_id)
    сгруппированы по проектам

POST /api/orders
  Тело: { supplier_id, position_ids: [] }
  → создать закупку, проставить order_id позициям
  Ответ: Order

PATCH /api/orders/{id}/approve  → согласовать
PATCH /api/orders/{id}/pay      → оплатить
  (при оплате: позиции → purchased)

POST /api/orders/{id}/positions → добавить позиции в order
DELETE /api/orders/{id}/positions/{posId} → убрать
```
