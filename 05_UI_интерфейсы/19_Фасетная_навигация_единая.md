# Единая фасетная навигация (scope-контракт)

Решение 2026-07-22 (обсуждение с пользователем). Один механизм фильтра
по номенклатуре на ВСЁ приложение: каталог, изменение состава,
поставщики, позиции проекта, закупки. Устраняет накопившийся разнобой
(где-то фасеты, где-то только текстовый поиск; чипы то есть, то нет).

═══════════════════════════════════════════════════
1. ДВА УРОВНЯ: фасеты по номенклатуре, результат по scope
═══════════════════════════════════════════════════
```
Фасеты (категория + свойства) ЖИВУТ НА УРОВНЕ НОМЕНКЛАТУРЫ
(element_positions — у товара есть категория и свойства).
Результат-список — на уровне SCOPE-сущности:
  catalog  → сами товары (element_positions)
  project  → экземпляры project_positions (статус/задача/КП)
  supplier → строки order_items (цена/дата/заказ)

Фильтр по номенклатуре сужает, КАКИЕ экземпляры показать.
Категория/свойство — про товар; цена/статус/дата — про экземпляр.
```

═══════════════════════════════════════════════════
2. SCOPE — «откуда взять множество номенклатуры» (шаг 0)
═══════════════════════════════════════════════════
```
Единственное, что различается между разделами — ШАГ 0: из какой
таблицы и по какому условию собрать множество nomenclature_id
(element_positions.id). ВЕСЬ алгоритм после — общий.

scope           шаг 0 — множество берётся из
─────────────── ──────────────────────────────────────────────
catalog         element_positions (все)
project:{id}    project_positions WHERE project_id → distinct position_id
supplier:{id}   order_items WHERE supplier_id → distinct nomenclature_id
order:{id}      order_items WHERE order_id → distinct nomenclature_id
purchases       все order_items → distinct nomenclature_id
(+ расширяемо: warehouse:{id}, all-positions…)

scope может нести доп-предикат (напр. project + доступность по
статусу в окне выбора позиций) — влияет только на шаг 0.

НАГРУЗКА: по номенклатуре потолок — catalog (весь каталог, конечен,
кэшируем). По СТРОКАМ результата потолок — purchases (история
order_items растёт бесконечно) → результат ВСЕГДА пагинируется.
```

═══════════════════════════════════════════════════
3. ДВА ЗАПРОСА (раздельные — по-разному ведут себя с фильтрами)
═══════════════════════════════════════════════════
```
Facets { scope, categoryId?, filters? } → { Categories[], Facets[] }
  Отдаёт колонку 1 (категории со счётчиками, только релевантные scope)
  и колонку 2 (свойства + значения + счётчики). Кэшируем по
  (scope, category[, filters]).

Query { scope, categoryId?, filters?, search?, page } → результат
  ПЕР-РАЗДЕЛ (разные сущности/поля), но ОДИНАКОВЫЙ фасет-контракт.
  Порядок на сервере:
    0. scope → базовое множество nomenclature_id (всегда, независимо
       от фильтров)
    1. + categoryId (сужает через element.category_id)
    2. + search (имя/артикул)
    3. + filters (значения свойств — В КОНЦЕ)
    4. пагинация
  Для scope=project/supplier: фильтр даёт множество проходящих
  nomenclature_id → показываем экземпляры с этими номенклатурами.

Хелпер сервера: matchNomenclatureIds(scopeNomIds, category, filters,
  search) → Set<element_position_id> — переиспользуется обоими Query
  (catalog отдаёт сами позиции; scope-разделы джойнят экземпляры).
```

═══════════════════════════════════════════════════
4. DRILL-DOWN (каскадные фасеты)
═══════════════════════════════════════════════════
```
Выбор значения пересчитывает счётчики ДРУГИХ свойств под текущий
выбор (как в маркетплейсах). Facets принимает filters:
  для каждого свойства P:
    1. scope-множество
    2. применить фильтры ВСЕХ свойств, КРОМЕ P  ← «исключить своё»
    3. счётчики значений P по оставшимся

«Исключить своё» обязательно: иначе выбор одного значения свойства
обнулит остальные значения ЭТОГО ЖЕ свойства, и мультивыбор сломается.

СЕМАНТИКА:
  внутри свойства  → OR  (Диаметр = М8 ИЛИ М10 — болты обоих)
  между свойствами → AND (Диаметр М8 И Металл оц.сталь)
Переключатель AND/OR НЕ нужен: свойства номенклатуры однозначны
(один диаметр на позицию), AND внутри свойства всегда пусто. OR
внутри уже даёт «выбрать несколько диаметров — показать все».
(AND внутри осмыслен лишь для многозначных свойств — сертификаты,
совместимость; появятся — добавим точечно.)
```

═══════════════════════════════════════════════════
5. UI-ПОВЕДЕНИЕ
═══════════════════════════════════════════════════
```
НУЛИ — серые неактивные (0), НЕ скрывать. Скрытие дезориентирует
  (пользователь думает «фильтр пропал»), прыгает интерфейс, теряется
  «вариант существует, но не в этом срезе». Скрывать — только если
  значений сотни (у нас не тот случай).
  Своё свойство при этом не гаснет (внутри — OR, можно добавить).

ПОКАЗАТЬ ЕЩЁ — порог N (напр. 8) на свойство; сверх — «Показать ещё
  (M)» / «Свернуть». Порядок значений: выбранные → доступные (>0) →
  серые нули. Длинные списки (диаметры, размеры) сортируем по
  sort_order справочника (стабильно, не прыгает); короткие — по
  счётчику. Обрезка в UI, сервер отдаёт все значения свойства.

ЧИПЫ активных фильтров — ВСЕГДА в тулбаре, даже когда панель закрыта
  (закрытие панели ≠ сброс фильтров). ✕ на каждом + «сбросить всё».
  Панель = где ВЫБИРАЕШЬ; чипы = что ПРИМЕНЕНО.

ЗНАЧЕНИЯ Unit-свойств (ширина/высота/угол…) — НЕ свободный ввод числа,
  а список выбираемых значений (чекбоксы), как у перечислений. Причина
  отказа от свободного ввода: поле числа моргало и теряло фокус при
  рефетче; выбор из готовых значений удобнее и не ломается.
  УНИВЕРСУМ значений Unit — числа, встречающиеся в scope ∩ категория
  (БЕЗ drill-фильтров), ReferenceId = само число. Счётчик считается по
  прошедшим drill-down; значение с Count=0 → СЕРЫЙ неактивный ноль
  (как у enum), НЕ скрываем. Прежнее решение «у Unit нулей нет, только
  присутствующие» ОТМЕНЕНО (2026-07-23): при выборе, напр., «Угол» из
  среза пропадали значения «Длины» и само свойство исчезало — это
  дезориентировало; теперь свойство/значения СЕРЕЮТ, а не исчезают.
  (Число бесконечно только в теории — берём фактический универсум среза,
  он конечен и не «выдуман».)

БЕЗ МОРГАНИЯ — запрос Facets (и Query разделов) держит placeholderData:
  keepPreviousData (React Query v5): при рефетче на смену фильтра данные
  НЕ обнуляются, панель не пересобирается, поля не перемонтируются →
  нет мерцания навигации и потери фокуса в полях ввода.

ПАНЕЛЬ — overlay (всплывает поверх, Esc/клик вне) или static
  (докнутая колонка, ужимает список). Кнопка «Фильтр» (бейдж со
  счётчиком активных) открывает; 📌 pin/unpin переключает overlay⇄
  static на лету. Дефолт per-раздел в коде, выбор пользователя
  перекрывает и СОХРАНЯЕТСЯ в localStorage per-scope (как erp_theme).
  Дефолты: каталог/изменить состав — static+открыто; поставщики/
  позиции/закупки — overlay+закрыто.
```

═══════════════════════════════════════════════════
6. КОМПОНЕНТ: общий фильтр + уникальный результат
═══════════════════════════════════════════════════
```
ОБЩЕЕ (один компонент на всё): колонка категорий, колонка свойств
  (drill-down/нули/показать ещё), чипы, поиск, кнопка+pin, пагинатор,
  состояние FacetState (categoryId, search, filters).

УНИКАЛЬНОЕ (per-раздел): scope (+доп-предикат), результат-список
  (какие колонки/данные), действия на строках (в задачу/убрать/
  история/split).

СВЯЗЬ: общий компонент — управляемый, отдаёт FacetState наружу;
  раздел кладёт его в свой Query{scope,…} и РИСУЕТ результат
  (children/render-prop). Панель+обвязка общие, начинка справа — своя.
```
