Формат: интерфейс (ASCII) → действия → API (ссылки на МОДЕЛИ_API.md) → таблицы.
Финальный этап. Самый простой.

1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Менеджер отправляет финальный документ заказчику, фиксирует
приёмку и финальную оплату, завершает проект.

Роль:  Менеджер
Вход:  все позиции задокументированы (documented) + сводный документ
Выход: проект "Завершён" (completed)
```

3. ДЕЙСТВИЯ СОТРУДНИКА
═══════════════════════════════════════════════════
```
МЕНЕДЖЕР:
  ① Проверить готовность (чек-лист)
  ② Отправить финальный документ заказчику
  ③ Отметить принятым заказчиком
  ④ Отметить финальную оплату
  ⑤ Завершить проект (→ completed)
```

═══════════════════════════════════════════════════
4. ПРАВИЛА
═══════════════════════════════════════════════════
```
- Готовность: все позиции documented, сводный документ есть
- Завершение → projects.status = completed + completed_at
- Финальная оплата — часть модели цен (позже, пока флаг)
- Проект в архив завершённых (не удаляется)
- Что если не принял (замечания) — доработка (позже)
```

═══════════════════════════════════════════════════
5. СТАТУСЫ
═══════════════════════════════════════════════════
```
Проект (Project.status):
  active → won → ... → completed

Этап без сложных подзадач (в отличие от 1-4):
  простой чек-лист + финальные действия
  (можно без subtasks — прямые операции над проектом)

Внутренние состояния сдачи (опц.):
  ready → sent (документ отправлен) →
  accepted (принят) → paid (оплачен) → completed
```

6. API (детально)
═══════════════════════════════════════════════════

## 6.1 Готовность к сдаче
```
GET /api/projects/{projectId}/handover-status
  (чек-лист готовности)
  Ответ: 200
    {
      positions_total: int,
      positions_installed: int,
      positions_documented: int,
      summary_ready: bool,        (сводный документ есть)
      acts_signed: int, acts_total: int,
      ready_for_handover: bool
    }
```

## 6.2 Финальный документ и приёмка
```
GET /api/projects/{projectId}/final-document
  Ответ: 200 { file_path: string }  (сводный PDF)

POST /api/projects/{projectId}/handover/send
  (отправить заказчику)
  Тело: { email_to?: string }
  Ответ: 200 { sent_at: datetime }
  Побочно: handover.status → sent

POST /api/projects/{projectId}/handover/accept
  (отметить принятым)
  Тело: { accepted_by?: string, accepted_at?: date }
  Ответ: 200 Handover  (status → accepted)

POST /api/projects/{projectId}/handover/payment
  (финальная оплата — заглушка, часть модели цен)
  Тело: { confirmed: bool }
  Ответ: 200 Handover  (final_payment_confirmed = true)
```

## 6.3 Завершение проекта
```
POST /api/projects/{projectId}/complete
  Ответ: 200 Project  (status → completed, completed_at = now)
  Ошибки: 409 ApiError
    (не все позиции documented / не принят / не оплачен)
  Побочно: проект в архив (по фильтру status)
```

6Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════
Простой этап. [SQL] — где сложный запрос.

───────────────────────────────────────────────────
GET /api/projects/{projectId}/handover-status
Чек-лист готовности к сдаче
───────────────────────────────────────────────────
```
1. project_positions → SELECT COUNT где project_id={id}
                    группировка по status
                    → всего / installed / documented

2. position_documents → SELECT COUNT где position_id IN (позиции)
                    → сколько с документами

3. work_acts      → SELECT COUNT где project_id={id}
                    группировка по status
                    → актов всего / подписано

СБОРКА: {
  positions_total, positions_installed, positions_documented,
  acts_signed, acts_total,
  summary_ready (есть ли сводный документ),
  ready_for_handover (все documented И акты подписаны)
}

[SQL] Шаг1 — один SELECT COUNT ... GROUP BY status.
  Проверка готовности — агрегаты, эффективнее одним запросом.
```

───────────────────────────────────────────────────
GET /api/projects/{projectId}/final-document
Финальный документ
───────────────────────────────────────────────────
```
1. projects       → SELECT final_document_path где id={id}
   ИЛИ documents-summary (сводный из документации)

Возврат: { file_path }
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/handover/send
Отправить заказчику
───────────────────────────────────────────────────
```
1. projects       → SELECT где id={id}  → email заказчика, документ

2. contractors    → SELECT где id={customer_id}  → email

3. (отправка email с финальным документом — внешний сервис)

4. projects (или handover) → UPDATE handover_status='sent',
                    handover_sent_at=now
   ИЛИ handover → INSERT/UPDATE (если отдельная таблица)

5. event_log      → INSERT 'handover_sent' (detail: заказчику)

Возврат: { sent_at }
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/handover/accept
Отметить принятым
───────────────────────────────────────────────────
```
1. projects (или handover) → UPDATE handover_status='accepted',
                    accepted_by, accepted_at

2. event_log      → INSERT 'handover_accepted' (detail: принял)

Возврат: Handover (или Project)
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/handover/payment
Финальная оплата (заглушка)
───────────────────────────────────────────────────
```
1. projects (или handover) → UPDATE final_payment_confirmed=true

2. event_log      → INSERT 'handover_paid'

Возврат: Handover
Примечание: детали оплаты — часть модели цен (позже)
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/complete
Завершить проект
───────────────────────────────────────────────────
```
1. project_positions → SELECT COUNT где project_id={id}
                    AND status != 'documented'
                    → ПРОВЕРКА: все documented?
                    → если есть недокументированные → 409

2. projects       → SELECT handover_status, final_payment где id={id}
                    → ПРОВЕРКА: принят и оплачен?
                    → если нет → 409

3. projects       → UPDATE status='completed', completed_at=now
                    где id={id}

4. project_stages → UPDATE status='completed' где project_id={id}
                    AND stage_type='handover'
                    → этап сдачи завершён

5. event_log      → INSERT 'project_completed'

Возврат: Project
→ проект в архив (по фильтру status=completed)

[SQL] Шаг1 — COUNT с условием (проверка полноты одним запросом).
```

───────────────────────────────────────────────────
СВОДКА — таблицы этапа Сдача
───────────────────────────────────────────────────
```
projects:          SELECT, UPDATE (status, completed_at, handover поля)
project_positions: SELECT COUNT (проверка готовности)
position_documents: SELECT COUNT (документы)
work_acts:         SELECT COUNT (акты)
contractors:       SELECT (email заказчика)
project_stages:    UPDATE (этап сдачи завершён)
handover:          опц. (если отдельная таблица)
event_log:         INSERT
```

```
ЧИТАЕТ:

  без отдельной таблицы handover — этап простой.
```
