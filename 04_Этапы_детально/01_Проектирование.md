## 2.3 Модалка "Добавить позицию" — ФАСЕТНЫЙ ПОИСК
```
Переиспользует фасетную навигацию каталога номенклатуры
(категории + динамические фасеты из свойств + поиск).

┌─ Добавить позицию ─────────────────────────────────────────────┐
│ Поиск: [___________________________] 🔍                        │
├────────────┬──────────────────┬────────────────────────────────┤
│ КАТЕГОРИИ   │ ФАСЕТЫ           │ ПОЗИЦИИ                         │
│            │                  │                                │
│ ▸ Вентиляц.│ Тип изделия ▾    │ ○ Воздуховод 500х300х1250 оц   │
│   Воздухов.│  ☑ Воздуховод    │ ○ Воздуховод 500х300х2000 оц   │
│   Отводы   │  ☐ Отвод         │ ○ Воздуховод 500х400х1250 оц   │
│ ▸ Крепёж   │ Сечение ▾        │ ○ Воздуховод 400х200х1250 оц   │
│ ▸ Оборудов.│  ☑ Прямоугол.    │                                │
│            │ Металл ▾         │ найдено: 4                     │
│            │  ☑ оц.сталь      │                                │
│            │ Ширина ▾         │                                │
│            │  [400][500]...   │                                │
│            │ [Сбросить]       │                                │
├────────────┴──────────────────┴────────────────────────────────┤
│ Не нашли? [+ Создать новую позицию]                            │
│ ─────────────────────────────────────────────────────          │
│ Выбрано: Воздуховод 500х300х1250 оц                            │
│ Количество: [14]    Привязка к КП: [Приточная П1 ▾]           │
│                                  [Отмена] [Добавить]           │
└─────────────────────────────────────────────────────────────────┘

Три колонки (как каталог номенклатуры):
  КАТЕГОРИИ — дерево категорий (клик → фильтр)
  ФАСЕТЫ    — динамические фильтры из свойств
              (тип изделия, сечение, металл, размеры...)
              значения-чекбоксы, диапазоны для чисел
  ПОЗИЦИИ   — результат (сужается фасетами + поиском)
Выбор позиции (radio) → количество + КП → добавить.
Быстрый поиск: сначала фасеты сужают, потом выбор.
[+ Создать новую] — если позиции нет (→ конструктор).
```

### ★ УТОЧНЕНИЕ (2026-07-21, реализовано) — «Изменить состав» через ЧЕРНОВИК
```
Кнопка в задаче: [Изменить состав] (не «Добавить позицию»).
Окно ШИРОКОЕ, но читается как модалка (~1240px, уточнение
2026-07-21 — не fullscreen). 4 колонки, вертикальный скролл у
КАЖДОЙ отдельно; у каталога И корзины одинаковые шапки колонок
и пагинация (симметрия). Заблокированные позиции в корзине —
группой внизу, значения ТЕКСТОМ (как в задаче):

┌─ Изменить состав спецификации ────────────────────────────────┐
│ [🔍 поиск]                                                    │
│ (Категория: Крепёж ✕)(Металл: оц.сталь ✕)  [Сбросить всё]    │ ← чипы активных
├──────────┬─────────┬──────────────────┬───────────────────────┤   фильтров с ✕
│ КАТЕГОРИИ │ ФАСЕТЫ  │ РЕЗУЛЬТАТЫ       │ В СПЕЦИФИКАЦИИ (4)    │
│ Все       │ Ширина  │ Возд. 500х… [→] │ имя │кол│КП▾│✕        │ ← КП per-СТРОКА
│ ▸ Вентиляц│ [....]  │ Болт М10х50 [→] │ Воздуховод ×20 [П1▾] │   (не общий селект)
│   Крепёж  │ ☑ оц.ст │                 │ Отвод ×6 · новая      │
│           │[Сброс]  │ 1–8 из 13  ◀ ▶  │                       │
├──────────┴─────────┴──────────────────┴───────────────────────┤
│ [+ Создать новую позицию]            [Отмена] [ИЗМЕНИТЬ]      │
└───────────────────────────────────────────────────────────────┘

[Изменить] НЕ вызывает API — возвращает ЧЕРНОВИК в задачу.
ВСЁ редактирование состава — через черновик (и окно, и инлайн-
правки количества/КП/удаление в таблице задачи):
  черновик ≠ пусто → dirty-точка ● на вкладке (механизм 05/12 §6,
  SaveGuard: «Сохранить и закрыть» в диалоге несохранённого),
  в задаче кнопки [Сохранить состав] [Отменить изменения];
  готовность/завершение — немедленные, при черновике заблокированы.
[Сохранить состав] применяет РАЗНИЦУ: новые → Positions.Add,
убранные → Positions.Delete, кол-во/КП → Positions.Update.
→ тот же паттерн черновика, что в снабжении (05/03) —
  один механизм на все составы задач.
```

### Как работают фасеты (кратко)
```
Клик категории → показывает релевантные фасеты этой категории
Отметил фасет (Металл=оц.сталь) → список сужается + другие фасеты
  пересчитываются (показывают только доступные значения)
Числовые свойства (Ширина) → выбор значения/диапазона
Двунаправленно: можно от свойства, можно от категории
(механизм из nomenclature_full.md, раздел фасетной навигации)
```

  Обратный перевод (готова → проектируется) разрешён, пока позицию
  не взяло снабжение. + кнопка [Все готовы к закупке (N)] для всех разом.
  При несохранённом черновике состава переключение заблокировано.
⑧ Прикрепить чертежи
⑨ Завершить спецификацию

4. ПРАВИЛА
═══════════════════════════════════════════════════
```
- КП необязательна, но желательна (без неё — "не привязана")
- Редактирование позиции: только status IN (created, spec_ready)
  и current_subtask относится к проектированию.
  В закупке (in_purchase+) — запрещено.
- UI (2026-07-21): нередактируемые позиции показываются ТЕКСТОМ
  (без полей ввода) отдельной группой «Переданы дальше по конвейеру —
  только просмотр». Подписи статусов в контексте проектирования:
  created=«Проектируется», spec_ready=«Готова к закупке»,
  in_purchase=«В закупке», ready_for_payment=«Ожидает оплаты»,
  purchased=«Закуплена», delivered=«Доставлена» и т.д.
- Завершение → все позиции авто spec_ready
- Досрочная пометка готовности (опережающий поток, частями)
- Эксклюзивность: позиция в одной активной задаче

## 5.3 Работа с позициями
```
POST /api/projects/{projectId}/positions
  (добавить позицию в спецификацию)
  Тело:
    {
      position_id: int,        (FK номенклатуры)
      quantity: decimal,
      kp_item_id: int?,        (привязка к КП, опц.)
      spec_subtask_id: int     (в какую спецификацию)
    }
  Ответ: 201
    PositionDetail            (status = created)
  Побочно: subtask_positions (связь), position_history (created)

PATCH /api/positions/{id}
  (изменить количество / привязку к КП)
  Тело:
    { quantity?: decimal, kp_item_id?: int }
  Ответ: 200
    PositionDetail
  Ошибки: 409 ApiError (позиция в закупке — редактирование запрещено)

DELETE /api/positions/{id}
  Ответ: 204
  Ошибки: 409 ApiError (позиция не в статусе проектирования)

PATCH /api/positions/{id}/ready
  (пометить одну готовой)
  Тело: {}
  Ответ: 200
    PositionDetail            (status: created → spec_ready)
  Побочно: position_history (ready)

POST /api/subtasks/{id}/positions/mark-ready
  (перевод статуса: кнопки на строке или «все готовы»)
  Тело:
    { position_ids: int[] | "all", ready?: bool }
    ready: true (по умолчанию) — created → spec_ready;
    ready: false — ОБРАТНО spec_ready → created, только для позиций,
    не взятых снабжением (current_subtask_id IS NULL)
  Ответ: 200
    { updated: int, items: PositionDetail[] }
```

## 5.4 Фасетный поиск / создание номенклатуры
```
GET /api/nomenclature/categories
  (дерево категорий для левой колонки)
  Ответ: 200
    { items: Category[] }

GET /api/nomenclature/facets?category_id={id}&filters={...}
  (динамические фасеты — пересчитываются от выбранных фильтров)
  Параметры:
    category_id: int?        (выбранная категория)
    filters: object?         (уже выбранные фасеты)
             { property_id: [values] }
  Ответ: 200
    { facets: Facet[] }
    где Facet { property_id, property_name, type (enum|range),
                values: [{ value, label, count }] |
                range: { min, max } }

GET /api/nomenclature/positions/search
  (результат поиска с фасетными фильтрами)
  Параметры:
    q: string?               (текстовый поиск)
    category_id: int?
    filters: object?         ({ property_id: [values] })
    limit: int, offset: int
  Ответ: 200
    { items: NomenclatureSearchResult[], total: int }

POST /api/nomenclature/positions
  (создать новую позицию — если не нашли)
  Тело:
    {
      element_id: int,
      properties: [ { property_id: int, value: string|int } ]
    }
  Ответ: 201
    ElementPosition           (name/article сгенерированы)
```

Фасетный поиск переиспользует механизм каталога номенклатуры
(nomenclature_full.md, раздел 5 — фасетная навигация).
Тот же API что и в разделе Номенклатура.

5Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════
Каждый API — последовательность операций по ОДНОЙ таблице.
Шаг → таблица → операция → что делаем.
[SQL] — где сложный/объединённый запрос оправдан (скорость).

═══════════════════════════════════════════════════
GET /api/projects/{projectId}/subtasks?stage=design
Список задач-спецификаций
═══════════════════════════════════════════════════
```
1. subtasks       → SELECT где project_id={id} AND stage_id=design
                    → список подзадач (id, name, status, responsible_id)

2. users          → SELECT где id IN (responsible_id из шага 1)
                    → имена ответственных (map: user_id → name)

3. subtask_positions → SELECT COUNT где subtask_id IN (шаг 1)
                       группировка по subtask_id
                    → кол-во позиций на задачу

4. project_positions → SELECT COUNT где id IN (позиции шага 3)
                       AND status='spec_ready'
                    → кол-во готовых на задачу

СБОРКА (в коде): объединить 1+2+3+4 → SubtaskListItem[]
  каждая задача: + responsible_name + positions_count + ready_count

[SQL] Шаги 3+4 можно одним запросом с GROUP BY + FILTER:
  SELECT subtask_id, COUNT(*),
         COUNT(*) FILTER (WHERE status='spec_ready')
  FROM subtask_positions sp JOIN project_positions pp ...
  → сильно ускорит (иначе N запросов на задачу)
```

═══════════════════════════════════════════════════
POST /api/projects/{projectId}/subtasks
Создать спецификацию
═══════════════════════════════════════════════════
```
1. projects       → SELECT где id={id}
                    → проверка существования, статус проекта

2. subtasks       → INSERT (project_id, stage_id=design,
                    name, status='new', created_at)
                    → новая подзадача

3. event_log      → INSERT (project_id, event_type='subtask_created',
                    entity_type='subtask', entity_id=новый id,
                    detail=name, changed_by, changed_at)

Возврат: Subtask (из шага 2)
```

═══════════════════════════════════════════════════
PATCH /api/subtasks/{id}/take
Взять в работу
═══════════════════════════════════════════════════
```
1. subtasks       → SELECT где id={id}
                    → проверка (статус new? существует?)

2. subtasks       → UPDATE status='in_progress',
                    responsible_id={currentUser} где id={id}

3. event_log      → INSERT (event_type='subtask_taken',
                    entity_type='subtask', entity_id={id},
                    from_status='new', to_status='in_progress',
                    detail=responsible, changed_by)

Возврат: Subtask (обновлённая)
```

═══════════════════════════════════════════════════
GET /api/subtasks/{id}
Деталь спецификации
═══════════════════════════════════════════════════
```
1. subtasks       → SELECT где id={id}
                    → сама подзадача

2. users          → SELECT где id={responsible_id}
                    → имя ответственного

3. subtask_positions → SELECT где subtask_id={id}
                    → id позиций этой задачи

4. project_positions → SELECT где id IN (шаг 3)
                    → данные позиций (status, quantity, kp_item_id...)

5. element_positions → SELECT где id IN (position_id из шага 4)
                    → названия/артикулы номенклатуры

6. kp_items       → SELECT где id IN (kp_item_id из шага 4)
                    → названия КП-записей (для отображения привязки)

7. file_attachments → SELECT где entity_type='subtask' AND entity_id={id}
                    → чертежи/файлы

СБОРКА: SubtaskDetail (подзадача + positions[] с именами + files[])
  позиции = шаг4 + имена(шаг5) + КП-имена(шаг6)

[SQL] Шаги 4+5+6 можно одним join:
  project_positions JOIN element_positions JOIN kp_items
  → PositionDetail[] сразу с именами (удобнее, быстрее)
```

═══════════════════════════════════════════════════
POST /api/projects/{projectId}/positions
Добавить позицию в спецификацию
═══════════════════════════════════════════════════
```
1. element_positions → SELECT где id={position_id}
                     → проверка что номенклатура существует

2. kp_items       → SELECT где id={kp_item_id} (если указан)

3. project_positions → INSERT (project_id, position_id, quantity,
                     kp_item_id, spec_subtask_id={subtask},
                     status='created', created_at)
                     → новая позиция конвейера

4. subtask_positions → INSERT (subtask_id, project_position_id=шаг3)
                     → связь позиция↔спецификация

5. event_log      → INSERT (event_type='position_created',
                     entity_type='position', entity_id=шаг3,
                     to_status='created', detail=название+кол-во)

Возврат: PositionDetail (позиция + имя номенклатуры)
  (доп. SELECT element_positions для имени, или из шага 1)

Изменить количество / привязку к КП
