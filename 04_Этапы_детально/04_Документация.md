# ЭТАП 4 — ДОКУМЕНТАЦИЯ — полная детализация

Формат: интерфейс (ASCII) → действия → API (ссылки на МОДЕЛИ_API.md) → таблицы.

═══════════════════════════════════════════════════
1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Сотрудник навешивает документы (сертификаты/паспорта) на позиции,
формирует финальный сводный документ для заказчика.

Роль:  Сотрудник документации
Вход:  позиции "установлены" (installed)
Выход: позиции "задокументированы" (documented) +
       сводный документ
```

2. ИНТЕРФЕЙС (макеты)
═══════════════════════════════════════════════════

## 2.1 Список задач документации (раздел Документы)
```
┌──────────────────────────────────────────────────────────────┐
│ Документы (24 установлено, нужен документ)  [+ Создать задачу]│
├──────────────────────────────────────────────────────────────┤
│  АКТИВНЫЕ                                                      │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ Документы на воздуховоды П1         ⏳ в работе          │ │
│ │ 👤 Козлов · 8 позиций · 5 с документами                 │ │
│ └──────────────────────────────────────────────────────────┘ │
│  ЗАВЕРШЁННЫЕ ▸ (2)                                            │
├──────────────────────────────────────────────────────────────┤
│ [Сформировать сводный документ]  (когда все готовы)          │
└──────────────────────────────────────────────────────────────┘

Счётчик установленных, которым нужен документ.
```

## 2.2 Деталь задачи документации
```
┌──────────────────────────────────────────────────────────────┐
│ ← Документы на воздуховоды П1        ⏳ в работе   [⋯]       │
├──────────────────────────────────────────────────────────────┤
│ ПОЗИЦИИ (8)                              [+ Выбрать]          │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ Воздуховод 500х300 ×14   [Сертификат оц.сталь №12 ▾] ✓  │ │
│ │ Отвод 500х300 ×8         [Сертификат оц.сталь №12 ▾] ✓  │ │
│ │ Вентилятор ×1            [Паспорт вентилятора №5 ▾]  ✓   │ │
│ │ Клапан ×2                [выбрать документ ▾]        ○   │ │
│ │ Хомут ×20                [выбрать документ ▾]        ○   │ │
│ └──────────────────────────────────────────────────────────┘ │
│ Нет нужного? [+ Добавить документ в справочник]              │
├──────────────────────────────────────────────────────────────┤
│ Статус: [В работу ▾]              [Завершить задачу]         │
└──────────────────────────────────────────────────────────────┘

У каждой позиции — выпадающий выбор документа.
Один документ можно на несколько позиций.
✓ есть документ / ○ нужен.
Завершение → позиции documented.
```

## 2.3 Выпадающий выбор документа
```
┌─ Выбрать документ ────────────────────┐
│ Поиск: [оцинк____] 🔍                  │
│ ─────────────────────────             │
│ СЕРТИФИКАТЫ (действующие):             │
│   Сертификат оц.сталь №12  до 12.2026 │
│   Сертификат оц.сталь №8   до 06.2026 │
│ ПАСПОРТА:                              │
│   Паспорт вентилятора №5              │
│ ─────────────────────────             │
│ (истёкшие НЕ показываются)             │
│ Нет нужного? [+ Добавить в справочник] │
└──────────────────────────────────────────┘

Только ДЕЙСТВУЮЩИЕ (expiry_date >= сегодня).
Истёкшие не в списке (но хранятся в справочнике).
```

## 2.4 Модалка "Добавить документ в справочник"
```
┌─ Добавить документ ────────────────────┐
│ Тип:    ○ Сертификат  ○ Паспорт        │
│ Название: [________________]           │
│ Номер:    [________________]           │
│ Срок годности: [__.__.____]            │
│ Кем выдан: [________________]          │
│ Файл: [+ Загрузить] 📄 sertifikat.pdf  │
│         [Отмена] [Сохранить]           │
└──────────────────────────────────────────┘

Документы также приходят с ДОСТАВКОЙ (снабжение загрузило).
Здесь можно добавить недостающие.
```

## 2.5 Финальный сводный документ
```
┌─ Сводный документ проекта ─────────────────┐
│ Группировка по документам:                 │
│                                             │
│ 📄 Сертификат оц.сталь №12                  │
│    Воздуховод 500х300 ×14                   │
│    Отвод 500х300 ×8                          │
│ 📄 Паспорт вентилятора №5                   │
│    Вентилятор ×1                             │
│ 📄 Сертификат крепежа №3                    │
│    Хомут ×20, Болт ×50                       │
│ ─────────────────────────                   │
│         [Экспорт PDF] [Готово к сдаче]      │
└──────────────────────────────────────────────┘

Один документ → несколько позиций (группировка).
Экспорт пакета для заказчика.
```

═══════════════════════════════════════════════════
3. ДЕЙСТВИЯ СОТРУДНИКА
═══════════════════════════════════════════════════
```
① Создать задачу документации
② Взять в работу
③ Выбрать позиции (единое окно, из установленных)
④ Навесить документ на позицию (выпадающий)
⑤ Добавить документ в справочник (если нет)
⑥ Завершить задачу (→ позиции documented)
⑦ Сформировать сводный документ (группировка)
⑧ Экспорт PDF для заказчика
```

═══════════════════════════════════════════════════
4. ПРАВИЛА
═══════════════════════════════════════════════════
```
- Позиции выбираются из установленных (installed) и свободных
- Каждая позиция ДОЛЖНА иметь документ (сертификат ИЛИ паспорт)
- Документ на ПОЗИЦИЮ ПРОЕКТА (болты разных фирм — разные сертификаты)
- Один документ можно на несколько позиций
- В выпадающем — только ДЕЙСТВУЮЩИЕ (срок не истёк)
- Истёкшие хранятся, но не в выборе
- Завершение задачи → позиции documented
- Финальный сводный: группировка по документу → позиции
- Документы приходят с доставкой (снабжение) + добавляются здесь
```

═══════════════════════════════════════════════════
5. ВНУТРЕННИЕ СТАТУСЫ (stage_status)
═══════════════════════════════════════════════════
```
new         — новая (позиции выбраны)
in_progress — навешивание документов
done        — все позиции с документами (documented)
```

6. API (детально)
═══════════════════════════════════════════════════

## 6.1 Список задач
```
GET /api/projects/{projectId}/subtasks?stage=docs
  Ответ: 200 { items: SubtaskListItem[], installed_count: int }
    (installed_count — установленных, нужен документ)

POST /api/projects/{projectId}/subtasks
  Тело: { name: string, stage: "docs" }
  Ответ: 201 Subtask
```

## 6.2 Выбор позиций
```
GET /api/projects/{projectId}/positions/available
  Параметры: stage=docs, group_by=(spec|kp)
  Ответ: 200 { groups: [...] }
    selectable = (status=installed AND current_subtask_id=null)

POST /api/subtasks/{id}/positions
  Тело: { items: [ { position_id: int } ] }
  Ответ: 200 { items: PositionDetail[] }
```

## 6.3 Документы справочника
```
GET /api/documents?active=true&type={cert|passport}&q={search}
  (для выпадающего — только действующие)
  Ответ: 200 { items: Document[] }
    фильтр: expiry_date >= today (истёкшие исключены)

POST /api/documents
  (добавить документ в справочник)
  Тело: multipart +
    { type, name, number, expiry_date, issuer }
  Ответ: 201 Document
```

## 6.4 Привязка документа к позиции
```
POST /api/positions/{positionId}/document
  (навесить документ)
  Тело: { document_id: int }
  Ответ: 200 PositionDocument
  Побочно: position_documents

DELETE /api/positions/{positionId}/document/{docId}
  (снять документ)
  Ответ: 204

POST /api/subtasks/{id}/document-bulk
  (навесить один документ на несколько позиций)
  Тело: { document_id: int, position_ids: int[] }
  Ответ: 200 { items: PositionDocument[] }
```

## 6.5 Завершение
```
POST /api/subtasks/{id}/complete
  Ответ: 200 SubtaskDetail (status → done)
  Побочно: позиции → status=documented, position_history
  Ошибки: 409 ApiError (есть позиции без документа)
    — все позиции задачи должны иметь документ
```

## 6.6 Сводный документ
```
GET /api/projects/{projectId}/documents-summary
  (сводка: группировка по документу → позиции)
  Ответ: 200
    { groups: [ { document: Document,
                  positions: PositionDetail[] } ] }

POST /api/projects/{projectId}/documents-summary/export
  (экспорт PDF пакета)
  Ответ: 200 { file_path: string }
```

6Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════
Каждый API — последовательность по ОДНОЙ таблице. [SQL] — где сложный запрос.

───────────────────────────────────────────────────
GET /api/projects/{projectId}/subtasks?stage=docs
Список задач документации
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где project_id={id} AND stage_id=docs
2. users          → SELECT ответственных
3. subtask_positions → SELECT COUNT где subtask_id IN (шаг1) GROUP BY
                    → позиций на задачу
4. position_documents → SELECT COUNT где position_id IN (позиции задач)
                    → сколько с документами (для "5 из 8")
5. project_positions → SELECT COUNT где status='installed'
                      AND current_subtask_id IS NULL AND project_id={id}
                    → installed_count (нужен документ — заголовок)

СБОРКА: SubtaskListItem[] + с_документами/всего + installed_count

[SQL] Шаги 3+4 — можно объединить с подсчётом документов.
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/subtasks
Создать задачу документации
───────────────────────────────────────────────────
```
1. projects       → SELECT  → проверка
2. subtasks       → INSERT (stage_id=docs, status='new')
3. event_log      → INSERT 'subtask_created'
Возврат: Subtask
```

───────────────────────────────────────────────────
GET /api/projects/{projectId}/positions/available
Окно выбора (доступные = установленные)
───────────────────────────────────────────────────
```
Параметры: stage=docs, group_by=(spec|kp)

1. project_positions → SELECT где project_id={id}
                    → selectable: status='installed' AND свободна
2. element_positions → названия
3. Группировка (spec|kp)
4. раздробленные → цвет

СБОРКА: { groups: [...] } (доступны installed)
[SQL] Тот же join-компонент, параметр stage=docs.
```

───────────────────────────────────────────────────
POST /api/subtasks/{id}/positions
Добавить позиции
───────────────────────────────────────────────────
```
Для каждой:
  1. project_positions → SELECT → проверка installed + свободна
  2. project_positions → UPDATE current_subtask_id={id} где id={pos}
                       (status остаётся installed до навешивания докум.)
  3. subtask_positions → INSERT
  4. event_log       → INSERT 'position_added_task'
Возврат: { items: PositionDetail[] }
```

───────────────────────────────────────────────────
GET /api/subtasks/{id}
Деталь задачи документации
───────────────────────────────────────────────────
```
1. subtasks       → SELECT
2. subtask_positions → SELECT позиции
3. project_positions → SELECT данные
4. element_positions → названия
5. position_documents → SELECT где position_id IN (позиции)
                    → какие документы уже навешены
6. documents      → SELECT где id IN (document_id из шаг5)
                    → данные документов (для показа "Сертификат №12")

СБОРКА: SubtaskDetail + позиции + их документы (✓/○)

[SQL] Шаги 3+4+5+6 — join позиции с документами и именами.
```

───────────────────────────────────────────────────
GET /api/documents (для выпадающего)
Действующие документы справочника
───────────────────────────────────────────────────
```
Параметры: active=true, type?, q?

1. documents      → SELECT где expiry_date >= today()
                    (истёкшие исключены — только действующие)
                    AND type={type} (если фильтр)
                    AND name LIKE %{q}% (если поиск)
                    ORDER BY type, name

Возврат: { items: Document[] }

[SQL] Индекс на expiry_date для быстрой фильтрации действующих.
```

───────────────────────────────────────────────────
POST /api/documents
Добавить документ в справочник
───────────────────────────────────────────────────
```
1. (сохранение файла документа)
2. documents      → INSERT (type, name, number, file_path,
                    expiry_date, issuer, is_active=true)
3. event_log      → INSERT 'document_added' (detail: тип, название, срок)
Возврат: Document
```

───────────────────────────────────────────────────
POST /api/positions/{positionId}/document
Навесить документ на позицию
───────────────────────────────────────────────────
```
1. documents      → SELECT где id={document_id}
                    → проверка: действующий (expiry_date >= today)?
                    → если истёк → 400 (нельзя навесить истёкший)

2. position_documents → SELECT где position_id={positionId}
                    → есть ли уже документ (заменить или добавить?)

3. position_documents → INSERT (position_id, document_id)
                    (ИЛИ UPDATE если замена)

4. event_log      → INSERT 'document_attached' (detail: документ)

Возврат: PositionDocument
```

───────────────────────────────────────────────────
POST /api/subtasks/{id}/document-bulk
Навесить один документ на несколько позиций
───────────────────────────────────────────────────
```
Тело: { document_id, position_ids: [] }

1. documents      → SELECT где id={document_id}  → проверка действующий

2. position_documents → INSERT batch (по каждой позиции:
                    position_id, document_id)

3. event_log      → INSERT 'document_attached' (по позиции)

Возврат: { items: PositionDocument[] }

[SQL] Batch INSERT (один документ → много позиций сразу).
```

───────────────────────────────────────────────────
DELETE /api/positions/{positionId}/document/{docId}
Снять документ
───────────────────────────────────────────────────
```
1. position_documents → DELETE где position_id, document_id
2. event_log      → INSERT 'document_detached'
Возврат: 204
```

───────────────────────────────────────────────────
POST /api/subtasks/{id}/complete
Завершить задачу документации
───────────────────────────────────────────────────
```
1. subtask_positions → SELECT где subtask_id={id}  → позиции

2. position_documents → SELECT где position_id IN (позиции)
                    → у каждой ли есть документ?
                    → ПРОВЕРКА: все позиции имеют документ?
                    → если есть без документа → 409 (нельзя завершить)

3. project_positions → UPDATE status='documented',
                    current_subtask_id=NULL где id IN (позиции)

4. subtasks       → UPDATE status='done'

5. event_log      → INSERT 'subtask_completed' + 'position_documented'

Возврат: SubtaskDetail

[SQL] Шаг2 — проверка полноты: LEFT JOIN позиции с position_documents,
  найти позиции БЕЗ документа. Если есть → блок.
```

───────────────────────────────────────────────────
GET /api/projects/{projectId}/documents-summary
Сводный документ (группировка по документу)
───────────────────────────────────────────────────
```
1. project_positions → SELECT где project_id={id}
                      AND status='documented'  → позиции проекта

2. position_documents → SELECT где position_id IN (шаг1)
                    → связи позиция↔документ

3. documents      → SELECT где id IN (document_id из шаг2)
                    → данные документов

4. element_positions → SELECT → названия позиций

ГРУППИРОВКА (в коде): по document_id →
  документ → [позиции с этим документом]

СБОРКА: { groups: [ {document, positions[]} ] }

[SQL] ЖЕЛАТЕЛЬНО одним запросом:
  documents JOIN position_documents JOIN project_positions
  JOIN element_positions
  ORDER BY document_id
  → сразу сгруппированная структура. Сводный строится часто.
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/documents-summary/export
Экспорт PDF пакета
───────────────────────────────────────────────────
```
1. (получить данные сводного — как GET выше)
2. (генерация PDF на C# — PdfSharp/iText, группировка по документам)
3. event_log      → INSERT 'summary_generated'
Возврат: { file_path }
```
