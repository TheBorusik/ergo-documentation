# ЭТАП 2Б — ЗАКУПКИ (оплата) — полная детализация

Этап закупок между Снабжением и Доставкой.
Снабженец создаёт задачу (subtask, stage=закупки) → бухгалтер
оплачивает (создаёт order) → снабжение продолжается.

Формат: интерфейс → действия → API → серверная логика → таблицы.

═══════════════════════════════════════════════════
1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Бухгалтер обрабатывает заявки на оплату от снабжения:
создаёт order (финансовую запись), оплачивает, прикрепляет счёт.
ИЛИ отменяет заявку (нашёл проблему).

Роль:  Бухгалтер
Вход:  задача закупки от снабжения (позиции "готов к оплате")
Выход: order оплачен, позиции "purchased" → снабжение продолжается
       ИЛИ отмена → снабжение получает отказ

ВАЖНО (памятка): задача = subtask (stage=закупки), НЕ новая сущность.
  order — отдельная финансовая таблица (создаёт бухгалтер).
```

═══════════════════════════════════════════════════
2. ИНТЕРФЕЙС (макеты)
═══════════════════════════════════════════════════

## 2.1 Список задач оплаты (раздел Закупки / Мои задачи бухгалтера)
```
┌──────────────────────────────────────────────────────────────┐
│ Оплата — задачи                                               │
├──────────────────────────────────────────────────────────────┤
│  К ОПЛАТЕ:                                                     │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ Воздуховоды П1 → ПостА              ⏳ новая             │ │
│ │ Завод сыра · 17400₽ · согласовано Иванов · 3 позиции    │ │
│ ├──────────────────────────────────────────────────────────┤ │
│ │ Клапаны П1 → ПостБ                  ⏳ новая             │ │
│ │ Завод сыра · 8200₽ · 2 позиции                           │ │
│ └──────────────────────────────────────────────────────────┘ │
│  ОПЛАЧЕННЫЕ ▸ (5)   ОТМЕНЁННЫЕ ▸ (1)                          │
└──────────────────────────────────────────────────────────────┘

Задачи закупки (subtask stage=закупки), ответственный бухгалтер.
Карточка: название, проект, сумма, поставщик, позиции.
```

## 2.2 Деталь задачи оплаты (создание order)
```
┌──────────────────────────────────────────────────────────────┐
│ ← Оплата: Воздуховоды П1 → Поставщик А       ⏳ новая  [⋯]   │
├──────────────────────────────────────────────────────────────┤
│ ЗАЯВКА ОТ СНАБЖЕНИЯ:                                          │
│ Проект: Завод сыра · Поставщик: ПостА                        │
│ Позиции:                                                      │
│   Воздуховод ×10    1200₽    12000₽                          │
│   Отвод ×8           450₽     3600₽                          │
│   Итого: 17400₽                                              │
│ Прайс поставщика: 📄 prays_A.pdf                              │
├──────────────────────────────────────────────────────────────┤
│ ДЕЙСТВИЯ БУХГАЛТЕРА:                                          │
│  [Создать order и оплатить]   [Отменить заявку]              │
└──────────────────────────────────────────────────────────────┘

Бухгалтер видит что прислал снабженец, решает: оплатить/отменить.
```

## 2.3 Создание order + оплата (модалка/экран)
```
┌─ Создать order и оплатить ─────────────────┐
│ Order №: [ORD-101] (автонумерация)          │
│ Поставщик: Поставщик А                      │
│ Сумма: 17400₽                               │
│ ─────────────────────                       │
│ Счёт: [+ Прикрепить] 📄 schet_A.pdf          │
│ Дата оплаты: [15.01.2026]                   │
│ ─────────────────────                       │
│         [Отмена] [Провести оплату]          │
└───────────────────────────────────────────────┘

При проведении:
  order создаётся (orders таблица),
  позиции → order_id + status=purchased,
  задача → done, снабжение продолжается.
```

## 2.4 Отмена заявки
```
┌─ Отменить заявку ──────────────────────────┐
│ Причина: [________________________]        │
│  (ошибка в прайсе / нет средств / ...)     │
│         [Отмена] [Отменить заявку]         │
└───────────────────────────────────────────────┘

При отмене:
  задача → cancelled (+ причина),
  позиции разблокируются (назад в снабжение),
  снабженец видит отказ + причину.
```

═══════════════════════════════════════════════════
3. ДЕЙСТВИЯ СОТРУДНИКА
═══════════════════════════════════════════════════
```
БУХГАЛТЕР:
  ① Открыть задачу оплаты (видит заявку)
  ② Создать order + оплатить (счёт, дата)
     → order создан, позиции purchased, задача done
  ③ ИЛИ отменить заявку (причина)
     → задача cancelled, позиции разблокированы

Снабженец НЕ действует здесь (только создал задачу в снабжении).
```

═══════════════════════════════════════════════════
4. ПРАВИЛА
═══════════════════════════════════════════════════
```
- Задача закупки = subtask (stage=закупки), единый механизм
- Позиции задачи — через subtask_positions (не по статусу!)
- Позиции заблокированы (status=ready_for_payment) пока оплата
- ORDER создаёт ТОЛЬКО бухгалтер (снабженец не трогает)
- order — отдельная финансовая таблица (для раздела Закупки)
- Бухгалтер решает: оплатить (order) ИЛИ отменить (без order)
- Оплата → позиции purchased, снабжение продолжается (авто)
- Отмена → позиции назад, снабжение получает отказ + причину
- Связь: subtask(закупки) ↔ subtask(снабжения) через parent_id/поле
```

═══════════════════════════════════════════════════
5. ВНУТРЕННИЕ СТАТУСЫ (stage_status)
═══════════════════════════════════════════════════
```
new         — заявка создана снабженцем (ждёт бухгалтера)
in_progress — бухгалтер взял (оформляет order)
done        — оплачено (order создан, позиции purchased)
cancelled   — отменено (причина, позиции назад)

Общий Subtask.status совпадает.
```

6. API (детально)
═══════════════════════════════════════════════════

## 6.1 Список задач оплаты
```
GET /api/projects/{projectId}/subtasks?stage=purchase
  ИЛИ глобально (бухгалтер поверх проектов):
GET /api/purchases/tasks?status=new
  Ответ: 200 { items: SubtaskListItem[] }
    (задачи закупки, ответственный=бухгалтер)
```

## 6.2 Деталь задачи (заявка)
```
GET /api/subtasks/{id}
  Ответ: 200 SubtaskDetail
    (позиции из subtask_positions, поставщик, прайс,
     ссылка на задачу снабжения)
```

## 6.3 Создать order + оплатить
```
POST /api/subtasks/{id}/pay
  (бухгалтер создаёт order и оплачивает)
  Тело:
    { invoice_file: string, paid_at: date }
  Ответ: 200 { order: Order, subtask: Subtask }
  Побочно:
    orders INSERT (number, supplier_id, total_amount, invoice_file,
                   paid_by, paid_at),
    позиции → order_id + status=purchased,
    subtask → done,
    задача снабжения → продолжается (stage_status),
    event_log (order_created, payment_paid, position_purchased)
```

## 6.4 Отменить заявку
```
POST /api/subtasks/{id}/cancel
  Тело: { reason: string }
  Ответ: 200 Subtask
  Побочно:
    subtask → cancelled (+ cancel_reason),
    позиции → status назад (in_purchase), разблокированы,
    задача снабжения → отказ (снабженец видит причину),
    event_log (subtask_cancelled)
```

═══════════════════════════════════════════════════
6Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════

───────────────────────────────────────────────────
GET /api/purchases/tasks — список задач оплаты
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где stage_id=purchase
                    AND status IN (new, in_progress)
                    (глобально или по проекту)

2. subtask_positions → SELECT COUNT где subtask_id IN (шаг1)
                    → кол-во позиций на задачу

3. project_positions → SELECT где id IN (позиции)
                    → суммы (для отображения)

4. contractors    → SELECT поставщиков

СБОРКА: SubtaskListItem[] (название, проект, сумма, поставщик)

[SQL] Шаг2+3 — join для сумм по задаче.
```

───────────────────────────────────────────────────
GET /api/subtasks/{id} — деталь заявки
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где id={id}  → задача + связь снабжения

2. subtask_positions → SELECT где subtask_id={id}
                    → СВОИ позиции (не по статусу — по связи!)

3. project_positions → SELECT где id IN (шаг2)  → данные, цены

4. element_positions → SELECT → названия

5. contractors    → SELECT поставщика

6. file_attachments → SELECT прайс (entity=subtask снабжения)

СБОРКА: SubtaskDetail (позиции + суммы + поставщик + прайс)

[SQL] Шаги 3+4 — join позиции с именами.
```

───────────────────────────────────────────────────
POST /api/subtasks/{id}/pay — создать order + оплатить ★
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где id={id}  → проверка (new/in_progress)

2. subtask_positions → SELECT где subtask_id={id}  → позиции задачи

3. project_positions → SELECT где id IN (шаг2)  → суммы
                    → total_amount = СУММА

4. orders         → INSERT (number=автоген, supplier_id,
                    total_amount, invoice_file, paid_by=бухгалтер,
                    paid_at)
                    → создана финансовая запись (order)

5. project_positions → UPDATE order_id={order}, status='purchased'
                    где id IN (позиции)
                    → позиции привязаны к order + куплены

6. subtasks       → UPDATE status='done', stage_status='done'
                    где id={id}
                    → задача закупки завершена

7. subtasks       → UPDATE stage_status (продолжить) где
                    id={задача снабжения из связи}
                    → снабжение получает "оплачено" → к доставке

8. event_log      → INSERT order_created + payment_paid +
                    position_purchased (позиции) + subtask_completed

Возврат: { order, subtask }

[Транзакция обязательна — order + позиции + две задачи]
[SQL] Шаг3 — SUM по позициям. Шаг5 — UPDATE ... WHERE id IN.
Автонумерация order — последовательность/счётчик.
```

───────────────────────────────────────────────────
POST /api/subtasks/{id}/cancel — отменить заявку
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где id={id}  → проверка

2. subtask_positions → SELECT где subtask_id={id}  → позиции

3. project_positions → UPDATE status='in_purchase' (назад),
                    current_subtask_id=(задача снабжения?)
                    где id IN (позиции)
                    → разблокировать, вернуть в снабжение

4. subtasks       → UPDATE status='cancelled',
                    cancel_reason={reason} где id={id}

5. subtasks       → UPDATE stage_status (отказ) где
                    id={задача снабжения}
                    → снабженец видит отказ + причину

6. event_log      → INSERT subtask_cancelled (+ reason)

Возврат: Subtask

[Транзакция — позиции + две задачи]
```

7. ТАБЛИЦЫ
═══════════════════════════════════════════════════
```
ЧИТАЕТ:
  subtasks (задачи закупки)
  subtask_positions (СВОИ позиции — связь, не статус!)
  project_positions (данные, суммы)
  element_positions (названия)
  contractors (поставщики)
  file_attachments (прайс)

СОЗДАЁТ/МЕНЯЕТ:
  orders                (НОВАЯ — финансовая запись, создаёт бухгалтер)
  project_positions     (order_id, status: ready_for_payment→purchased
                         ИЛИ назад при отмене)
  subtasks              (задача закупки: done|cancelled;
                         + задача снабжения: продолжение|отказ)
  event_log             (история)

НОВАЯ ТАБЛИЦА:
  orders (id, number, supplier_id, total_amount, invoice_file,
          paid_by, paid_at, created_at)
  → для раздела "Закупки" (все покупки по проектам)
```

═══════════════════════════════════════════════════
8. СТАТУСЫ И ПЕРЕХОДЫ
═══════════════════════════════════════════════════
```
Задача закупки (Subtask.status): new → in_progress → done|cancelled
Позиция: ready_for_payment → purchased (оплата)
                          → in_purchase (отмена, назад)
Order: создаётся оплаченным (paid) сразу
Связь: задача снабжения продолжается (done) | отказ (cancelled)
```

9. ОТКРЫТЫЕ ВОПРОСЫ
═══════════════════════════════════════════════════
```
- Согласование менеджера — до задачи оплаты? (цена одобрена)
- Один order = одна задача, или бухгалтер объединяет заявки?
- Отмена → снабженец переделывает или задача снабжения откат?
- Доставка — на order или в задаче снабжения продолжается?
- Order межпроектный (объединение заявок)?
- Автонумерация order — формат (ORD-101)?
```
