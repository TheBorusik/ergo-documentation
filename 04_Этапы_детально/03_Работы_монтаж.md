1. НАЗНАЧЕНИЕ
═══════════════════════════════════════════════════
```
Прораб создаёт задачи на работы, выбирает позиции которые
монтируются вместе, выполняет, закрывает. Оформляет акты.

Роль:  Прораб (+ рабочие выполняют)
Вход:  позиции "доставлены" (delivered)
Выход: позиции "установлены" (installed)
```

## 2.3 Окно выбора позиций (единое)
```
То же окно что в снабжении (position_selection_window.md),
но доступны позиции "доставлены" (delivered) и свободные.

┌─ Выбор позиций для задачи работ ───────────────────┐
│ Вид: [По спецификации] [По КП]      Поиск: 🔍      │
├──────────────────────────────────┬─────────────────┤
│ ДОСТУПНЫЕ (доставленные)          │ ВЫБРАНО         │
│ ▾ Помещение 1                     │ Воздуховод ×14  │
│   Воздуховод ×14  ✅ доставлен [→]│ Отвод ×8        │
│   Отвод ×8        ✅ доставлен [→]│ Хомут ×20       │
│   Хомут ×20       ✅ доставлен [→]│                 │
│   Вентилятор ×1   🛒 в закупке    │                 │
│     (серый, ещё не доставлен)     │                 │
├──────────────────────────────────┴─────────────────┤
│              [Отмена] [В задачу]                    │
└──────────────────────────────────────────────────────┘

Человеческий фактор: видно недоставленные (серые) →
прораб решает подождать крепёж или монтировать без.
```

3. ДЕЙСТВИЯ СОТРУДНИКА
═══════════════════════════════════════════════════
```
ПРОРАБ:
  ① Создать задачу на работу
  ② Выбрать вид работ (из справочника)
  ③ Взять в работу
  ④ Выбрать позиции (единое окно, из доставленных)
  ⑤ Исключить позицию из работы
  ⑥ Прикрепить файлы/фото работ
  ⑦ Завершить работу (→ позиции installed)
  ⑧ Создать акт (выбрать выполненные работы)
```

═══════════════════════════════════════════════════
4. ПРАВИЛА
═══════════════════════════════════════════════════
```
- Позиции выбираются из доставленных (delivered) и свободных
- Работа = НАБОР позиций (потребляет воздуховоды+крепёж+болты вместе)
- Выбранные заняты (эксклюзивность, current_subtask_id)
- Завершение работы → ВСЕ её позиции → installed (авто)
- Акт — ОТДЕЛЬНЫЙ механизм (не трогает статусы позиций)
- Акт: одна/несколько завершённых работ
- Объём работы — под вопросом (пока не используем)
- Вид работ из справочника works
```

5. ВНУТРЕННИЕ СТАТУСЫ (stage_status)
═══════════════════════════════════════════════════
```
new         — новая (позиции выбраны)
in_progress — в работе (монтаж идёт)
done        — выполнена (позиции installed)

Проще снабжения (3 статуса vs 14).
Общий статус (Subtask.status) совпадает по смыслу.
```

6. API (детально)
═══════════════════════════════════════════════════

## 6.1 Список задач + справочник работ
```
GET /api/projects/{projectId}/subtasks?stage=works
  Ответ: 200 { items: SubtaskListItem[], delivered_count: int }
    (delivered_count — доставленных позиций, готовых к монтажу)

GET /api/works
  (справочник видов работ для выбора)
  Ответ: 200 { items: Work[] }

POST /api/projects/{projectId}/subtasks
  Тело: { name: string, stage: "works", work_id?: int }
  Ответ: 201 Subtask
```

## 6.2 Деталь и вид работ
```
GET /api/subtasks/{id}
  Ответ: 200 SubtaskDetail  (позиции, файлы, вид работ)

PATCH /api/subtasks/{id}/work
  (выбрать/сменить вид работ)
  Тело: { work_id: int }
  Ответ: 200 Subtask
  Побочно: subtask_works
```

## 6.3 Выбор позиций (единое окно)
```
GET /api/projects/{projectId}/positions/available
  Параметры: stage=works, group_by=(spec|kp)
  Ответ: 200 { groups: [...] }
    selectable = (status=delivered AND current_subtask_id=null)

POST /api/subtasks/{id}/positions
  Тело: { items: [ { position_id: int } ] }
    (для работ дробление обычно не нужно — берём как есть)
  Ответ: 200 { items: PositionDetail[] }
  Побочно: current_subtask_id = задача, subtask_positions

DELETE /api/subtasks/{id}/positions/{positionId}
  Ответ: 204 (освобождение позиции)
```

## 6.4 Выполнение и завершение
```
PATCH /api/subtasks/{id}/take
  Ответ: 200 Subtask (status → in_progress)

POST /api/subtasks/{id}/complete
  Ответ: 200 SubtaskDetail (status → done)
  Побочно: ВСЕ позиции задачи → status=installed,
           current_subtask_id=null (освобождены),
           position_history (installed)
```

## 6.5 Файлы
```
POST /api/subtasks/{id}/files
  Тело: multipart (фото/документы работ)
  Ответ: 201 FileAttachment
```

## 6.6 Акты
```
GET /api/projects/{projectId}/work-acts
  Ответ: 200 { items: WorkAct[] }

POST /api/projects/{projectId}/work-acts
  (создать акт из выполненных работ)
  Тело:
    { number: string, date: date, subtask_ids: int[] }
    (subtask_ids — завершённые работы в акт)
  Ответ: 201 WorkAct
  Побочно: work_acts, work_act_items
    (НЕ трогает статусы позиций — отдельный механизм)

PATCH /api/work-acts/{id}
  Тело: { status: "signed" }
  Ответ: 200 WorkAct
```

6Б. СЕРВЕРНАЯ ЛОГИКА (очередь операций по таблицам)
═══════════════════════════════════════════════════
Каждый API — последовательность операций по ОДНОЙ таблице.
[SQL] — где сложный запрос оправдан. Паттерн: проверки → операция → event_log → сборка.

───────────────────────────────────────────────────
GET /api/projects/{projectId}/subtasks?stage=works
Список задач работ
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где project_id={id} AND stage_id=works
                    → задачи работ

2. subtask_works  → SELECT где subtask_id IN (шаг1)
                    → вид работ каждой задачи (work_id)

3. works          → SELECT где id IN (work_id из шаг2)
                    → названия видов работ

4. users          → SELECT где id IN (responsible_id)  → ответственные

5. subtask_positions → SELECT COUNT где subtask_id IN (шаг1) GROUP BY
                    → кол-во позиций на задачу

6. project_positions → SELECT COUNT где status='delivered'
                      AND current_subtask_id IS NULL AND project_id={id}
                    → delivered_count (готовых к монтажу — заголовок)

СБОРКА: SubtaskListItem[] + вид работ + delivered_count

[SQL] Шаг5 — GROUP BY. Шаги 2+3 — join subtask_works+works.
```

───────────────────────────────────────────────────
GET /api/works
Справочник видов работ
───────────────────────────────────────────────────
```
1. works          → SELECT где is_active=true
                    → список видов работ

Возврат: { items: Work[] }
```

───────────────────────────────────────────────────
POST /api/projects/{projectId}/subtasks
Создать задачу работ
───────────────────────────────────────────────────
```
1. projects       → SELECT где id={id}  → проверка

2. works          → SELECT где id={work_id} (если указан)  → проверка

3. subtasks       → INSERT (project_id, stage_id=works, name,
                    status='new', created_at)

4. subtask_works  → INSERT (subtask_id, work_id) (если work_id указан)

5. event_log      → INSERT 'subtask_created'

Возврат: Subtask
```

───────────────────────────────────────────────────
GET /api/subtasks/{id}
Деталь задачи работ
───────────────────────────────────────────────────
```
1. subtasks       → SELECT где id={id}

2. subtask_works  → SELECT где subtask_id={id}  → вид работ
3. works          → SELECT где id={work_id}     → название вида

4. subtask_positions → SELECT где subtask_id={id}  → id позиций
5. project_positions → SELECT где id IN (шаг4)      → данные
6. element_positions → SELECT → названия

7. file_attachments → SELECT где entity_type='subtask' AND entity_id={id}
                    → фото/файлы работ

8. users          → SELECT ответственного

СБОРКА: SubtaskDetail + вид работ + позиции + файлы

[SQL] Шаги 5+6 — join для позиций с именами.
```

───────────────────────────────────────────────────
PATCH /api/subtasks/{id}/work
Выбрать/сменить вид работ
───────────────────────────────────────────────────
```
1. works          → SELECT где id={work_id}  → проверка

2. subtask_works  → SELECT где subtask_id={id}
                    → есть ли уже запись

3. subtask_works  → UPDATE work_id (если есть) ИЛИ INSERT (если нет)

4. event_log      → INSERT 'work_type_set' (detail: вид работ)

Возврат: Subtask
```

───────────────────────────────────────────────────
GET /api/projects/{projectId}/positions/available
Окно выбора (доступные = доставленные)
───────────────────────────────────────────────────
```
Параметры: stage=works, group_by=(spec|kp)

1. project_positions → SELECT где project_id={id}
                    → все позиции, selectable вычисляем:
                      status='delivered' AND current_subtask_id IS NULL

2. element_positions → SELECT → названия

3. Группировка (spec|kp):
   subtasks ИЛИ kp_items → названия групп

4. project_positions → SELECT раздробленные (parent_position_id)
                    → цветовая связка

СБОРКА: { groups: [...] } — как в снабжении, но доступны delivered

[SQL] Тот же join что в снабжении (project_positions+element_positions
  +группировка). Единый компонент, параметр stage меняет фильтр.
```

POST /api/subtasks/{id}/positions
Добавить позиции в работу
───────────────────────────────────────────────────
```
Тело: { items: [{ position_id }] }
(для работ дробление обычно не нужно — берём целиком)

Для каждой:
  1. project_positions → SELECT → проверка delivered + свободна
  2. project_positions → UPDATE current_subtask_id={id},

  3. subtask_positions → INSERT (subtask_id, project_position_id)
  4. event_log       → INSERT 'position_added_task'
```

8. СТАТУСЫ И ПЕРЕХОДЫ
═══════════════════════════════════════════════════
```
Задача (Subtask.status): new → in_progress → done
Задача (stage_status): new → in_progress → done (совпадает)

Позиция (Position.status):
  delivered → installed
  (installed при завершении работы, авто для всех позиций задачи)

Акт (WorkAct.status): draft → signed
  (независимо от конвейера позиций)
```

```

- Одна работа = одна подзадача, или несколько видов в одной?
  (сейчас: один work_id на подзадачу)
- Переход к документации — после завершения работы (позже уточняли)
```
