# Проверка моделей: нормализация + полнота для UI

Аудит перед началом UI: связи нормализованы? полей хватает?
что добавить для удобства интерфейса?

═══════════════════════════════════════════════════
1. НОРМАЛИЗАЦИЯ СВЯЗЕЙ — проверка
═══════════════════════════════════════════════════
```
✓ Project → Contractor (customer_id) — норм
✓ KpGroup → Project, KpItem → KpGroup — норм (иерархия КП)
✓ Position → Project, ElementPosition, KpItem, Subtask — норм
✓ Position самоссылки: parent_position_id, current_subtask_id,
  spec_subtask_id — норм (дробление, эксклюзивность)
✓ Position → Order (order_id) — ДОБАВЛЕНО (закупка)
✓ Subtask → Project, Stage, parent (дерево) — норм
✓ Subtask авторство: created_by, responsible_id, closed_by — норм
✓ Order → Contractor (supplier) — норм, позиции обратной связью
✓ event_log → полиморфная (entity_type+entity_id) — норм
✓ Contractor (customer|supplier|both) — единая таблица, норм
✓ subtask_positions (M2M задача↔позиции) — норм связь

ВЕРДИКТ: связи нормализованы (3НФ).
  Нет дублирования, все FK на месте, справочники вынесены.
  Денормализация только осознанная (created_by/closed_by в Subtask,
  *_name в *ListItem — для скорости UI, оправдано).
```

═══════════════════════════════════════════════════
2. ПОЛНОТА ДЛЯ UI — чего не хватает
═══════════════════════════════════════════════════

## 2.1 Что ДОБАВИТЬ для удобства UI (рекомендации)
```
Project:
  + code/number: string?    (номер проекта "П-2026-042" для UI)
  + deadline: date?         (срок сдачи — нужен дашборду/спискам!)
  + responsible_id: FK→User? (менеджер проекта — для "мои проекты")
  + description: text?       (описание)
  (сейчас нет срока и ответственного — а UI их показывает!)

Position:
  + note: string?           (примечание к позиции)
  ~ status покрывает конвейер — ок

Subtask:
  + deadline: date?         (срок задачи — для просрочек в дашборде)
  + description: text?      (описание задачи)
  + priority: enum?         (приоритет — опц.)

Contractor:
  + contact_person: string? (контактное лицо)
  + note: text?             (заметки)
  + is_active: bool         (архивные скрывать)

User:
  + avatar: string?         (для UI — аватарки в задачах)
  + phone: string?

Order:
  ~ полный — ок
```

## 2.2 КРИТИЧНО (UI без этого не соберёшь)
```
★ Project.deadline — срок сдачи
  дашборд показывает "просрочено", "ближайшие сроки",
  карточка проекта — срок. БЕЗ ЭТОГО никак.

★ Project.responsible_id — менеджер проекта
  "мои проекты", фильтр по ответственному, дашборд.

★ Subtask.deadline — срок задачи
  просрочки задач, триггеры проблем в дашборде.

Эти три — обязательны до UI.
```

3. МОДЕЛИ ДЛЯ СПИСКОВ (ListItem) — есть?
═══════════════════════════════════════════════════
```
UI-списки нужны облегчённые модели (не тянуть всё):

✓ SubtaskListItem — есть (name, status, счётчики, supplier_name)
✓ PositionDetail — есть (+ nomenclature_name/article)
✓ NomenclatureSearchResult — есть

НЕ хватает для UI:
  ✗ ProjectListItem — для списка проектов
    { id, name, code, customer_name, stage, progress,
      deadline, responsible_name, status }
    (сейчас Project без customer_name/progress/deadline —
     список проектов не собрать одним запросом)

  ✗ OrderListItem — для раздела Закупки
    { id, number, supplier_name, total_amount,
      payment_status, projects_names[], paid_at }

  ✗ ContractorListItem — для справочника
    { id, name, type, inn, is_active, projects_count? }
```

4. ВЫЧИСЛЯЕМЫЕ ПОЛЯ (для UI, не хранить)
═══════════════════════════════════════════════════
```
Считаются на бэке для отображения (не колонки БД):

Project:
  progress: int          (% готовности — из статусов позиций/этапов)
  customer_name: string  (JOIN Contractor)
  responsible_name: string
  current_stage: string  (текущий этап)

→ это НЕ поля БД, а обогащение в ответе API (ListItem/Detail)
```
