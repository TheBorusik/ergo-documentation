## ФУНДАМЕНТАЛЬНОЕ РЕШЕНИЕ — конвейер
```
Позиция = единый источник истины, ТЕЧЁТ по конвейеру.
Задачи НЕ двигаются — принадлежат этапу.
Этап: вход (позиции в статусе) → работа (подзадачи) → выход (новый статус).
Статус позиции меняется АВТОМАТИЧЕСКИ от статуса задачи.
Только вперёд (без откатов), терминальные: installed/utilized/to_warehouse.
```

## ПРАВИЛО ЭКСКЛЮЗИВНОСТИ
```
Позиция в ОДНОЙ активной подзадаче за раз (на всех этапах).
Занята → никуда больше, пока не завершится.
Требует дробления для разных судеб (часть у А, часть у Б → 2 позиции).
current_subtask_id: NULL=свободна, заполнен=занята.
```

## ДРОБЛЕНИЕ
```
Разная судьба (поставщик/склад/доставка) → отдельные позиции (строки).
Связь через общий kp_item_id + parent_position_id (опц).
subtask_positions без quantity (позиция целиком).
```

## Механизм подзадач (единый на всех этапах)
```
"Получил → раздrobил → передал дальше" через статус позиции.
subtasks — дерево (parent_id), stage_id фиксирован.
Два статуса: общий (new/in_progress/done/paused) + внутренний (свой на этап).
Добавление/исключение позиций (пул свободных).
```

## 1. Продажа
```
КП: ГРУППА (Оборудование, Холод, Материалы...) → ЗАПИСЬ (kp_item)
    → раскрывается в позиции.
Две цены: cost/sales (отложены). Проигранные тоже в системе (lost).
```

## 2. Проектирование
```
Проектировщик создаёт ПОДЗАДАЧИ-спецификации (по помещениям/системам).
Позиции привязывает к КП (kp_item_id, практически всегда, один).
Спецификация → снабжение целиком, там дробится.
Новые позиции (обход столба и т.п.) — source=added, к тому же КП.
```

## 3. Снабжение (14 статусов)
```
получена→распределена→отправлена→цены→маппинг→согласование(менеджер)→
согласована→оплата(бухгалтерия)→оплачена→сроки→готова→доставка→доставлена→готово
- Запрос одному (можно повторить другому), is_selected
- Отправка: шаблон+email
- НЕСКОЛЬКО файлов в ответе (supplier_quotes → import_files → lines)
- Частичная доставка — пока НЕТ
```

## Цены и маппинг (проверено на реальных данных!)
```
Маппинг 89% автоматом (8 из 9 на счёте Центавра).
Причина: поставщик копирует наш запрос → названия почти идентичны.
Алгоритм: нормализация → exact по артикулу → exact по названию →
          история → fuzzy (порог 0.6) → ручной.
Сохраняем:
  supplier_position_links — соответствие наша↔поставщика (название, артикул)
  supplier_prices — история цен (меняются 3 раза/год, is_current)
  supplier_mappings — обучение (варианты + счётчик)
Нюансы: единицы/фасовка (уп↔шт), доставка как строка.
```

## Парсинг прайсов (доказан на PDF ВЕЗА/Центавр)
```
Многоуровневый: автопарсинг → визуальная разметка колонок →
шаблон поставщика → AI (Claude, вероятно основной в будущем).
Двухфазно: каталог (без цен, соответствия) + прайс (цены).
Всё на C# сервере. Форматы: Excel/PDF/Word приоритет.
```

## 4. Монтаж и ПНР
```
Справочник works (Монтаж воздуховодов, оборудования, ПНР...).
Работа = НАБОР позиций (потребляет воздуховоды+крепёж+болты).
Сотрудник выбирает позиции вручную.
Работа завершена → её позиции "установлена" (авто).
Акты (work_acts) — ОТДЕЛЬНЫЙ механизм, статусы не трогают.
Объём работы — под вопросом.
```

## 5. Документация
```
Два вида: СЕРТИФИКАТЫ (простые позиции), ПАСПОРТА (оборудование).
documents (справочник): срок годности КЛЮЧЕВОЙ.
Привязка на ПОЗИЦИЮ ПРОЕКТА (болты разных фирм — разные сертификаты).
Обязательно для КАЖДОЙ позиции (сертификат или паспорт).
Истёкшие не в выпадающем списке.
Финальный сводный: группировка по документу → позиции.
Документы приезжают с доставкой (запрос отсутствующих — позже).
```

## 6. Сдача
```
Финальный документ → приёмка → финальная оплата → Завершён.
Минимум: projects.status=completed + completed_at.
Оплата — часть модели цен (позже).
```

## Единое окно выбора позиций (для всех этапов)
```
Показывает ВСЕ позиции, выбрать можно только доступные по статусу:
  Снабжение → готовые к закупке, Работы → доставленные
Недоступные серые (видны, но не выбрать).
ЧЕЛОВЕЧЕСКИЙ ФАКТОР: видимость помогает решать
  (снабженец видит негтовые → подождёт объединить доставку).
Можно объединять позиции из разных помещений.
```

## Разрешение дилеммы "партии vs гибкость" (ВАЖНОЕ)
```
Единица движения — ПОЗИЦИЯ, помещение — ЯРЛЫК на ней (spec_subtask_id).
Задачи НЕ едут (в этапах), позиции текут по конвейеру.
Связь с помещением на позиции (не теряется).
Снабжение видит пул готовых → группирует свободно.
Наглядность — срез по помещению в "Позициях".
→ гибкость + связь + наглядность одновременно.
```

## Статусы этапов (общий уровень, для наглядности)
```
Каждый ЭТАП имеет свой статус (не только задачи внутри):
  ⚪ ОЖИДАНИЕ  — ждёт предыдущий этап
  🔵 В РАБОТЕ  — активен
  🟢 ЗАВЕРШЁН  — готов (ответственный нажал "завершить")

Единая цветовая схема. Видно на ДОСКЕ (над колонками) и
в разделах (индикатор у этапа).

Начальные: Проектирование ждёт КП, Снабжение ждёт Проектирование...
Переходы:
  ① Ожидание→В работе — ВРУЧНУЮ ("Начать этап"), пока
  ② Завершён→Доработка — с ПРИЧИНОЙ текстом (обход столба и т.п.)
  ③ Завершить при незавершённом предыдущем — можно (независимы)

Статус этапа = наглядный ИНДИКАТОР, не жёсткая блокировка
(согласуется с опережающим потоком позиций).
project_stages: status (waiting|in_progress|completed) + даты + кто.
```

## Унифицированная история (event_log)
```
ВСЕ события системы → ОДИН журнал event_log
(заменяет position_history, расширяет до всех сущностей).

event_log: project_id, event_type, entity_type (stage|position|
  subtask|project|...), entity_id, from_status, to_status,
  reason (причина доработок), detail, changed_by, changed_at

Срезы (фильтром):
  история позиции = entity_type=position
  история этапа = entity_type=stage
  timeline проекта = все события по времени
  доработки = event_type=stage_reopened (причины, частота)
  действия юзера = changed_by

Единый метод LogEvent(...) → INSERT после каждого действия.
Каталог ~75 событий (03/10, 12 — ЖИВОЙ, финализация в конце).
detail: текст или JSONB (PostgreSQL) — решим позже.
```

```
notifications (БД) — ТОЛЬКО адресное человеку ("ВАМ назначили"):

  Колокольчик (05/00) читает ИЗ БД; push лишь будит счётчик.
```

═══════════════════════════════════════════════════
ЭТАП ЗАКУПКИ (2Б) + ORDER — финансовый контур
═══════════════════════════════════════════════════
```
Новый этап между Снабжением и Работами.

ПОТОК: Снабжение → ЗАКУПКИ → Работы
  Снабженец [Отправить на оплату] →
    создаётся задача (subtask, stage=закупки) для бухгалтера,
    позиции → "готов к оплате" (заблокированы)
  Бухгалтер (раздел Закупки / Мои задачи):
    создаёт ORDER (финансовая таблица) → оплачивает → счёт → закрыть
    ИЛИ отменяет заявку (причина) → позиции назад
  Автомат: позиции purchased → снабжение продолжается к доставке

КЛЮЧЕВОЕ (использованы готовые механизмы — см. памятку 04):
  задача = subtask (stage=закупки), НЕ отдельная сущность
  позиции задачи — через subtask_positions (не по статусу)
  ORDER — отдельная финансовая таблица, создаёт ТОЛЬКО бухгалтер
    (снабженец не трогает финансовые модели)
  order НЕ привязан к задаче: позиция знает order (position.order_id)
```
