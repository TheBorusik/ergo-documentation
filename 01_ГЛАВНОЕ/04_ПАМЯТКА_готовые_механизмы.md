# ПАМЯТКА — готовые механизмы (НЕ выдумывать заново!)

Ключевые архитектурные решения. Перед добавлением нового —
свериться здесь, использовать существующее.

═══════════════════════════════════════════════════
1. КОНВЕЙЕР — позиция течёт, задачи стоят
═══════════════════════════════════════════════════
```
Задачи НЕ мигрируют между этапами (stage_id фиксирован).
Позиция ТЕЧЁТ по конвейеру, меняя СТАТУС.
Результат задачи = смена статуса позиции → следующий этап.

project_positions.status (конвейер):
  created → spec_ready → in_purchase → purchased →
  delivered → mounting → installed → documented
  (+ финансы: ready_for_payment между in_purchase и purchased)

Этап фильтрует позиции по входному статусу.
```

═══════════════════════════════════════════════════
2. ЗАДАЧИ — всегда SUBTASK (единый механизм!)
═══════════════════════════════════════════════════
```
ВСЕ задачи всех этапов — это subtasks. НЕ выдумывать
отдельные сущности (была ошибка: payment_request).

subtasks:
  stage_id     — этап (design|supply|ЗАКУПКИ|works|docs|handover)
                 ФИКСИРОВАН (не мигрирует)
  parent_id    — дерево подзадач (откуда пришла)
  status       — new|in_progress|done|paused|cancelled (общий)
  stage_status — свой для этапа (снабжение: 14 статусов)
  responsible_id — ответственный

Новый этап = новый stage_id, задачи те же subtasks.
Задача закупки = subtask где stage=закупки. Не новая сущность!
```

═══════════════════════════════════════════════════
3. СВЯЗЬ ПОЗИЦИЙ С ЗАДАЧЕЙ — два механизма
═══════════════════════════════════════════════════
```
subtask_positions (таблица связи) — КАКИЕ позиции в задаче:
  subtask_id ↔ project_position_id
  → "свои позиции задачи" = SELECT WHERE subtask_id={id}
  НЕ по статусу! (статус общий, задач много)

current_subtask_id (поле позиции) — ГДЕ позиция активна СЕЙЧАС:
  → эксклюзивность (одна активная задача за раз)
  NULL = свободна

Добавить позиции в задачу:
  1. subtask_positions INSERT (связь)
  2. current_subtask_id = задача (занятие)
Освободить:
  current_subtask_id = NULL, subtask_positions DELETE
```

═══════════════════════════════════════════════════
4. ЭКСКЛЮЗИВНОСТЬ
═══════════════════════════════════════════════════
```
Позиция в ОДНОЙ активной задаче за раз.
current_subtask_id = занятость.
Доступна для выбора: status=нужный AND current_subtask_id IS NULL.
Завершил задачу → current_subtask_id = NULL (освободил).
```

═══════════════════════════════════════════════════
5. ДРОБЛЕНИЕ — отдельные позиции
═══════════════════════════════════════════════════
```
Разная судьба → отдельные позиции.
split: исходная UPDATE (остаток) + INSERT клон (часть)
  parent_position_id связывает, общий kp_item_id.
merge: обратно (сумма quantity, лишние удалить).
Цветовая связка пар — из parent_position_id.
```

═══════════════════════════════════════════════════
6. СВЯЗИ ПОЗИЦИИ (что она помнит)
═══════════════════════════════════════════════════
```
project_positions:
  kp_item_id       — из какой записи КП
  spec_subtask_id  — из какой спецификации (помещение)
  parent_position_id — если раздроблена
  current_subtask_id — где активна сейчас
  order_id         — в какой закупке (финансы, ставит бухгалтер)
  supplier_id, warehouse_id — поставщик, склад
  status           — конвейерный статус

Позиция — единый источник истины, несёт все связи.
```

═══════════════════════════════════════════════════
7. УНИФИЦИРОВАННАЯ ИСТОРИЯ — event_log
═══════════════════════════════════════════════════
```
ВСЕ события → ОДИН журнал event_log (не отдельные логи).
  entity_type (stage|position|subtask|project|...) + entity_id
  срезы фильтром. Единый метод LogEvent(...).
НЕ выдумывать отдельные истории — всё в event_log.
```

═══════════════════════════════════════════════════
8. СТАТУСЫ ЭТАПОВ — project_stages
═══════════════════════════════════════════════════
```
project_stages (6 на проект): waiting|in_progress|completed.
Наглядность на доске. Начало вручную, доработка с причиной.
Индикатор, не блокировка (опережающий поток).
```

═══════════════════════════════════════════════════
9. ОКНО ВЫБОРА ПОЗИЦИЙ — единое
═══════════════════════════════════════════════════
```
Один компонент для всех этапов (снабжение/работы/документы).
Параметр stage → доступность (spec_ready/delivered/installed).
Дерево (по спецификации/КП) + корзина. Дробление, merge, цвет.
GET /positions/available?stage=... — единый API.
```

═══════════════════════════════════════════════════

```

═══════════════════════════════════════════════════
11. ORDER — финансовая таблица (отдельно от задачи!)
═══════════════════════════════════════════════════
```
order — САМОСТОЯТЕЛЬНАЯ финансовая сущность.
  СОЗДАЁТ ТОЛЬКО БУХГАЛТЕР (снабженец не трогает).
  НЕ привязан к задаче. Позиция знает order (position.order_id).
Задача закупки (subtask) — рабочий процесс.
Order — финансовый факт (для раздела "Закупки").
Это РАЗНЫЕ вещи: задача-процесс vs order-финансы.
```

```
