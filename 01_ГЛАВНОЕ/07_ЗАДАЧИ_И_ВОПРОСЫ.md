```
Проанализирована библиотека пользователя (auth-websocket.ts, 523 стр).
ЧТО ЕСТЬ:
  ✓ sendRequest(payload, apiMethod, opts) → promise (request/response)
  ✓ корреляция по CorrelationId, шифрование, heartbeat, reauth
  ✓ протокол ОБЪЯВЛЯЕТ: MessageType.Event, Subscribe, Unsubscribe,
    UnsubscribeAll, RequireConfirm, ConfirmOperation, RenewConfirmationCode

ПРОБЕЛ 1 — СОБЫТИЯ НЕ ДОХОДЯТ ★★
  handleMessage маршрутизирует ТОЛЬКО по CorrelationId:
    const handler = this.messageHandlers.get(message.CorrelationId)
    if (handler) handler(message)
  → событие от сервера (нет ожидающего запроса) МОЛЧА ОТБРАСЫВАЕТСЯ.
  Сервер умеет, клиент — нет.

  СДЕЛАТЬ: эмиттер в библиотеку (~30 строк)
    в handleMessage, ПЕРЕД роутингом по CorrelationId:
      if (message.Type === MessageType.Event) { this.emit(message); return }

  ПУБЛИЧНЫЙ API:
    on(type: string, handler): () => void    // точечно (эфемерное в компоненте)
    onAny(handler): () => void               // ВСЕ события (для моста приложения)
    subscribe(topics) / unsubscribe(topics) / unsubscribeAll()   // протокол

  ★ 4 ТРЕБОВАНИЯ (из опыта пользователя + грабли React):
    ① onAny ОБЯЗАТЕЛЕН — глобальный мост слушает всё, имён заранее не знает.
       Реестр "имя→хендлеры" этого не даёт. (wildcard "Area.Entity.*" — опция,
       но onAny проще и покрывает случай)
    ② Map<string, Set<Handler>> вместо массива — проход по массиву O(n)
       на КАЖДОЕ событие; конвейер + много подписок = тормоза.
       Set: защита от дублей + дешёвое удаление.
    ③ Подписка ВОЗВРАЩАЕТ функцию отписки — критично для вкладок
       (открывают/закрывают постоянно → утечка + работа с размонтированным).
       useEffect(() => ws.on("X", h), [])  ← React сам вызовет unsubscribe
       ★ React StrictMode в dev монтирует ДВАЖДЫ → без отписки двойные
         срабатывания, ищутся долго.
    ④ try/catch вокруг КАЖДОГО хендлера — один упавший подписчик
       не должен ронять остальные (ошибка в модалке парсинга
       не должна убить инвалидацию всего приложения).

  ФОРМА:
    private handlers = new Map<string, Set<Handler>>()
    private anyHandlers = new Set<Handler>()

    private emit(m: Message<any>) {
      const call = (h: Handler) => { try { h(m) } catch (e) { console.error(e) } }
      this.handlers.get(m.Type)?.forEach(call)
      this.anyHandlers.forEach(call)
    }

  ★ on() vs Subscribe() — РАЗНОЕ:
    on()        — локальная подписка в браузере (кто в этой вкладке слушает)
    Subscribe() — сказать СЕРВЕРУ "шли мне события по этому проекту"
    Второе нужно если сервер не рассылает всё подряд.
    → вопрос "на что подписываться (проект? раздел? всё?)" — в разделе C

ПРОБЕЛ 2 — RequireConfirm СЛОМАЕТ ЗАПРОС ★★
  Сейчас: сервер отвечает RequireConfirm на CorrelationId →
  waitResponseForMessage видит "это не Error" → РЕЗОЛВИТ промис
  с RequireConfirm вместо данных + удаляет handler → запрос повис.

  СДЕЛАТЬ: перехват в waitResponseForMessage + КОЛБЭК НАРУЖУ
  ★ библиотека НЕ знает про UI — спрашивает код через опцию (как onRequiredAuth):

    createAuthWebSocket({ url, onRequireConfirm: async (payload) => {...код...} })

    if (responseMessage.Type === MessageType.RequireConfirm) {
      // НЕ резолвим, НЕ удаляем handler — ждём настоящий ответ
      const code = await this.options.onRequireConfirm?.(responseMessage.Payload)
      this.sendRawMessage({ Type: ConfirmOperation,
        CorrelationId: message.CorrelationId, Payload: { code } })
      return  // handler остаётся, дождётся Response
    }
  → компонент пишет обычное await api.getCompany(1), про код не знает.
```

## A2. Слой ergo-query.ts (хуки поверх ErgoWsApi) ★

```
  Поэтому метод РАЗБИВАЕМ ПО ТОЧКАМ:

  const keyOf = (method: string, payload?: unknown) => [...method.split("."), payload]
  // "ErgoArea.Companies.GetAll" → ["ErgoArea","Companies","GetAll", {Query}]

  export function useApiQuery<T>(method: ApiMethod, payload: object = {}) {
    const api = useApi()                    // возвращает null пока нет сессии
    return useQuery({
      queryKey: keyOf(method, payload),
      queryFn: () => api!.sendMessage(payload, method).then(m => m.Payload as T),
      enabled: !!api,                        // ★ не дёргаем пока не авторизованы
    })
  }
```

## A3б. ФОРМАТ СОБЫТИЯ ★ (правило для бэка)
```
★ НЕ нужно несколько ТИПОВ событий — нужны НЕЗАВИСИМЫЕ ПОЛЯ.
  Одно событие может быть и инвалидацией, и уведомлением сразу.

ФОРМАТ:
{
  "Type": "ErgoArea.Subtasks.Assigned",        // Area.Entity.Action
  "Invalidate": ["ErgoArea.Subtasks",          // что протухло (список)
                 "ErgoArea.Projects"],          //   (чужие сущности тоже)
  "Notify": { "UserIds": [5],                   // кому показать
              "Text": "Вам назначена задача" },
  "Payload": { "SubtaskId": 42, "ResponsibleId": 5 }
}

КЛИЕНТ — один обработчик, ветки независимые:
  ws.onEvent((e) => {
    if (e.Invalidate) e.Invalidate.forEach(t =>
      queryClient.invalidateQueries({ queryKey: t.split(".") }))
    if (e.Notify?.UserIds?.includes(me.id))
      notifications.push(e.Notify.Text)
  })

ЧЕТЫРЕ ПОВЕДЕНИЯ → ПОЛЯ (не типы):
  ① инвалидация  → поле Invalidate (нет поля → конвенция по имени: Type минус Action)
  ② уведомление  → поле Notify
  ③ протокол(код)→ НЕ событие: MessageType.RequireConfirm (уже отдельный тип!)
  ④ эфемерное    → Event БЕЗ обоих полей → глобальный слой пропускает,
                    компонент подписан точечно по e.Type (parse.progress)

ПРАВИЛО ДЛЯ БЭКА (3 строки):
  1. Имя: Area.Entity.Action — тот же Area.Entity что у ЗАПРОСОВ этой сущности
     (запросы ErgoArea.Subtasks.Query → событие ErgoArea.Subtasks.Created)
  2. Затронута только своя сущность → Invalidate можно не указывать
     (сработает конвенция: отбросить последний сегмент)
  3. Затронуты ЧУЖИЕ → перечислить: Invalidate: ["ErgoArea.Projects", ...]
     ★ типичный случай: subtask.created протухает и Projects
       (в проекте счётчик задач на вкладке: sections:[{count:3}])

ЛОВУШКА: одно событие часто трогает НЕСКОЛЬКО сущностей — именем не выразить.
  Поэтому Invalidate обязателен когда затронуты чужие.

"ГРУБАЯ" ИНВАЛИДАЦИЯ — НЕ ПРОБЛЕМА:
  ["ErgoArea","Subtasks"] протухает все задачи всех проектов, НО
  React Query перезапрашивает только АКТИВНЫЕ (смонтированные) запросы.
  Открыт список снабжения одного проекта → 1 запрос. Остальные помечены
  устаревшими, подтянутся при переключении.
  → начинать с грубой. Точечная (по ProjectId) — через predicate, если упрёмся.

ПОЧЕМУ Invalidate НА СЕРВЕРЕ: сервер ЗНАЕТ что изменилось, фронт может гадать.
  → добавили событие на бэке → фронт не трогаем вообще.
  → "разметка поведения события" переезжает на бэк, где при написании
    обработчика видно какие таблицы задеты.
  Событие самоописываемо: смотришь и видишь что делает.
```
